## 摘要
并发的LLM智能体共享可变自然语言状态会导致结构性竞态条件（SRC）：写-写冲突和跨分片的陈旧读冲突，这些冲突会悄然破坏智能体输出。现有多智能体框架（LangGraph、CrewAI、AutoGen）对共享状态不提供写所有权语义。我们提出S-Bus，一个HTTP中间件，其核心机制是服务器端DeliveryLog：每个智能体一个的HTTP GET操作日志，在提交时自动重建每个智能体的读集，无需在HTTP/1.1下更改智能体SDK。DeliveryLog提供的一致性属性——可观察读隔离（ORI），一种对读集的HTTP可观察投影的部分因果一致性——防止智能体通过共享分片协作时的结构性竞态条件。三项贡献：（C1）基于HTTP流量的自动读集重建的DeliveryLog机制，附带三层机械化证据：ReadSetSoundness和ORICommitSafety在TLAPS中机器检验（保留一个类型公理）；在N=3时穷举TLC（20,763,484个不同状态，零违规）；Dafny证明9个归纳可靠性引理。（C2）在共享分片竞争扫描中，与PostgreSQL 17 SERIALIZABLE和Redis 7 WATCH/MULTI相当的结构性冲突预防效果，427,308个活跃HTTP-409冲突中，三个后端均无类型I损坏。（C3）ORI的工作范围取决于拓扑：在专属分片工作负载中语义中性；在单分片协作写入中有害，因为保持性会传播并发矛盾。源代码：https://github.com/sajjadanwar0/sbus

## Abstract
Concurrent LLM agents sharing mutable natural-language state produce Structural Race Conditions (SRCs): write-write and cross-shard stale-read conflicts that silently corrupt agent output. Existing multi-agent frameworks (LangGraph, CrewAI, AutoGen) provide no write-ownership semantics over shared state.   We present S-Bus, an HTTP middleware whose central mechanism is a server-side DeliveryLog: a per-agent log of HTTP GET operations that automatically reconstructs each agent's read set at commit time without agent SDK changes under HTTP/1.1. The consistency property the DeliveryLog provides -- Observable-Read Isolation (ORI), a partial causal consistency over the HTTP-observable projection of the read set -- prevents structural race conditions when agents collaborate via shared shards.   Three contributions: (C1) The DeliveryLog mechanism for automatic HTTP-traffic-based read-set reconstruction, with three-tier mechanised evidence: ReadSetSoundness and ORICommitSafety machine-checked in TLAPS (modulo one retained typing axiom); exhaustive TLC at N=3 (20,763,484 distinct states, zero violations); Dafny discharges 9 inductive soundness lemmas. (C2) Empirical structural-conflict prevention parity against PostgreSQL 17 SERIALIZABLE and Redis 7 WATCH/MULTI on shared-shard contention sweeps with 427,308 active HTTP-409 conflicts: zero Type-I corruptions across all three backends. (C3) ORI's operating envelope is topology-conditional: semantically neutral in dedicated-shard workloads; harmful in single-shard collaborative writing because preservation propagates concurrent contradictions. # S-Bus：面向多智能体LLM状态协调的自动读集重建

## 1. 论文的核心问题与整体含义

- **研究动机与背景**：在大语言模型（LLM）驱动的多智能体系统中，多个智能体需要共享和修改可变自然语言状态（例如共享的数据库模式、代码片段）。由于智能体是并发运行的，它们之间会产生**结构性竞态条件（Structural Race Conditions, SRCs）**。这主要包括两类问题：
    1.  **写-写冲突（Write-Write Conflict）**：两个智能体同时读取同一份共享数据，各自生成修改并提交，后者会悄然覆盖前者的修改。
    2.  **跨分片脏读（Cross-Shard Stale-Read）**：一个智能体基于过时的共享数据（分片）进行推理和修改，导致其输出与其他智能体基于新数据的输出不一致。
    - 现有的多智能体框架（如LangGraph, CrewAI, AutoGen）虽然提供了消息传递或工作流编排功能，但**没有提供对共享可变状态的写所有权语义**，因此无法从根本上解决SRCs，从而悄无声息地破坏最终的系统输出质量。

- **核心含义**：论文旨在解决多智能体协作中的状态一致性问题，特别是由“读集”不一致导致的冲突。其核心思想是**自动、无侵入地捕获每个智能体实际“读取”了哪些共享数据，然后在提交时验证这些数据在读取后是否被其他智能体修改过**，从而实现乐观并发控制。

## 2. 论文提出的方法论

- **核心思想**：论文提出了一个名为 **S-Bus** 的HTTP中间件。它通过在服务器端维护一个 **DeliveryLog** 来透明地捕获每个智能体的读集，并引入一个名为 **可观测读隔离（Observable-Read Isolation, ORI）** 的一致性模型来保证该读集的新鲜度。

- **关键技术细节**：
    1.  **分片模型（Sharding）**：共享状态被划分为多个键值对（分片）。每个分片有唯一的键、版本号和内容。
    2.  **自动读集重建（DeliveryLog）**：
        - 服务器为每个智能体 (α) 维护一个 `DeliveryLog_α`，这是一个**只增**的日志。
        - 每当智能体（通过HTTP GET）读取某个分片 `k` 时，服务器在返回响应**之前**，将 `(k, 当前版本号v)` 记录到该智能体的DeliveryLog中。这让“读取”行为变为服务器可审计的。
        - 当智能体提交修改时，服务器将 `DeliveryLog_α` 中记录的版本号与注册表中分片的当前版本号进行比对。
    3.  **乐观并发控制（Optimistic Concurrency Control, OCC）**：
        - **提交协议（Algorithm 1）**：智能体提交时，服务器首先重建其有效读集（包括DeliveryLog记录和智能体显式声明的内容）。
        - **跨分片验证（Cross-Shard Validation）**：验证读集中所有**其他**分片的当前版本是否与该智能体读取时的版本一致。如果不一致，则提交失败，返回HTTP 409冲突，智能体需要重新读取、重新生成、重新提交。
        - **版本匹配验证（Version Mismatch）**：验证该智能体所要写入的分片本身的版本是否与其期望的一致（防止写-写冲突）。
    4.  **一致性模型——可观测读隔离（ORI）**：
         - ORI 是对系统在**HTTP层面可观察到的读操作** (R_obs) 提供的一种偏序因果一致性。它保证：
           1. **写-写序列化**：对同一个分片的所有写入操作是严格有序的。
           2. **跨分片R_obs新鲜度**：在提交时，该智能体读取过的任何其他分片上的版本都没有在它读取之后被更新过。
         - ORI位于“读已提交”和“快照隔离”之间：它能防止R_obs上的反依赖环形异常（G2-item），但对于HTTP不可见的隐藏读集（R_hidden）则无法保证，允许写偏斜（write-skew）。

## 6. 论文的主要结论与发现

1.  **结构性冲突预防有效**：S-Bus能够彻底防止基于HTTP可观测读集的结构性竞态条件。在严格的冲突测试中实现了“零损坏”，效果与PostgreSQL SERIALIZABLE和Redis WATCH/MULTI相当。
2.  **读集覆盖率的局限性**：系统只能自动捕获约**26.1%** 的读操作（R_obs）。高达**73.9%** 的读操作属于HTTP不可见的隐藏读集（R_hidden，例如LLM从自身对话历史和上下文中提取的信息）。虽然通过会话级的DeliveryLog累积，覆盖率可提升至99.8%，但这得益于自报告数据可能被高估。
3.  **拓扑依赖性**： **ORI的正确性/价值与工作负载拓扑高度相关**。
    - **专属分片（Dedicated-shard）**：每个智能体写入自己独有的分片，只读取共享分片。ORI在这种场景下**语义中性**（不会降低质量），主要价值是**保证所有智能体的贡献都被保留**。
    - **单分片协作写入（Single-shard）**：所有智能体竞争写入同一个分片。ORI强制保留所有冲突的写入，导致最终状态充满矛盾，**语义上有害**。作者明确建议此类场景使用序列化协调。
4.  **与现有框架的对比**：在并行协调效率上，S-Bus显著优于CrewAI，与LangGraph处于同一水平。在任务成功率上，弱于采用层级监督者模型的AutoGen，这揭示了并行效率与任务成功率的权衡。
5.  **硬件/软件资源消耗**：S-Bus的**架构值体现在操作简单性和LLM原生集成**，而不是吞吐量的突破。与数据库方案相比，它避免了网络往返和额外的协议开销，同时**无需改造智能体的SDK**。

## 7. 优点

1.  **问题新颖且实用**：精准地切入并解决了多智能体LLM系统中一个真实且鲜有人解决的工程难题——因共享可变状态缺乏所有权语义导致的结构性竞态条件。
2.  **解决方案优雅且无侵入**：通过服务器端的 `DeliveryLog` 自动重建读集，使得现有的基于HTTP的智能体无需做任何修改即可获得事务性保证，这是一大亮点。
3.  **严谨的形式化验证**：跨越了**TLC模型检测（穷举）**、**TLAPS定理证明**和**Dafny归纳证明**三个层面，对核心算法的一致性属性给出了极高置信度的数学保证，这在系统工程论文中尤为出色。
4.  **实验设计系统且诚实**：通过精心设计的实验，不仅能证明其有效性，还坦诚地暴露了其局限性（如隐藏读集问题、拓扑依赖性）。特别是Exp. Shared-State直接证明了其自身机制在特定场景下会帮倒忙，体现了科研的客观性。
5.  **与工业标准的对标**：将自身的解决效果与业界成熟且经过实战检验的并发控制系统（PostgreSQL, Redis）进行直接比较，这极大地增强了其 claims 的说服力。

## 8. 不足与局限

1.  **隐藏读集（R_hidden）问题未彻底解决**：这是**最大的局限性**。系统主要保证基于HTTP GET请求的显式读取，而LLM大量依赖内部上下文（对话历史）进行“隐式读取”。作者虽探索了代理方案（proxy），但未取得满意结果。对于长期依赖上下文的复杂任务，ORI的保障会被严重削弱。
2.  **语义质量的评估依赖不可靠的评判者**：论文使用LLM（GPT-4o-mini）作为评判者来评估最终输出的语义一致性，但承认**该LLM评判者未通过与人类专家的交叉验证**。这是一个重大的证据缺口，降低了所有语义质量相关结论的可靠性。
3.  **工作负载的泛化性有限**：绝大多数实验集中在SWE-bench衍生的代码协调任务上。虽然补充了非代码的数据管道设计实验，但结论是否能推广到更广泛的场景（如文档协同、RAG编排）仍有待验证。
4.  **HTTP/2兼容性问题**：DeliveryLog依赖于HTTP/1.1的连接顺序保证。在使用HTTP/2的多路复用连接中，这个假设会失效，限制了其在现代网络环境下的应用。
5.  **分布式保障不完整**：虽然实现了基于Raft的分布式版本，但其正确性仅限于**经验验证（Exp. DR-9）** 和抽象模型检查，**缺乏完整的TLAPS形式化证明**。存在一个约5毫秒的并发故障转移窗口，期间可能丢失更新。
6.  **单分片场景下的有害性**：论文明确指出，对于“多智能体协作写入同一个分片”的工作场景，**使用S-Bus反而有害**。这限制了其应用范围。
