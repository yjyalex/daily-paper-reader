## 摘要
我们提出了 TraceFix，这是一个针对大语言模型（LLM）多智能体协作的验证优先流水线。智能体根据任务描述将协议拓扑合成为结构化中间表示（IR），生成 PlusCal 协作逻辑，并利用 TLA+ 模型检查器（TLC）提供的反例迭代修复协议，直到验证通过。经过验证的过程体被编译为每个智能体的系统提示词，并在运行时监控器下执行，该监控器会拒绝拓扑结构之外的协作操作。在涵盖 16 个场景系列的 48 个任务中，所有任务均通过了完整的 TLC 验证；其中 62.5% 的任务在首次尝试时即通过，且没有任务需要超过四次修复迭代。状态空间跨越六个数量级，但每个任务的验证均在 60 秒内完成。一项包含 3,456 次运行的运行时对比显示，受拓扑监控的执行实现了最高的任务完成率（平均 89.4%，完全完成 81.5%）；且当模型能力降低时，使用验证协议的运行时性能下降速度约为仅提示词（prompt-only）和仅聊天（chat-only）基准方案的一半。在固定运行时的配对消融实验表明，经 TLC 验证的协议将死锁/活锁（DL/LL）率从 31.1% 降低至 14.1%，且在故障注入情况下表现出最大的差距。

## Abstract
We present TraceFix, a verification-first pipeline for Large Language Model (LLM) multi-agent coordination. An agent synthesizes a protocol topology as a structured intermediate representation (IR) from a task description, generates PlusCal coordination logic, and iteratively repairs the protocol using counterexamples from the TLA+ model checker (TLC) until verification succeeds. Verified process bodies are compiled into per-agent system prompts and executed under a runtime monitor that rejects out-of-topology coordination operations. On 48 tasks spanning 16 scenario families, all tasks reach full TLC verification; 62.5% pass on the first attempt and none requires more than four repair iterations. State spaces span six orders of magnitude yet verification completes in under 60 s for every task. A 3,456-run runtime comparison shows that topology-monitored execution achieves the highest task completion (89.4% average, 81.5% full) and that runtimes using the verified protocol degrade at roughly half the rate of prompt-only and chat-only baselines when model capability is reduced. A paired ablation under a fixed runtime shows that TLC-verified protocols cut deadlock/livelock (DL/LL) from 31.1% to 14.1%, with the largest separation under fault injection.

### 1. 核心问题与研究动机
*   **核心问题**：独立运行的 LLM 智能体在并发协作时，由于异步通信和共享状态，极易产生竞争、死锁、握手丢失和过早终止等并发故障。
*   **研究动机**：
    *   传统的“加锁”方案往往只是将竞争转化为死锁，且 LLM 难以预见所有可能的交错执行路径（Interleavings）。
    *   现有的多智能体框架（如 AutoGen, LangGraph）缺乏形式化语义，无法在设计阶段保证协作协议的安全性。
    *   需要一种“验证优先”的方法，在协议执行前通过形式化工具发现并修复潜在的逻辑漏洞。

### 2. 方法论
TraceFix 提出了一个包含设计、验证、修复和运行时的完整流水线：
*   **协议拓扑合成（IR）**：智能体首先生成一个结构化的中间表示（IR），定义参与者（Agents）、共享资源（Locks/Counters）和有向通道（Channels）。
*   **PlusCal 逻辑生成**：将拓扑转化为 PlusCal 代码（一种类算法语言），明确定义每个智能体的协作步骤（send, receive, acquire, release）和非确定性分支（either/or）。
*   **TLA+ 翻译与模型检查（TLC）**：将 PlusCal 翻译为 TLA+ 形式化规范，利用 TLC 检查器在有界范围内穷举所有可能的执行路径，检查死锁自由度、互斥性、资源清理和通道排空等属性。
*   **反例驱动修复（CEGIS 思想）**：若 TLC 发现违规，会生成一个具体的反例追踪（Trace）。修复智能体根据该追踪定位错误（如循环等待、过早退出），修改 PlusCal 代码并重新验证，直到通过。
*   **运行时监控（Runtime Monitor）**：将验证后的协议编译为智能体提示词，并在执行时通过监控层拦截任何违反拓扑定义的非法协作操作。

### 6. 主要结论与发现
*   **修复收敛性**：100% 的任务最终通过验证。62.5% 首次通过，其余在 4 次迭代内收敛。
*   **性能优势**：TraceFix（Topology-monitored）在完成率（89.4%）和安全性上均优于基准。
*   **鲁棒性**：当模型能力从 mini 降至 nano 时，使用验证协议的系统性能下降幅度仅为基准方案的一半。
*   **故障减少**：验证后的协议将死锁/活锁率从 31.1% 显著降低至 14.1%。
*   **监控必要性**：仅靠提示词（Prompt-only）不足以防止罕见交错路径下的死锁，运行时监控是必要的补充。

### 7. 优点
*   **形式化严谨性**：首次将 TLA+ 模型检查与 LLM 智能体协作协议的动态修复相结合。
*   **反例价值**：将抽象的验证错误转化为 LLM 可理解的最小失败示例，极大地提高了修复效率。
*   **平衡性**：既保留了智能体在领域任务上的自主性，又在协作层面上施加了硬约束。
*   **高效性**：证明了有界模型检查在处理复杂 LLM 协作协议时的计算可行性。

### 8. 不足与局限
*   **有界性限制**：验证是在有限边界（如通道深度、计数器上限）下进行的，超出边界可能仍存在未发现的漏洞。
*   **运行时差距**：监控器目前仅强制执行拓扑约束（如“谁能发给谁”），而不强制执行具体的步骤顺序（Step-ordering），这导致仍有 8.8% 的死锁残余。
*   **语义漂移**：在处理超时导致的修复时，LLM 可能会为了通过验证而简化协议逻辑，从而偏离原始任务的语义需求。
*   **模型依赖**：协议设计的成功高度依赖于底层模型（如 Claude Opus）遵循 PlusCal 语法和理解反例的能力。
