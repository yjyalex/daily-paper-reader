Abstract 

Large Language Model (LLM) agents are increasingly used in real-world products, where personalized and context-aware user in-teractions are essential. A central enabler of such capabilities is the agent’s long-term semantic memory system, which extracts implicit and explicit signals from noisy longitudinal behavioral data, stores them in a structured form, and supports low-latency retrieval. Building industrial-grade long-term memory for LLM agents raises five challenges: scalability, low-latency retrieval, pri-vacy constraints, cross-domain generalizability, and observability. We introduce the Hierarchical Long-Term Semantic Memory (HLTM )framework, which organizes textual data into a schema-aligned memory tree that captures semantic knowledge at multiple levels of granularity, enabling scalable ingestion, privacy-aware storage, low-latency retrieval, and transparent provenance; HLTM further incorporates an adaptation mechanism to generalize across diverse use cases. Extensive evaluations on LinkedIn’s Hiring Assistant show that HLTM improves answer correctness and retrieval F1 significantly by more than 10% 1, while significantly advancing the Pareto frontier between query and indexing latency. HLTM has been deployed in LinkedIn’s Hiring Assistant to power core per-sonalization features in production hiring workflows.

1 Introduction 

Large language models (LLMs) such as GPT [ 1], Claude [ 2], and Gemini [ 5] have demonstrated strong performance across a wide range of tasks in real-world agent systems, including question an-swering, code generation, and retrieval-augmented search [ 11 , 26 ,32 ]. Modern LLM-based agents [ 22 ] augment these models with external tools [ 19 ] and carefully designed workflows to enhance their reasoning and planning capabilities [ 23 , 25 ]. For example, LinkedIn’s Hiring Assistant (LiHA) [ 8] is an AI recruiting agent that helps recruiters source, evaluate, and outreach to candidates at scale. Agentic memory systems [ 4, 10 , 24 , 28 , 29 , 31 ] support an agent’s long-term interaction with its environment while main-taining continuity over time. They enable agents to persist user-specific signals from historical data and interactions and to lever-age them to personalize future agent–user exchanges. For instance, LiHA [ 8] needs to extract personalized hiring preferences from recruiters’ past projects and activities, and to recall and reason over this recruiter-specific memory to generate tailored, contextually appropriate responses in future interactions. Achieving this level of personalization at scale requires a memory system that can reli-ably extract implicit and explicit signals from noisy longitudinal behavioral data, store them in a structured form, and retrieve them efficiently during real-time conversations. Recent work has advanced agents’ long-context capabilities by introducing explicit memory modules. Systems such as MemGPT, Mem0, MemoryBank, ReadAgent, SimpleMem, and SCM [ 4, 12 , 14 ,16 , 21 , 31 ] extend LLMs with external stores that swap, persist, and

Complementary efforts explore more structured representa-tions: GraphRAG, HippoRAG, A-Mem, and G-Memory [ 6, 10 , 24 , 28 ]organize knowledge into graphs to support multi-hop reasoning, while RAPTOR, TreeRAG, and MemTree [ 17 , 18 , 20 ] exploit hierar-chical structure for scalable long-context retrieval. Despite these advances, designing long-term semantic memory for large-scale, industrial-grade agents remains challenging across several key di-mensions: 

• Challenge 1: Industrial-scale Data Ingestion A produc-tion memory system must ingest and process large-scale structured or unstructured data from diverse sources and formats. This demands compact memory representations, highly parallelizable ingestion pipelines, and lossless incre-mental updates. 

• Challenge 2: Low-latency Retrieval Many scenarios im-pose strict latency constraints. For example, chat-based agents must respond within tight budgets to maintain a flu-ent conversation, leaving little room for expensive retrieval or oversized context windows at serving time. As a result, most computation is ideally amortized offline or nearline, with only a lean, predictable retrieval path online. 

• Challenge 3: Privacy-aware Memory Isolation Enter-prise agents must comply with strict data-governance poli-cies (e.g., GDPR [ 7]). With these policies, the memory must enforce strict isolation, preventing cross-tenant retrieval, and support clean deletion when required. 

• Challenge 4: Adaptability and Generality The memory framework should be generic enough to support multiple products and use cases, yet adaptive enough to incorporate new query patterns and evolving schemas without rely-ing on brittle, hand-crafted business rules or per-feature pipelines. 

• Challenge 5: Observability, Provenance, and Consis-tency under source-of-truth updates The memory sys-tem should preserve references to its underlying informa-tion sources and surface them during retrieval, enabling inspection, debugging, and governance. It should also re-main consistent as the source of truth (SOT) evolves (e.g., edits, backfills, or deletions), preventing stale or contradic-tory memories. To address these challenges, we propose HLTM , a hierarchical long-term semantic memory framework for industrial LLM agents. 

HLTM organizes memory into a tree-structured index that supports massively parallel execution and lossless incremental ingestion, gen-erate multi-granularity memory representations for low-latency retrieval at serving time, and supports enhanced privacy policies by aligning the memory hierarchy with business scopes such as seats and projects. The framework remains general and adaptable through automatic query pattern analysis and adaptive knowl-edge extraction. Each memory node is bound to a stable business identifier, providing explicit provenance and observability for in-spection, debugging, and governance. 2 Related Work 

Recent work has substantially improved agents’ long-context pro-cessing by introducing explicit memory mechanisms. MemGPT [ 16 ]adopts an OS-inspired design that swaps information between the limited context window and external memory. Mem0 [ 4 ] stores discrete memory items and retrieves them primarily via seman-tic search. MemoryBank [ 31 ] maintains timestamped interaction memories with time-based decay to emphasize recency and salience. ReadAgent [ 12 ] compresses long documents into compact gist mem-ories with on-demand lookup of supporting passages. SCM [ 21 ]equips agents with a controller that adaptively decides when to store, update, or retrieve from a stream of past interactions. Beyond unstructured memory stores, recent studies explore structured representations to improve organization and retrieval. GraphRAG [ 6 ] builds a knowledge graph and leverages community-based summarization to better capture entity-centric dependencies. HippoRAG [ 10 ] draws inspiration from hippocampal indexing, inte-grating graph-based retrieval with personalized PageRank for multi-hop reasoning and continual knowledge integration. A-mem [ 24 ]uses Zettelkasten-style atomic notes with dynamic linking to sup-port autonomous memory evolution, while G-Memory [ 28 ] gener-alizes this direction to multi-agent settings with hierarchical graph structures that model both reusable insights and collaboration tra-jectories. In parallel, RAPTOR [ 18 ] and TreeRAG [ 20 ] leverage hier-archical structures for long-context retrieval, combining multi-level organization with structure-aware access to improve downstream QA. MemTree [ 17 ] complements this line with an incrementally updated memory tree that maintains node-level summaries and embeddings as agent knowledge evolves. Despite these advances, enterprise deployment remains challeng-ing. Many methods lack principled memory partitioning aligned with access boundaries, raising privacy and scope-isolation risks. Graph-based approaches often require extensive LLM usage during indexing and incur online latency, limiting scalability [ 30 ]. Tree-style methods frequently depend on the LLM’s world knowledge for aggregation, which can weaken domain-specific controllabil-ity [ 15 ]. Finally, most systems lack robust citation practices and provenance tracking, reducing transparency and debuggability in production [3]. 

3 Method 3.1 Overview 

We propose HLTM , a hierarchical long-term semantic memory that organizes information in a tree: higher-level nodes capture broad patterns, while lower-level nodes store fine-grained, entity-specific information. Together, these levels support both generic and highly targeted queries within a unified structure (Figure 3). In production, the tree topology aligns with business scope and access-control boundaries, enabling privacy enforcement by restricting retrieval to appropriate subtrees (Section 3.2). Node contents are built offline via data-driven preference extraction and hierarchical aggregation (Sections 3.3–3.4), and are queried at serving time using hybrid embedding-based retrieval and answered via in-context learning (Section 3.5). HLTM also supports adaptation to evolving query patterns and lossless incremental updates, enabling nearline freshness at low indexing cost (Sections 3.6–3.7). 

3.2 Schema-aligned Hierarchical Topology 

Most hierarchical memory systems adopt semantic hierarchies , clus-tering embeddings of text segments into progressively coarser nodes [ 18 , 20 ]. While effective for organizing free-form text, this ap-proach is ill-suited to multi-tenant enterprise privacy requirements: clustering can mix content across business scopes, creating pri-vacy and compliance risks under strict isolation and access-control policies [ 9, 27 ]. Semantic hierarchies are also brittle under incre-mental updates: new data often triggers re-clustering and topology changes, while keeping the topology fixed can lead to quality drift and periodic full re-indexing. 

Hierarchical schemas are ubiquitous in enterprise data (owner-ship chains), documents (structural nesting), and conversations (sessions and messages). HLTM exploits this by constructing a 

schema-aligned hierarchy whose topology is dictated by the enter-prise data model and ownership boundaries. In the hiring domain, for example, leaf nodes represent fine-grained work units (e.g., hir-ing projects), intermediate nodes represent recruiter seats, and root nodes represent organizational accounts. We formalize this as a tree index T = (V , E) , where V contains business entities and edges E follow schema-defined parent–child relations. Each node 

𝑣 ∈ V owns the memory for its entity, and retrieval is restricted to subtrees consistent with the query scope and access-control policy. This schema-aligned design provides three deployment-critical benefits: (i) privacy-aware scoping via subtree-restricted retrieval, enabling per-entity isolation and controlled sharing; (ii) localized incremental updates that touch only affected leaves and their ancestor nodes, reducing update cost from 𝑂 (|V |) to the tree height; and (iii) topology stability over time, avoiding drift and periodic full re-indexing commonly seen in clustering-based hierarchies.

3.4 Hierarchical Memory Aggregation 

Summarization-style queries that ask for a summary or synthesis over a broad scope typically require aggregating evidence from multiple leaf nodes [ 6]. A straightforward strategy is to retrieve a large set of relevant leaf-level documents at query time and ask the LLM to summarize them on the fly; however, this approach is often impractical for three reasons: (i) high online cost , be-cause summarizing detail-heavy evidence is both token-intensive and latency-critical; (ii) the needle-in-a-haystack effect , where relevant signals are drowned out by irrelevant details, increasing the likelihood of omissions and factual drift; and (iii) embedding-space misalignment , where broad summarization queries may not align well with leaf-level embeddings, leading to poor recall and unstable retrieval quality. To address these challenges, HLTM performs offline hierarchical aggregation , composing child memories into broader-scope parent representations in a bottom-up manner. Aggregation follows the parent–child relations defined by the tree topology (Section 3.2) and is applied recursively up to the root. For each parent node 𝑝 ∈ V ,we collect its children’s memory representations {𝑀 𝑐 } as input and prompt an LLM agent to generate parent-level representations in the same multi-view form. Specifically, we guide the LLM to (i) group semantically similar evidence across children, (ii) merge redundant content and reconcile conflicts, and (iii) optionally prune low-salience details that rarely recur across children.

HLTM performs identity-scoped multi-signal retrieval to efficiently retrieve privacy-compliant memory content for a given user query. multi-signal retrieval to robustly surface relevant memories un-der diverse query formulations.

3.7 Lossless Incremental Nearline Indexing 

Data freshness is critical for user-facing agents; for example, re-cruiter memories in LinkedIn’s Hiring Assistant should reflect new interactions within minutes. Full indexing that re-runs all LLM-based summarization and embedding steps is too expensive; incre-mental updates such as GraphRAG [ 6] are typically lossy and still require periodic full re-indexing. 

HLTM natively supports lossless incremental nearline indexing, keeping large-scale memories fresh without full re-indexing. As shown in Figure 4, we maintain a set 𝑉 ★ of leaf nodes newly created or modified since the last cycle; for each 𝑣 ∈ 𝑉 ★, we recompute its memory representation 𝑀 𝑣 as discussed in Section 3.3, and then update only its ancestor path {𝑢 ∈ 𝑉 | 𝑣 ⪯ 𝑢 } up to the root. Because each parent is always recomputed from its latest children, the resulting tree is identical to a full rebuild over the latest data snapshot. This enables nearline update windows of a few minutes that are substantially cheaper than full re-indexing while preserving the semantics of a fresh global rebuild. 

4 Experiments 4.1 Dataset 

We constructed a human-labeled benchmark dataset derived from LinkedIn’s Hiring Assistant usage logs. The dataset spans diverse hiring domains, with gold reference answers curated by domain experts. Each data point contains (i) raw hiring documents , in-cluding a recruiter’s historical hiring projects, within-project in-teractions on candidates, and anonymized candidate and project data; (ii) memory queries issued by the Hiring Assistant, spanning both summary-style queries that aggregate signals across projects to infer preferences (e.g., typical titles, locations, and qualifications) and retrieval-style queries that request targeted projects or evidence snippets for downstream reasoning; and (iii) gold answers , which are personalized to the underlying hiring documents. 4.2 Evaluation Metrics 

4.2.1 Performance Metrics. 

Lexical Similarity Metrics. Prior work in conversational AI and agent memory evaluation [ 4, 14 , 24 ] commonly reports lexical over-lap metrics. We report token-level F1 to balance spurious content and missing coverage, and BLEU-1 to provide a complementary, precision-oriented lexical similarity signal. 

Semantic Correctness Metrics. To evaluate answer correctness beyond surface-form overlap, we additionally employ an LLM-as-a-judge score. The judge is given the query, the system-generated answer, and the gold reference answer, and scores factual con-sistency with the reference by checking key constraints such as numbers and named entities. The judge is instructed to rely only on the gold reference and outputs a scalar score in [0, 1] along with a brief rationale. We validate reliability by measuring agreement between judge ratings and human annotations, observing strong alignment (Cohen’s 𝜅 > 0.8). The full judging prompt is provided in Appendix C.1. 

4.2.2 Deployment Metrics. 

Latency. We measure both offline indexing latency and online query latency . Offline indexing latency indicates whether a method can refresh and maintain memory at production scale (e.g., millions of documents ingested daily for LinkedIn’s Hiring Assistant). Online query latency directly affects end-user experience in interactive, chat-based workflows. When a method supports parallelism, we run indexing and retrieval with a fixed concurrency setting to reflect realistic deployment conditions. 

LLM Invocation Counts and Token Consumption. We track both LLM call counts and token usage, since together they largely deter-mine serving cost. Our accounting covers all LLM interactions dur-ing the query phase, including query rewriting, planning, retrieval-time processing, and final answer generation. We compute token usage with a consistent tokenizer ( o200k_base ). 

4.3 Baselines 

To comprehensively evaluate HLTM , we compare against both basic baselines and advanced RAG/memory methods. We hold the backbone LLM and embedding model fixed across methods, and evaluate all systems under the same serving setup and context-window constraints. 

4.3.1 Basic Baselines. We include two simple yet strong base-lines, following the evaluation practice of Mem0 [ 4], to isolate the value of an external memory module. (i) Conventional retrieval-augmented generation (RAG) [ 13 ] chunks documents into fixed-length segments ( size ∈ { 512 , 1024 , 2048 }), embeds each chunk, and retrieves the top-𝑘 chunks by semantic similarity ( 𝑘 ∈ { 1, 3}) as context for question answering. (ii) Full-context prompting feeds the full available history (or the maximal portion that fits within the LLM context window) directly into the LLM without retrieval or an external store, providing an architecture-free baseline under the same context-length and serving constraints. 

4.3.2 Advanced Baselines. We further compare against represen-tative advanced RAG and memory systems that introduce explicit structure beyond flat retrieval. (i) Graph-based methods represent memory as nodes and edges, enabling entity-centric organization and multi-hop retrieval; we include GraphRAG [ 6 ], HippoRAG [ 10 ], A-Mem [ 24 ], and SimpleMem [ 14 ]. (ii) Tree-based methods build hierarchical indices via recursive clustering and summarization to balance coverage and retrieval efficiency; we evaluate RAPTOR [ 18 ], and leave TreeRAG [20] and MemTree [17] for future comparison once runnable implementations are publicly available. (iii) Other common baselines include ReadAgent [ 12 ] and Mem0 [ 4], which serve as widely used reference points for long-context retrieval and memory-augmented generation. 

4.4 Implementation Details 

HLTM was originally deployed in LinkedIn’s production environ-ment with PySpark-based offline ingestion and in-house docu-ment/vector stores. For this study, we re-implemented HLTM as a standalone Python library for benchmarking. All methods run on Python 3.12 and access LLMs via the Azure OpenAI Service API. We use OpenAI models text-embedding-3-large for embeddings and gpt-4o-mini and GPT-5.2 as representative low-cost and ad-vanced LLMs. We report gpt-4o-mini results in the main paper and include GPT-5.2 results and details in Appendix D. Baselines use default open-source settings unless noted, and all results are averaged over three runs.

4.6 Ablation Study and Analysis 

We conduct an ablation study to quantify the contributions of tree aggregation, adaptation, and each memory representation. Table 3 shows that disabling tree aggregation (i.e., restricting retrieval to leaf nodes only) causes a large drop in summary-style correct-ness (0.798 →0.678; ≈12 points) and retrieval-style F1 (0.712 →0.534; 

≈18 points). Figure 6 further verifies this trend across multiple 𝑘 

values. Removing adaptation also degrades quality (correctness 0.798 →0.701; ≈10 points), indicating that aligning aggregation with workload-specific query patterns remains important even when the core hierarchy is intact. Finally, single-representation variants suggest that summary memory contributes most to overall per-formance, while facet and QA memories provide complementary gains; combining all three yields the best correctness on summary-style queries and the best F1 on retrieval-style queries. 

4.7 Hyperparameter Analysis 

Figure 6 plots retrieval F1 as a function of the retrieval budget 𝑘 . We observe two desirable properties of HLTM . First, performance does not exhibit an early peak: it continues to improve as 𝑘 increases, whereas the leaf-only variant peaks and then degrades for larger 

𝑘 . This suggests that aggregated summaries provide the language model with a structured “navigation map,” reducing confusion from long lists of unstructured leaf-level details when 𝑘 is large. Second, 

HLTM ’s curve flattens quickly, indicating weak sensitivity to 𝑘 be-yond a small threshold. Because bottom-up aggregation has already pooled cross-leaf patterns into higher-level memories, a modest 𝑘 

recovers most useful information, simplifying hyperparameter tun-ing and enabling robust deployment with a fixed retrieval budget. 

4.8 Privacy Discussion 

Privacy is a first-class requirement for production memory systems. In HLTM , isolation is enforced structurally: each node is bound to an identity scope (e.g., recruiter, hiring project), and retrieval is restricted to the subtree rooted at that scope. Recruiter-scoped queries can access only that recruiter node and its owned projects; project-scoped queries are further constrained to a single project. This business-aligned hierarchy lets us index once and query at multiple levels without per-level re-indexing or global LLM routing, while preserving strict privacy isolation. 5 Production Use Case 

LinkedIn Hiring Assistant (LiHA) [ 8 ] is an AI agent for recruiters, powered by LinkedIn’s dynamic talent network, that helps re-cruiters discover and engage candidates with greater speed and scale. Architecturally, LiHA is a plan-and-execute system centered on a supervisor agent that interprets recruiter intent and orches-trates specialized sub-agents and tools (Figure 7). HLTM is deployed as LiHA’s long-term semantic memory service, providing persis-tent, context-aware signals to support personalization (Figure 1) and enabling the assistant to adapt recommendations and work-flows over time. All memory data remains scoped to the recruiter’s environment and is not used for training content-generating AI models. Customers retain control over their stored memory with robust privacy and management options. 

6 Conclusion 

We introduce HLTM , a hierarchical, schema-aligned long-term memory framework for industrial LLM agents, and demonstrate its end-to-end deployment in LinkedIn’s Hiring Assistant [ 8]. HLTM 

organizes memories into a tree topology that mirrors real-world business scopes (Section 3.2) and generates memory content via complementary multi-view representations with bottom-up hierar-chical aggregation (Sections 3.3–3.4). Experiments on production hiring data show that HLTM consistently outperforms strong RAG and memory baselines by more than 10% 4 for the summary-style and retrieval-style queries respectively, while maintaining favorable latency and token cost for real-time, chat-based workflows (Sec-tion 4.5). In production, HLTM supports large-scale indexing with freshness guarantees via lossless incremental ingestion (Section 3.7) and provides privacy and observability through business-keyed, transparent nodes (Section 4.8), making it a practical foundation for industrial GenAI products. HLTM is deployed in LinkedIn’s Hiring Assistant to support real recruiter hiring workflows (Section 5). Looking ahead, we plan to extend HLTM to richer cross-entity rea-soning, multimodal signals, and additional agent use cases beyond hiring, and to explore tighter integration with agent planners and controllers for end-to-end adaptive systems.
