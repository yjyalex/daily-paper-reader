## 摘要
大语言模型（LLMs）在通用代码生成方面表现强劲，但其在企业级领域特定语言（DSLs）中的适用性仍有待探索，特别是针对从单个自然语言（NL）指令生成跨越多个文件和文件夹结构的仓库级变更。我们报告了宝马（BMW）的一项工业案例研究，该研究调整了面向代码的 LLM，以为基于 Xtext 的 DSL 生成和修改项目根目录下的 DSL 工件，该 DSL 驱动下游的 Java/TypeScript 代码生成。我们开发了一个端到端流水线，用于数据集构建、多文件任务表示、模型适配和评估。我们将 DSL 文件夹层级编码为结构化的、保留路径的 JSON，从而实现仓库规模的单次响应生成，并学习跨文件依赖关系。我们在三种配置下评估了两个经过指令微调的代码 LLM（Qwen2.5-Coder 和 DeepSeek-Coder，7B）：基准提示、单样本上下文学习和参数高效微调（QLoRA）。除了标准的相似度指标外，我们还引入了特定于任务的度量标准，用于评估编辑正确性和仓库结构保真度。微调在不同模型和指标上均带来了最显著的提升，在多文件输出的留出集上实现了高精确匹配准确率、显著的编辑相似度以及 1.00 的结构保真度。同时，与基准提示相比，单样本上下文学习提供了较小但持续的改进。我们通过专家开发人员调查和使用现有代码生成器的基于执行的检查，进一步验证了其实用性。

## Abstract
Large language models (LLMs) perform strongly on general-purpose code generation, yet their applicability to enterprise domain-specific languages (DSLs) remains underexplored, especially for repository-scale change generation spanning multiple files and folder structures from a single natural-language (NL) instruction. We report an industrial case study at BMW that adapts code-oriented LLMs to generate and modify project-root DSL artifacts for an Xtext-based DSL that drives downstream Java/TypeScript code generation. We develop an end-to-end pipeline for dataset construction, multi-file task representation, model adaptation, and evaluation. We encode DSL folder hierarchies as structured, path-preserving JSON, allowing single-response generation at repository scale and learning cross-file dependencies. We evaluate two instruction-tuned code LLMs (Qwen2.5-Coder and DeepSeek-Coder, 7B) under three configurations: baseline prompting, one-shot in-context learning, and parameter-efficient fine-tuning (QLoRA). Beyond standard similarity metrics, we introduce task-specific measures that assess edit correctness and repository structural fidelity. Fine-tuning yields the most significant gains across models and metrics, achieving high exact-match accuracy, substantial edit similarity, and structural fidelity of 1.00 on our held-out set for multi-file outputs. At the same time, one-shot in-context learning provides smaller but consistent improvements over baseline prompting. We further validate practical utility via an expert developer survey and an execution-based check using the existing code generator.

### 1. 核心问题与整体含义
*   **研究动机**：虽然 LLM 在通用编程语言（如 Java、Python）上表现出色，但在处理企业内部专有的领域特定语言（DSL）时面临挑战。
*   **核心挑战**：工业场景下的 DSL 变更通常涉及“仓库级”操作，即一个自然语言指令需要同时修改多个具有相互依赖关系的文件夹和文件。
*   **背景**：宝马使用基于 Xtext 的 DSL 来驱动下游 Java/TypeScript 代码生成。手动维护这些 DSL 文件耗时且需要高度专业知识，因此需要一种能理解复杂目录结构和跨文件逻辑的自动化方案。

### 2. 方法论
*   **核心思想**：将整个 DSL 项目的目录结构和文件内容“线性化”为单个结构化的 JSON 文档，使模型能在单次推理中处理整个仓库的上下文。
*   **关键技术细节**：
    *   **线性化表示**：将文件夹层级编码为嵌套 JSON 对象，文件路径作为键（Key），文件内容作为值（Value）。这保留了路径信息并允许模型学习跨文件依赖（如 Import 关系）。
    *   **任务定义**：输入为“自然语言指令 + 变更前的 JSON 上下文”，输出为“变更后的完整 JSON 目标”。
    *   **模型适配策略**：对比了三种方案：
        1.  **基准提示（Zero-shot）**：直接输入指令和上下文。
        2.  **单样本学习（One-shot ICL）**：在提示词中加入 DSL 语法规则摘要和一个完整的示例。
        3.  **参数高效微调（QLoRA）**：使用 4-bit 量化和 LoRA 适配器对模型进行有监督微调（SFT）。
*   **算法流程**：数据准备（线性化） -> 模型训练（QLoRA） -> 推理生成 -> 评估（自动指标 + 专家评审 + 生成器校验）。

### 3. 实验设计
*   **数据集**：基于宝马内部专有 DSL 构建。包含 774 个训练样本和 105 个留出测试样本。涵盖了创建新市场配置、添加/删除属性、添加/删除金融产品等操作。
*   **对比方法**：
    *   **模型选择**：Qwen2.5-Coder-7B-Instruct 和 DeepSeek-Coder-6.7B-Instruct。
    *   **配置对比**：Raw（原始模型）、Raw+ICL（单样本提示）、FT（微调后）。
*   **评估指标**：
    *   **通用指标**：精确匹配（EM）、JSON 合法性、BLEU。
    *   **自定义指标**：**变更相似度（Change Similarity）**（仅针对修改行进行 Levenshtein 距离计算）和**结构保真度（Structural Fidelity）**（评估文件夹/文件树的 F1 分数）。
    *   **定性评估**：4 名资深开发人员对 20 个样本进行 9 个维度的打分，并运行现有的 Xtext 生成器进行执行校验。

### 5. 实验数量与充分性
*   **实验规模**：针对 2 个主流开源模型在 3 种配置下进行了对比，实验组设计合理。
*   **充分性**：

### 6. 主要结论与发现
*   **微调效果显著**：QLoRA 微调在所有指标上均大幅优于提示工程。DeepSeek-FT 达到了 0.657 的精确匹配率和 1.00 的结构保真度。
*   **结构易，语义难**：模型很容易学会 JSON 格式和目录结构（结构保真度接近满分），但难点在于跨文件的语义一致性（如在多个层级正确同步属性类型）。
*   **模型差异**：DeepSeek 在原始状态下表现优于 Qwen，但微调后两者差距缩小。
*   **实用价值**：开发人员给出了 4.5/5 的高分，认为该工具在“市场初始化”等任务中可节省 40%-80% 的时间。

### 7. 优点
*   **工业实战性强**：直接针对企业多文件、跨依赖的复杂场景，而非简单的单函数生成。
*   **创新的表示法**：通过 JSON 线性化巧妙解决了 LLM 处理文件系统层级结构的问题。
*   **多维评估体系**：结合了文本相似度、结构正确性、下游生成器校验和专家反馈，评估非常客观全面。

### 8. 不足与局限
*   **数据依赖**：微调效果高度依赖于高质量的人工标注数据，对于完全没有存量代码的新 DSL 难以冷启动。
*   **“添加”操作的脆弱性**：在“添加属性”等重依赖任务中，模型仍可能出现幻觉（如引用不存在的类型），导致下游生成器报错。
*   **规模限制**：目前仅在 7B 模型上验证，且受限于上下文窗口，无法处理超大规模（如数千个文件）的 DSL 仓库。
