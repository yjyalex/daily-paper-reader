## 摘要
包含 N 个智能体的多智能体流水线通常在每次运行时发出 N 次大语言模型（LLM）调用。将智能体合并到更少的调用中（复合执行）有望节省 Token，但简单的合并调用会因工具丢失和提示词压缩而导致质量在无形中下降。我们提出了 Agent Capsules，这是一种自适应执行运行时，它将多智能体流水线执行视为具有经验质量约束的优化问题。该运行时测量每个组的协调开销，对组合机会进行评分，在三种复合执行策略中进行选择，并根据滚动平均输出质量对每次模式切换进行门控。一项受控的负面结果证实，在合并调用中注入更多上下文会加剧压缩而非缓解，因此该框架的升级阶梯（标准、两阶段、随后是顺序执行）通过转向逐智能体调度而非重写合并提示词来恢复质量。在 LLM 评判的质量方面，该控制器在每个测量的（模型、组、模式）单元格上都与手动调优的基准（oracle）相匹配：在基准会选择复合路由时进行路由，并在质量低于底线时恢复到精细模式，且无需针对每个模型进行配置。与手动构建的 14 智能体竞争情报流水线的 LangGraph 实现相比，Agent Capsules 在精细模式和复合模式下分别减少了 51% 的输入 Token 和 42% 的复合模式输入 Token，且质量分别提升了 0.020 和 0.017。与 5 智能体尽职调查流水线的 DSPy 实现相比，该框架在质量持平的情况下比未编译的 DSPy 减少了 19% 的 Token，比 MIPROv2 减少了 68% 的 Token 且质量提升了 0.052。甚至在复合模式启动之前，该运行时就通过自动策略解析、缓存对齐的提示词和拓扑感知的上下文注入提供了效率，在无需训练数据或针对每个流水线进行工程设计的情况下，达到了手动调优和编译时基准的水平。

## Abstract
A multi-agent pipeline with N agents typically issues N LLM calls per run. Merging agents into fewer calls (compound execution) promises token savings, but naively merged calls silently degrade quality through tool loss and prompt compression. We present Agent Capsules, an adaptive execution runtime that treats multi-agent pipeline execution as an optimization problem with empirical quality constraints. The runtime instruments coordination overhead per group, scores composition opportunity, selects among three compound execution strategies, and gates every mode switch on rolling-mean output quality. A controlled negative result confirms that injecting more context into a merged call worsens compression rather than relieving it, so the framework's escalation ladder (standard, then two-phase, then sequential) recovers quality by moving toward per-agent dispatch rather than by rewriting merged prompts. On LLM-judged quality, the controller matches a hand-tuned oracle on every measured (model, group, mode) cell: routing compound whenever the oracle would, and reverting to fine whenever quality would fail the floor, without per-model configuration. Against a hand-crafted LangGraph implementation of a 14-agent competitive intelligence pipeline, Agent Capsules uses 51% fewer fine-mode input tokens and 42% fewer compound-mode input tokens, at +0.020 and +0.017 quality respectively. Against a DSPy implementation of a 5-agent due diligence pipeline, the framework uses 19% fewer tokens than uncompiled DSPy at quality parity, and 68% fewer tokens than MIPROv2 at +0.052 quality. Even before compound mode fires, the runtime delivers efficiency through automatic policy resolution, cache-aligned prompts, and topology-aware context injection, matching both hand-tuned and compile-time baselines without training data or per-pipeline engineering.

### Agent Capsules: 多智能体 LLM 流水线的质量门控粒度控制总结

#### 1. 核心问题与整体含义（研究动机和背景）
在多智能体系统（MAS）中，传统的执行方式是为每个智能体发起独立的大语言模型（LLM）调用（即“精细模式”）。这种方式虽然保证了质量，但带来了巨大的 **Token 成本、冗余的系统提示词开销以及高延迟**。
虽然将多个智能体合并为一个调用（“复合执行”）可以节省成本，但往往会导致**工具调用失效（Tool Loss）**和**提示词压缩（Prompt Compression）**，即模型无法在单次调用中处理过多的指令，导致输出质量下降。
论文提出了 **Agent Capsules**，这是一个自适应运行时框架，旨在自动平衡多智能体流水线的执行成本与质量，解决“何时合并智能体是安全的”以及“合并失败后如何恢复质量”的问题。

#### 2. 方法论：核心思想与关键技术
Agent Capsules 将流水线执行视为一个带经验质量约束的优化问题，核心包含三个机制：
*   **组合评分（Composition Score）**：一种行为指纹算法。通过测量运行时信号（协调开销比例、智能体数量、工具调用密度、依赖深度）来预测合并调用的安全性，而非依赖模型的能力排名。
*   **质量门控（Quality Gate）**：引入 LLM-as-Judge 机制，对复合执行的输出进行影子评估。通过计算滚动平均质量（Rolling Mean），当质量低于预设底线（Quality Floor）时，自动拦截切换或回退到精细模式。
*   **升级阶梯（Escalation Ladder）**：当低级复合模式失败时，系统不会直接放弃，而是按顺序升级执行策略：
    1.  **Standard Compound**：所有智能体指令合并为一次调用。
    2.  **Two-Phase Compound**：先并行执行工具调用（Phase A），再合并进行推理（Phase B）。
    3.  **Sequential Compound**：按拓扑顺序进行 N 次调用，但累积上下文并应用输出引导。
*   **运行时优化**：包括拓扑感知的上下文注入（减少无关 Token）、自动输出引导（控制字数）和 Anthropic 缓存对齐。

#### 7. 优点
*   **自适应性强**：无需针对每个流水线或模型进行手动提示词工程或训练数据准备。
*   **安全保障**：质量门控和升级阶梯机制为生产环境提供了“安全网”，防止因追求成本而导致业务质量崩溃。
*   **即插即用**：作为运行时层，可以包裹现有的框架（如 LangGraph 或 CrewAI）。

#### 8. 不足与局限
*   **Judge 依赖性**：系统的核心决策高度依赖于 LLM-Judge 的评分准确性，且不同 Judge 之间存在校准偏差。
*   **模型覆盖有限**：未测试具备原生思维链（CoT）能力的模型（如 OpenAI o1、DeepSeek-R1），这些模型的 Token 计费逻辑可能影响评分算法。
*   **规模上限**：实验涉及的智能体数量最高为 14 个，对于超大规模（如 50+ 智能体）的流水线，上下文累积压力可能导致新的瓶颈。
*   **领域偏差**：主要集中在信息检索和分析任务，在创意写作或纯代码生成领域的泛化性尚待验证。
