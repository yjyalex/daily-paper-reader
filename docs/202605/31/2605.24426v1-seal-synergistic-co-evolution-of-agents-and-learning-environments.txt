# Abstract 

Large Language Model (LLM) agents are increasingly improved through interaction rather than static supervision. Yet most self-evolution methods adapt either the policy or the learning environment in isolation, leaving a structural gap: as the agent’s capability frontier shifts during training, the environment that provides supervision often remains static or only weakly coupled to the agent’s revealed failures. We call this mismatch Agent-Environment Misalignment . We propose SEAL, a closed-loop co-evolution framework for interactive tool-use agents. SEAL collects on-policy trajectories under executable verification, diagnoses failed rollouts into turn-level labels, and uses these diagnoses as a shared signal for both environment-side adaptation and model-side policy optimization. Specifically, the training-time learning interface evolves to expose clearer tool affordance cues, constraint information, and recovery-oriented feedback, while the policy is updated with diagnosis-guided advantage reweighting. Across in-distribution and out-of-distribution multi-turn tool-use evaluations, SEAL consistently improves low-resource agent learning: with only 400 training samples, it yields +8.25 to +26.25 average-point gains across three backbones and exhibits positive out-of-distribution transfer. These results show that jointly adapting the learner and its training-time learning substrate is a practical path toward more robust self-improving LLM agents.

# 1 Introduction 

Large Language Model (LLM) agents have recently demonstrated strong capabilities in reasoning, planning, and tool use, enabling progress on interactive tasks that require multi-step decision making, multimodal reasoning, and external action execution [ 1 –5 ]. Recent work further improves these agents through rein-forcement learning, tool-use post-training, agentic data generation, and multimodal data evolution [ 6 –9]. A growing trend behind these advances is self-evolution through interaction : agents improve by collecting rollout trajectories, receiving feedback, and iteratively refining their behavior through reflection, reinforcement learning, self-generated supervision, or visual skill memory [ 10 – 14 ]. This paradigm turns interaction feedback into a reusable source of supervision, offering a scalable path beyond static offline training. 1

Practical motivation. This promise becomes especially important in realistic tool-use settings, where agents must operate through multi-turn interfaces, satisfy strict execution constraints, and recover from partial failures [ 15 –20 ]. In such environments, improvement depends not only on how the policy is optimized, but also on what learning signals the training-time environment exposes while the policy is changing. Yet current self-evolving agents often adapt only one side of this interaction loop. 

Two one-sided adaptation patterns. Most existing methods follow one of two paths. (i) Model-centric evolution. These methods improve the policy through rollout replay, self-reflection, reward optimization, or post-training updates [ 21 – 24 ]. While effective, they typically optimize the agent against a largely fixed learning environment, making the learning signal dependent on the current policy’s own rollout distribution. In long-horizon interactive settings with sparse rewards, this self-referential rollout distribution can lead to policy-induced exploration bias, unstable recovery, and inefficient credit assignment [ 25 ]. (ii) Environment-centric evolution. These methods adapt curricula, task distributions, synthetic instructions, or interaction experiences. They recognize that agent capability is shaped not only by optimization, but also by the experiences exposed during training. Yet when such adaptation is not grounded in the current agent’s executable failures—what it can solve, where it repeatedly fails, and why—the evolved environment may remain weakly coupled to the learner’s actual needs. Such methods can increase diversity or difficulty, but without executable failure grounding, they may still fail to target the capability gaps that currently limit agent performance [26–29]. 

A shared closed-loop bottleneck. These two adaptation patterns look different, but they expose the same structural problem: the training-time learning environment fails to track the agent’s evolving capability boundary, and therefore provides signals that are too static, weakly targeted, or insufficiently informative. We refer to this mismatch as Agent-Environment Misalignment . Importantly, by “environment” we refer to the training-time learning substrate—including task exposure, observation interfaces, action constraints, and recovery feedback—rather than changes to the evaluation benchmark, tool semantics, or executable verifier. A concrete example illustrates the issue. When a tool call fails because the agent uses a city name where an airport code is required, a fixed environment may only return a generic execution error. Such feedback tells the agent that the trajectory failed, but not whether the failure came from missing a prerequisite lookup, using an invalid argument type, or failing to recover after the error. As a result, the policy receives only weak diagnostic supervision, and future trajectories are collected under similarly uninformative conditions. 2Our approach. Motivated by this observation, we propose SEAL , a closed-loop co-evolution framework for interactive tool-use agents. SEAL uses verifier-grounded failure diagnoses as a shared signal for both sides of training: it evolves the training-time interface with schema cues, constraint information, and recovery-oriented feedback, and it reweights policy-gradient updates by diagnostic utility. This enables 

capability-aware environment adaptation: instead of relying on generic difficulty scaling or unguided data expansion, the training interface is adjusted according to the current policy’s recurring failures, producing more informative rollouts while leaving the benchmark protocol unchanged. Extensive experiments on low-resource multi-turn tool-use settings validate the effectiveness of this design. Our contributions are threefold: 

• We identify Agent-Environment Misalignment : as the agent’s capability frontier shifts during training, the learning environment often remains static or only weakly coupled to the agent’s revealed failures. 

• We introduce SEAL , which uses verifier-grounded failure diagnosis to jointly evolve the training-time learning interface and guide policy optimization. 

• We show that SEAL improves low-resource multi-turn tool-use learning, yielding up to +26.25 average-point gains across three backbones with only 400 training samples and demonstrating positive transfer to held-out settings. 

# 2 Related Work 

2.1 Model-Side Self-Improvement 

Prior work shows that LLM agents can improve through repeated interaction rather than static prompting alone. Methods based on recursive skill learning, self-consolidation, reflective prompt adaptation, memory-based improvement, and reinforcement learning from interaction feedback primarily refine the agent itself [30 – 33 ]. In this sense, they are largely model-centric : experience is internalized into the policy, prompt, memory, or skill library, while the training-time learning environment is typically kept fixed. SEAL is complementary: it uses failed trajectories not only for model-side improvement, but also as verifier-grounded evidence for adapting the environment from which future trajectories are collected. 

2.2 Environment-Side Adaptation 

Another line of work adapts what the learner is exposed to during training. Curriculum learning, automatic curriculum design, synthetic instruction generation, task evolution, and tool or skill construction reshape the training distribution or interface [ 34 – 40 ]. However, most such methods operate at the level of task diversity, difficulty, or coverage. By contrast, SEAL performs failure-conditioned environment adaptation: verifier-grounded diagnoses determine which affordance cues, constraint information, recovery feedback, and training-time signals are exposed to target the current policy’s capability gaps. 

2.3 Interactive Environments and Co-Evolution 

Interactive benchmarks for tool use, function calling, web navigation, operating-system control, embodied simulation, and software engineering expose the multi-turn dependencies, execution constraints, sparse rewards, and recovery dynamics central to realistic agent learning [ 15 , 41 – 44 ]. Recent work also studies environment design and agent–environment co-evolution [ 45 – 47 ]. SEAL is closest to this perspective, but under a stricter protocol: it keeps benchmark tasks, tool semantics, and the executable verifier fixed, and adapts only the training-time learning interface in a failure-conditioned, verifier-grounded way.

# 4 Experiments 

We organize the evaluation around four questions: What is the experimental setup? How much does SEAL improve in-distribution performance on BFCL V3? Do these gains transfer to held-out tool-use settings? and Which components matter most, and how do the gains emerge during training? 

4.1 Experiment Settings 

Benchmarks. We use the BFCL V3 multi-turn subset as the in-distribution benchmark [ 41 ]. It contains 800 examples from four categories: Base , Missing Functions , Missing Parameters , and Long Context . We focus on a 

low-resource setting with 400 training examples, sampling 100 from each category, and use the remaining 400 examples for held-in evaluation. For out-of-distribution evaluation, we use BFCL V4 Web Search and Memory, together with the Retail, Airline, and Telecom domains of 𝜏 2-bench. These held-out benchmarks differ from BFCL V3 in tool domains, schema structure, and interaction patterns. Detailed protocols are provided in Appendix A. 

Models and baselines. We evaluate SEAL on three backbones: Qwen2.5-3B-Instruct, Qwen2.5-7B-Instruct, and ToolACE-2-Llama-3.1-8B. The Qwen models represent general instruction-tuned agents, while ToolACE-2 provides a stronger tool-specialized initialization. For each backbone, we compare against Vanilla RL under the same training split, rollout budget, optimizer family, and verifier reward, yielding a controlled comparison .We also report representative open-source and proprietary tool-use systems only as reference points , since they differ in scale and training recipe and are therefore not controlled baselines. 

Hyperparameters. We use a GRPO-style optimizer with 8 rollouts per prompt. The actor learning rate is set to 1 × 10 −6, with a train batch size of 32, a PPO mini-batch size of 8, and a PPO micro-batch size of 1 per GPU. We use a training temperature of 1.0 and a validation temperature of 0.0. The maximum prompt length, response length, and model length are set to 8192, 4096, and 16384, respectively. Training is conducted with vLLM-based asynchronous rollout workers on 4 GPUs. Additional implementation details are provided in Appendix C.

4.3 Out-of-Distribution Generalization 

Having established strong in-distribution gains, we next ask whether they transfer to held-out tool-use settings with different task distributions, tool domains, and interaction patterns. OOD transfer. Table 2 shows that SEAL improves both evaluated backbones and outperforms Vanilla RL on the aggregate BFCL V4 and 𝜏 2-bench metrics. For Qwen2.5-3B-Instruct, it raises BFCL V4 from 4.69% to 8.63% and 𝜏 2-bench from 9.64% to 11.76%. For Qwen2.5-7B-Instruct, it improves BFCL V4 from 9.17% to 12.71% and 𝜏 2-bench from 16.11% to 20.79%. 

Interpretation. These results suggest that SEAL learns transferable tool-use behaviors—such as argument grounding, error recovery, and multi-turn state management—rather than simply fitting the 400 BFCL V3 training examples. At the same time, absolute OOD scores remain modest, so cross-benchmark generalization is still challenging. Taken together, the in-distribution and out-of-distribution results point to a mechanism-level advantage: verifier-grounded diagnosis improves both sample-efficient learning on BFCL V3 and transfer to held-out settings. We therefore next ask which parts of the closed loop matter most. 

4.4 Ablation Study 

To explain where the gains come from, we ablate the three mechanisms built on verifier-grounded diagnosis: environment-side adaptation, diagnosis-guided reweighting, and closed-loop interface updates. Since diagnosis is the shared input to all SEAL components, removing it entirely would reduce the method to Vanilla RL. 

Relative contribution of each component. Removing diagnosis-guided reweighting causes a larger drop than removing environment-side adaptation, suggesting that diagnostic labels are especially important for allocating policy-gradient pressure. Still, environment-side adaptation contributes +4.50 points over the variant without it, and Full SEAL outperforms both single-sided variants. This pattern suggests that the two uses of diagnosis are complementary. 4.5 Training Dynamics 

Having identified which components matter most, we finally examine how the gains emerge during training. 

Learning dynamics. Figure 3 compares the validation trajectories of Vanilla RL and SEAL across the four BFCL V3 categories. SEAL improves more rapidly and reaches higher validation accuracy on most subsets, especially on Missing Functions and Missing Parameters , where failures often require identifying missing tool use or repairing invalid arguments. On Base , both methods improve early, but SEAL climbs faster and stabilizes at a higher level. The contrast is larger on Missing Functions and Missing Parameters , where Vanilla RL improves only gradually while SEAL opens a clearer gap as training proceeds. Long Context remains the hardest subset for both methods, but SEAL still shows a more consistent upward trend and a higher final accuracy. The overall curve is also smoother and higher for SEAL, suggesting that the gains are not driven by a single subset but accumulate across failure types over the course of training. The widening gap after the early phase is consistent with SEAL’s closed-loop design: as the policy’s dominant failures change, the training interface is updated to expose more relevant cues, whereas Vanilla RL continues to receive the same sparse terminal signal. This pattern supports the view that verifier-grounded diagnosis not only improves final performance, but also makes learning more targeted and sample-efficient throughout training. 10 5 Conclusion and Limitations 

We introduced SEAL, a closed-loop framework for co-evolving tool-use agents and their training-time learning environments. By using verifier-grounded failure diagnosis as a shared signal for learning-interface evolution and policy optimization, SEAL addresses Agent-Environment Misalignment while preserving tool semantics, task labels, and the evaluation verifier. Experiments on in-distribution and out-of-distribution multi-turn tool-use benchmarks show consistent gains in low-resource learning across multiple backbones. More broadly, our results suggest that improving tool-use agents is not only a matter of optimizing the policy against a fixed environment. Adapting the training-time interface to the agent’s revealed failures can make learning more targeted, sample-efficient, and robust to structured multi-turn errors. Several limitations remain. SEAL relies on executable environments with tool schemas, execution traces, and verifier feedback, so more open-ended domains may require richer diagnosis mechanisms and weaker forms of supervision. Its environment evolution is intentionally conservative, modifying the training-time interface rather than tool semantics or evaluation rules, which preserves fairness but limits the scope of adaptation. Diagnostic utility weights are also fixed across tasks and backbones, and cross-domain generalization remains challenging. Future work includes extending diagnosis to less structured environments, learning adaptive utility weights, and studying how co-evolution scales to broader tool ecosystems and longer-horizon agentic workflows. 11 References

# A Benchmark and Evaluation Details 

BFCL V3. Our in-distribution evaluation uses the multi-turn subset of the Berkeley Function-Calling Leaderboard (BFCL) V3. The benchmark evaluates whether an agent can correctly use executable tools over multi-turn dialogue contexts. Each task contains a user instruction sequence, a set of available tools, and an executable evaluation protocol. The subset contains 800 examples across four categories: Base , Missing Functions , Missing Parameters , and Long Context . Base evaluates standard multi-turn function calling with complete tool information. Missing Functions tests whether the agent can recognize when required tools are unavailable or when no valid tool applies. Missing Parameters evaluates whether the agent can handle underspecified arguments rather than hallucinating missing values. Long Context stresses state tracking and tool-use consistency over extended dialogue histories. Following the low-resource setting in the main experiments, we construct a 400-example training split by sampling 100 instances from each category. The remaining 400 examples are used for held-in evaluation. All SEAL and Vanilla RL runs use the same training split and held-in test split, ensuring that performance differences are attributable to the training method rather than data scale or split variation. 

BFCL V4. For out-of-distribution evaluation, we use the Web Search and Memory tracks from BFCL V4. These tracks differ from BFCL V3 in tool schema structure, task distribution, and interaction pattern. The Web Search track evaluates whether an agent can retrieve and synthesize information using search-style tools. The Memory track evaluates whether an agent can store, retrieve, and update memory-like information across interactions. Since SEAL is trained only on the BFCL V3 training split, BFCL V4 serves as a held-out benchmark for testing cross-distribution transfer. 

𝜏 2-bench. We also evaluate on 𝜏 2-bench across the Retail, Airline, and Telecom domains. These tasks require multi-turn tool use in simulated customer-service environments, where the agent must follow domain policies, interact with tools, and maintain task state across turns. Compared with BFCL, 𝜏 2-bench introduces different tool semantics, domain constraints, and user-agent interaction patterns, making it a complementary testbed for OOD generalization. 

Evaluation protocol and metric. All evaluations are performed using the original benchmark environments and verifiers. SEAL-specific diagnostic feedback, interface augmentation, capability-specific hints, and recovery-oriented training messages are disabled during evaluation. Thus, all reported results measure the policy under the original test-time protocol. We report success rate as the primary metric:

The comparison between Vanilla RL and SEAL is controlled along the following dimensions. Both methods use the same initial checkpoint, the same set of 400 BFCL V3 training samples, the same number of rollouts per prompt, the same optimizer, and the same decoding configuration. Both methods are evaluated using the original BFCL V3 multi-turn environment and the official verifier. Vanilla RL optimizes the standard GRPO objective using only verifier rewards. SEAL uses the same verifier rewards and the same GRPO backbone, but additionally uses verifier-grounded diagnoses from training rollouts to construct training-time interface updates and diagnostic utility weights. No method receives ground-truth tool trajectories, hidden parameter values, reference intermediate actions, or final answers during training. The evolved SEAL interface is used only during training. C.4 Logging and Reproducibility 

For each run, we log the random seed, base checkpoint, code version, training configuration, configuration overrides, rollout budget, decoding parameters, validation scores, round-level failure profiles 𝐶 𝑟 , interface states Ω𝑟 , diagnostic label distributions, and final evaluation outputs. These logs allow us to verify that changes in performance are attributable to SEAL components rather than differences in training budget, data split, decoding configuration, or evaluation protocol. 

# D Prompt and Interface Templates 

This section reports the prompt and training-time interface templates used in our BFCL implementation. The base tool-use prompt is shared by Vanilla RL and SEAL. SEAL modifies only the training-time environment interface, for example by annotating tool schemas, enriching failed tool feedback when enabled, or injecting short behavioral guidelines derived from recent diagnostic profiles. All SEAL-specific interface augmentations are disabled during evaluation. 

D.1 Base BFCL Tool-Use Prompt 

For Qwen-style BFCL experiments, the environment serializes the available tools into the system prompt using the following template. This base prompt is used by both Vanilla RL and SEAL. 19 BFCL Qwen Tool Prompt Template 

# Tools You may call one or more functions to assist with the user query. You are provided with function signatures within <tools></tools> XML tags: <tools> $tool_schema_json_1 $tool_schema_json_2 ... </tools> For each function call, return a json object with function name and arguments within <tool_call></tool_call> XML tags: <tool_call> {"name": <function-name>, "arguments": <args-json-object>} </tool_call> 

The base prompt defines the tool-call format and exposes the available tool schemas. It does not contain SEAL-specific diagnostic labels, recovery hints, or capability-dependent instructions. 

D.2 Schema Observation Annotation 

In our active BFCL configuration, SEAL uses a lightweight observation annotation mechanism. When 

observation_lite is enabled, SEAL does not change tool APIs, tool signatures, tool outputs, or verifier behavior. Instead, it augments the exposed schema descriptions with lightweight annotations that make existing constraints more explicit. Schema Observation Annotation Template 

$argument_description [required] $argument_description Allowed values: $enum_value_1, $enum_value_2, ... 

The environment also appends the following training-time instruction: Observation-Lite Instruction 

Observation note: parameters marked [required] must be provided exactly when calling a tool; do not invent tools outside the current schema. 

This annotation corresponds to the schema-affordance component 𝜙 schema . It exposes only public schema information, such as required arguments and enum constraints, and does not reveal the correct tool choice, hidden parameter values, reference tool trajectory, or final answer for any instance. 

D.3 Tool Feedback Evolution Template 

When tool_feedback_evolution is enabled, SEAL enriches only failed interactions. This component corresponds to the recovery-feedback component 𝜙 err . It converts low-level parse or execution errors into structured recovery hints, while preserving the original tool API and verifier. 20 For parser failures, the environment may append format-specific hints such as: Parse Error Hint Template 

$base_parse_error [Hint] Each tool call MUST be a JSON object inside <tool_call>...</tool_call> tags with both "name" and "arguments" keys, e.g. <tool_call>{"name": "<tool>", "arguments": {"<arg>": <value>}}</tool_call>. [Hint] Available tools: $tool_name_1, $tool_name_2, ... [Hint] If no available tool can serve the user 's request, reply in plain text instead of producing a malformed tool_call. 

For execution errors, the environment may append tool-specific but non-solution-revealing hints such as: Tool Error Hint Template 

[ToolHint] Required argument(s) $args for `$tool_name ` were missing. Re-emit the tool_call with these keys present in `arguments `.[ToolHint] `$tool_name ` does not accept argument `$arg `. Drop it and only use arguments listed in the tool schema. [ToolHint] `$tool_name ` rejected an argument type. Re-check argument types in the tool schema (string vs int vs list). 

These hints identify the class of tool-use error and the violated interface constraint. They do not provide the next correct tool call, instance-specific argument values, or the ground-truth solution trajectory. 

D.4 Diagnostic Evolution Guideline Template 

When diagnostic_evolution is enabled, SEAL reads recent failure statistics from the capability profile and injects short behavioral guidelines into the system prompt. This component corresponds to 𝜙 cap : the interface is adjusted according to recurring failure modes of the current policy rather than individual benchmark instances. Diagnostic-Driven System Guideline Template 

# Behavioral guidelines (env-evolved from recent failures): - [Cautious-call] If no available function clearly matches the user 's request, respond in plain text rather than fabricating a tool call. - [Active-respond] When the user 's request CAN be served by an available function, you MUST emit a tool_call rather than only replying in plain text. - [State-aware] Before issuing a state-mutating call, re-read the most recent tool results to confirm the current state matches what the arguments assume. - [Scope] Verify that the instance/target your call refers to actually appeared in earlier tool results before calling on it. - [Format] Match your final reply to the format implied by the user 's request: lists, units, exact strings, ordering. 

The guidelines are selected from aggregate failure patterns, such as spurious tool calls, missing tool calls, state 21 mismatches, or final-response format errors. They are capability-level instructions rather than per-instance hints. Therefore, they do not expose the reference trajectory, hidden state, hidden argument values, or target answer of any benchmark example. 

Active configuration. In the BFCL configuration used for the main rerun, the active training-time interface uses the base BFCL prompt together with observation_lite schema annotation. The additional tool_-feedback_evolution and diagnostic_evolution templates are included here to document the complete SEAL interface implementation, and should be interpreted as optional components when enabled by configuration. 

# E Qualitative Case Studies 

We present four qualitative case studies to illustrate how SEAL improves multi-turn tool use across different capability types: file-system state tracking, API-parameter recovery, cross-domain tool composition, and vehicle-control state maintenance. Each case compares the behavior of the base model before SEAL training with the behavior of the SEAL-trained model on the same BFCL task. We also include the expected outcome for post-hoc analysis only; these references are not exposed to the policy during training or evaluation. We focus on critical turns where the base model fails and the SEAL-trained model succeeds. These examples are not intended to replace the controlled Vanilla RL comparison in the main experiments; rather, they illustrate the types of tool-use failures that SEAL is designed to diagnose and correct.
