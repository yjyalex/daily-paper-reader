## 摘要
LLM多智能体系统通常通过自然语言对话或松散结构的共享内存进行协调，这使得中间状态难以验证、归因和审计。我们提出PatchBoard，一种基于模式约束的协作架构，用对共享结构化状态的经过验证的JSON Patch突变替代智能体间对话。一个架构师智能体构建任务特定的模式和流程规则，而一个确定性内核在事务性提交之前，根据模式约束、角色特定的写入契约和运行时不变性验证每个提出的状态突变。在630个匹配的ALFWorld场景中，PatchBoard实现了84.6%的成功率，而LangGraph为30.8%，Flock为61.6%；同时每个成功任务的令牌数减少到45.5k，而分别为368.3k和64.2k。

## Abstract
LLM multi-agent systems often coordinate through natural-language dialogue or loosely structured shared memory, making intermediate state difficult to validate, attribute, and audit. We introduce PatchBoard, a schema-grounded collaboration architecture that replaces inter-agent dialogue with validated JSON Patch mutations over a shared structured state. An Architect agent constructs a task-specific schema and workflow rules, while a deterministic kernel validates each proposed state mutation against schema constraints, role-specific write contracts, and runtime invariants before committing it transactionally. On 630 matched ALFWorld episodes, PatchBoard achieves an 84.6% success rate, compared with 30.8% for LangGraph and 61.6% for Flock, while reducing tokens per successful task to 45.5k, compared with 368.3k and 64.2k, respectively.

## 1. 核心问题与整体含义（研究动机与背景）

- **问题**：现有 LLM 多智能体系统通常通过自然语言对话或松散结构的共享内存进行协作，导致中间状态难以验证、归因和审计。错误（如未经验证的观察、过期计划、格式错误的中介信息）一旦进入共享上下文，会通过后续智能体放大，污染整个协作轨迹。
- **背景**：已有工作如 AutoGen、ChatDev、MetaGPT 等依赖自然语言交互；工作流图（LangGraph）、代码生成、黑板系统（Flock）等改进有限，但仍缺乏系统级边界来区分“模型建议”与“已提交的共享状态”。
- **整体含义**：PatchBoard 的目的是为多智能体协作提供一个可靠、可审计的基础设施，通过结构化的状态变异和事务性验证，从根本上限制不可靠输出的传播。

## 2. 方法论：核心思想、关键技术细节与算法流程

### 核心思想
用**经过验证的 JSON Patch 变异**替代智能体间的自由对话，操作在共享的结构化 JSON 状态上。一个 **Architect 智能体**在初始化时定义任务蓝图（模式、工人合同、工作流规则、上下文预算），一个**确定性内核**负责验证并事务性提交状态更新，拒绝所有不符合模式或授权的提议。

### 关键技术细节
- **蓝图（Blueprint）**：由 Architect 生成，包含：
  - 全局状态模式 Σ
  - 工人规格（读/写路径、视图预算、patch 格式约束）
  - 上下文预算（限制每个工人可观察的状态大小）
  - 工作流规则（将已提交状态事件映射到未来的工人调用）
- **JSON Patch 接口**：工人通过受限的 patch 操作（add、replace、test，默认禁止 remove）与状态交互。每个操作路径必须指向模式中的合法位置，值必须符合类型约束。
- **确定性内核**（核心模块）：
  - **验证**：检查 patch 语法、路径授权（写合同）、应用后临时状态是否满足模式及运行时不变性（公式 1）。
  - **事务性提交**：仅当所有检查通过时，才将 patch 应用到正式状态，并记录事务日志（工人ID、触发事件、视图哈希、patch、新状态哈希）。
  - **上下文切片**：根据工人的读合同和预算，从全局状态中提取有限视图（摘要+句柄），避免暴露无关字段，降低令牌消耗。
  - **事件调度**：基于已提交状态变更触发后续工人，不依赖自由消息。
  - **断路器策略**：跟踪连续无效 patch、无操作编辑、状态循环，超过阈值时采取重试、切换工人或终止等处理。
- **算法流程**（对应 Algorithm 1）：
  1. Architect 生成蓝图，验证格式。
  2. 内核初始化状态，启动事件队列。
  3. 循环从队列弹出一个工人事件，构建视图，调用工人返回候选 patch。
  4. 内核执行 `VALID_PATCH`（语法、授权、应用、模式验证、不变性）。
  5. 若通过，提交状态，记录事务，生成新事件加入队列；否则记录拒绝。
  6. 评估断路器策略，可能调整队列或状态。
  7. 循环直至队列空或预算耗尽。

## 3. 实验设计

### 数据集/场景
- **主要基准**：**ALFWorld**（文本化的具身智能环境），使用 126 个匹配的游戏文件，覆盖 6 种任务类型（如拾取清洁放置），每个游戏文件运行 5 个不同种子，总共 **630 个 episode**。
- **诊断基准**：**HotpotQA**（多跳问答），240 个预处理验证样例，用于评估证据链条和未支持的声明率，不是直接的任务成功比较。

### 对比方法
- **LangGraph**（v2024）：图结构工作流，监督者-子智能体模式（规划、动作、评论）。
- **Flock**（v2025）：黑板式多智能体编排，规划者+执行者工作流。
- **黑板控制**：plain blackboard（自然语言共享内存）和 structured JSON blackboard（仅结构共享，无验证事务）。
- **消融实验**：移除 patch/schema 接口、移除断路器、移除上下文切片、移除 Architect、移除模式强制、移除写合同共 6 个消融变体。
- **敏感性分析**：上下文预算（1k/2k/4k 字符）和模式来源（固定 vs 生成）。

### 评价指标
- 主要：任务成功率、环境步数、总令牌数、每个成功任务的令牌数。
- 诊断：答案准确率、未支持的声明率、证据覆盖率、已验证声明精度。
- 故障注入：污染率（无效JSON、错误路径/类型、未授权写入、假声明、循环终止率）。

## 6. 主要结论与发现

- **主结果**：PatchBoard 在 ALFWorld 上成功率 84.6%（533/630），远高于 LangGraph 的 30.8%（194/630）和 Flock 的 61.6%（388/630）。每个成功任务的令牌数降至 45.5k，是 LangGraph 的 1/8，Flock 的 70%。
- **黑板控制**：plain blackboard 成功率 77% 但令牌成本高（95.5k），structured JSON blackboard 成功率 77% 且成本较低（48.6k），但均低于 PatchBoard 的 84.6% 和 45.5k，说明事务性验证和写合同提供了额外增益。
- **消融**：移除 patch/schema 接口导致成功率下降最多（-0.15），成本翻倍；移除上下文切片同样显著（-0.15）；其他组件也有较小但一致的影响。
- **敏感性**：最小上下文预算（1k 字符）效果最好；生成的自定义模式优于固定模式。
- **故障隔离**：PatchBoard 对无效 JSON、错误路径/类型、未授权写入的污染率为 0；假声明仍可能（0.43），但低于黑板系统的 0.66-0.70；循环终止率高达 0.96，远高于基线。
- **诊断（HotpotQA）**：三个系统答案准确率相近，但 PatchBoard 的未支持声明率不是最低（Flock 最低），说明结构有效性不等同于语义正确性。

## 7. 优点

- **方法创新**：将多智能体协作转化为基于模式的、经过验证的状态变异，用结构化接口替代自由对话，从根本上提高了可审计性和可靠性。
- **验证机制完整**：确定性内核实现了多层验证（语法、授权、模式、运行时不变性），事务性提交防止污染，同时记录拒绝的 patch 用于追溯。
- **成本效率**：上下文切片和视图预算显著降低令牌消耗，每个成功任务的成本最低。
- **实验严谨**：大量对比、消融、敏感性、故障注入，统计检验充分，基线实现公开可重复。
- **可审计性**：事务日志记录每次状态变迁的完整上下文（工人、视图、patch、结果），支持重放和调试。
- **通用性**：架构设计不依赖特定模型或任务，可应用于其他需要结构化状态的应用。

## 8. 不足与局限

- **不保证语义正确性**：即使状态结构有效，模式允许的假声明仍可能被提交。HotpotQA 结果显示未支持声明率不低，因此需要更强的证据检索和验证器设计。
- **蓝图质量依赖**：Architect 生成的模式若过于稀疏会阻塞进展，过于宽松会削弱隔离。目前只在 ALFWorld 上验证，更开放领域（如软件工程、科学发现）需要更复杂的模式。
- **工程开销**：需要 schema 构建、上下文切片、patch 验证和日志记录，增加了系统复杂性，适合需要高审计性的场景，但简单任务可能过于重量级。
- **评估限制**：主要结果来自 ALFWorld（任务状态和成功条件清晰），在长生成、多轮对话等场景效果未知。HotpotQA 仅作为诊断，非竞争性基准。
- **伦理风险**：可审计性可能造成虚假信心，用户可能误认为模式有效即事实正确。事务日志可能包含敏感信息，需要访问控制和脱敏机制。
- **硬件与模型**：实验仅使用一种模型（Qwen-plus），其他模型的表现可能不同；没有探索训练成本或微调的影响。
