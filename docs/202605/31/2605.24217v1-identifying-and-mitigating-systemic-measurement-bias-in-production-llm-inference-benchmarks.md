## 摘要
随着大型语言模型（LLM）从研究环境过渡到生产部署，根据严格的服务等级目标（SLO）评估其性能变得至关重要。然而，当前的评估方法在大规模应用中存在严重的测量偏差。我们证明，广泛使用的基准测试工具依赖于单进程、基于asyncio的架构，在高并发下引入了根本性的客户端排队瓶颈。通过将基准测试客户端建模为$M/G/1$队列，我们从数学上展示了随着请求速率扩大，Python全局解释器锁（GIL）如何人为地膨胀首个令牌时间（TTFT）和每输出令牌时间（TPOT）指标。为了解决这一系统性不准确问题，我们提出了一种无偏的多进程评估框架，有效分布客户端负载，确保忽略不计的排队开销。此外，我们形式化了一个复合指标——标准化每输出令牌时间（NTPOT），以稳健地分摊端到端延迟，包括预填充和跨序列长度的调度延迟。我们的实证评估表明，该方法成功隔离了纯服务引擎性能，能够在超过每秒数千个查询的生产规模下对LLM进行准确、可重复的性能分析。

## Abstract
As Large Language Models (LLMs) transition from research environments to production deployments, evaluating their performance against strict Service Level Objectives (SLOs) has become critical. However, current evaluation methodologies suffer from severe measurement bias at scale. We demonstrate that widely used benchmarking utilities rely on single-process, asyncio-driven architectures that introduce fundamental client-side queuing bottlenecks under high concurrency. By modeling the benchmarking client as an $M/G/1$ queue, we mathematically demonstrate how the Python Global Interpreter Lock (GIL) artificially inflates Time to First Token (TTFT) and Time Per Output Token (TPOT) metrics as request rates scale. To resolve this systematic inaccuracy, we propose an unbiased, multi-process evaluation framework that effectively distributes client-side load, ensuring negligible queuing overhead. Furthermore, we formalize a composite metric, Normalized Time Per Output Token (NTPOT), to robustly amortize end-to-end latency, including prefill and scheduling delays across sequence lengths. Our empirical evaluation demonstrates that this methodology successfully isolates pure serving engine performance, enabling accurate, reproducible profiling of LLMs at production scales exceeding thousands of queries per second.

### 论文核心问题与整体含义（研究动机和背景）
- **核心问题**：当前广泛使用的 LLM 推理基准测试工具（如 vLLM Bench、SGLang benchmark、NVIDIA GenAI Perf 等）基于单进程 Python asyncio 架构，在高并发（数百至数千 QPS）下，Python 全局解释器锁（GIL）导致客户端侧出现严重的排队瓶颈，人为地膨胀了首个令牌时间（TTFT）和每输出令牌时间（TPOT）等关键延迟指标，使实际服务引擎性能被误判为“退化”。
- **整体含义**：评估工具自身的架构缺陷引入系统性测量偏差，使得基于这些工具的生产级 SLO 评估不可靠，进而误导基础设施决策和优化方向。需要一种无偏、可复现的评估方法论来真实反映 LLM 推理引擎在高负载下的性能。

### 论文提出的方法论
#### 核心思想
- 识别并消除客户端侧因单进程架构导致的排队瓶颈，通过**多进程分布式负载生成**绕过 GIL 限制，确保测量结果反映服务器真实行为而非客户端过载。
- 提出一个复合指标 **NTPOT (Normalized Time Per Output Token)**，将端到端延迟（包括排队等待、预填充计算、解码阶段）按照输出令牌数归一化，统一刻画服务引擎的计算效率。

#### 关键技术细节
- **客户端建模**：将单进程 asyncio 事件循环建模为 `M/G/1` 队列，客户端服务率μ受 GIL 限制，当请求到达率λ接近μ时，利用率ρ→1，客户端内部排队等待时间 Wq →∞，导致记录的 TTFT 和 TPOT 严重膨胀。
- **多进程架构**：将负载均匀分配给 k 个独立 Python 进程，每进程的利用率 ρₖ = λ/(kμ) 远小于 1，排队时延可忽略不计。典型实现如 Inference Perf，通过主协调进程 + n 个工作进程（对应 vCPU 数）实现负载分区。
- **动态并发调节**：通过 `max_workers` 和 `worker_max_concurrency` 参数自动计算最优并发度，确保客户端始终处于非饱和状态。
- **NTPOT 公式**：
  \[
  \text{NTPOT}_i = \frac{T^{(i)}_{\text{queue}} + (I_i \times T^{(i)}_{\text{prefill}}) + (N_i \times T^{(i)}_{\text{decode}})}{N_i}
  \]
  综合了服务器队列时延、预填充计算（与输入长度 I_i 成正比）和解码时延，除以输出长度 N_i 后获得的归一化延迟，可暴露出系统调度饱和信号。
- **连续延迟画像**：通过线性或几何级数逐步提高负载，生成延迟-吞吐量曲线，精确定位饱和拐点，避免单点静态测量。

### 论文的主要结论与发现
1. **单进程 asyncio 工具在高并发下必然引入测量偏差**：在 1000 QPS 时，vLLM Bench 和 Inference X 实际 QPS 仅为目标的 14.6% 和 44.3%，TTFT 被膨胀数秒至数十秒；5000 QPS 时完全无法完成。
2. **多进程架构可消除客户端排队瓶颈**：Inference Perf 在 5000 QPS 下实现 4,998 QPS，端到端额外开销仅 0.63 ms（vs 单进程工具 8 ms~58 s）。
3. **NTPOT 能有效反映系统饱和状态**：相比纯解码 TPOT，NTPOT 复合了预填充和排队时延，在系统接近饱和时 p99 NTPOT 会急剧上升，可作为可靠预警指标。
4. **客户端硬件对负载生成能力有显著影响**：2 vCPU 机器上多进程也只能达到 65.47 QPS（目标 100 QPS），需足够 vCPU 才能发挥多进程优势。
5. **跨范式验证确认 GIL 是核心瓶颈**：多进程 Python 可实现与 Go 原生 goroutine（k6）相同的负载保真度，而 Locust（单节点 Python）和 MLPerf（Python 接口）受 GIL 限制。

### 优点
- **理论严谨**：用 M/G/1 排队论建模客户端瓶颈，从数学上说明偏差根源，并提出定量缓解方案（多进程降低 ρ）。
- **实践指导性强**：提出的分布式评估框架（Inference Perf）可直接用于生产规模测试，并公开了调度精度、调度吞吐量等度量指标确保结果可信。
- **指标创新**：NTPOT 综合了请求生命周期的各阶段，比传统 TTFT/TPOT 更全面，特别适用于前缀缓存、RAG 等场景。
- **实验设计去耦**：使用模拟器确保任何性能差异仅来自客户端工具，避免模型服务器干扰。

### 不足与局限
- **未包含真实 GPU 推理验证**：论文仅用模拟器证明客户端偏差，没有在真实 GPU 集群（如 vLLM / TensorRT-LLM 部署）上验证多进程框架的端到端效果，实际应用时服务端排队与客户端排队的交互效应可能需要进一步研究。
- **客户端硬件依赖**：单机多进程依赖足够多的 vCPU，对于低端云实例或容器资源受限环境，仍可能成为瓶颈。论文指出扩展到 100,000+ QPS 需多节点分布式架构。
- **缺少系统消融实验**：只研究了 temperature 和数据集截断的影响，未讨论如不同 batch size、不同采样算法（top-k, top-p）、不同精度（FP16/INT8）等条件下的测量偏差表现。
- **NTPOT 的解释性**：虽然复合了多种延迟，但未给出明确的 SLO 阈值或与用户可感知延迟的直接映射，需要进一步实践积累。
- **非均匀流量缺失**：仅使用常数和泊松到达模式，未涉及真实环境中的突发流量、重尾分布，因此框架对突发负载的鲁棒性尚未验证。
