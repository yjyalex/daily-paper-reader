## 摘要
基于大语言模型（LLM）生成 JSON 等机器可读输出，因其在与外部系统集成方面的潜力而受到广泛关注。然而，现有方法无法严格强制生成的最大令牌（token）数量，导致无限生成或截断输出，进而引发系统故障。为了解决这一局限性，我们提出了 TruncProof，这是一种新颖的语法约束生成方法，使 LLM 能够在遵守预定义令牌限制的同时，生成语法有效的 JSON。通过利用 LL(1) 解析器的特性，TruncProof 在每个解码步骤中高效地估算完成语法有效输出所需的最小令牌数。在 Text-to-JSON 指令任务上的实验表明，即使在严格的令牌约束下，TruncProof 也能成功生成语法正确的输出。此外，我们展示了 TruncProof 可以与先进的解码策略有效结合，从而产生不仅语法有效而且语义准确的输出。

## Abstract
The LLM-based generation of machine-readable outputs such as JSON has attracted significant attention for integration with external systems. However, existing approaches cannot strictly enforce the maximum number of tokens to be generated, leading to infinite generation or truncated outputs that cause a system malfunction. To address this limitation, we propose TruncProof, a novel grammar-constrained generation method that enables LLMs to produce grammatically valid JSONs while adhering to a predefined token limit. By leveraging the properties of LL(1) parsers, TruncProof efficiently approximates the minimum number of tokens required to complete a grammatically valid output at each decoding step. Experiments on the Text-to-JSON instruction tasks demonstrate that TruncProof successfully generates syntactically correct outputs even under strict token constraints. Furthermore, we show that TruncProof can be effectively combined with advanced decoding strategies, resulting in outputs that are not only grammatically valid but also semantically accurate.

*   **研究动机**：在实际应用中，LLM 常被要求生成 JSON 格式以对接外部系统。现有的语法约束生成（GCG）方法（如 Outlines, SynCode 等）虽然能保证语法正确，但**无法在严格的 Token 长度限制下保证输出的完整性**。
*   **核心问题**：当生成过程达到预设的最大 Token 限制时，现有方法会直接中断，导致输出的 JSON 结构不完整（如缺少反括号），从而引发下游系统解析错误。
*   **研究目标**：开发一种能够动态估算“完成当前语法结构所需最小 Token 数”的方法，确保模型在预算耗尽前能“安全着陆”，生成一个语法完整且符合长度限制的输出。

### 2. 方法论：核心思想与关键技术
*   **核心思想**：利用 **LL(1) 解析器** 的预测特性，在每个解码步骤计算从当前状态到达语法终点所需的最小 Token 成本。如果选择某个 Token 会导致后续无法在剩余预算内完成语法闭环，则该 Token 会被屏蔽（Mask）。
*   **关键技术细节**：
    *   **LL(1) 解析器应用**：相比常用的 LALR(1) 解析器，LL(1) 更易于枚举可能的后续序列。
    *   **成本验证器（Cost Validator）**：
        *   **预计算阶段**：使用 Dijkstra 算法计算每个终结符（Terminal）和非终结符（Non-terminal）对应的最小 Token 长度。
        *   **运行时阶段**：计算当前已消耗 Token 数 + 补全当前 Token 序列所需数 + 闭合剩余语法树所需数。若总和超过 $N_{max}$，则屏蔽该路径。
    *   **解码策略集成**：TruncProof 作为一个 Logit 修改器，可无缝集成到 **贪婪搜索（Greedy）**、**束搜索（Beam Search）** 和 **蒙特卡洛树搜索（MCTS）** 中。

*   **实验场景**：设置了极度严格的 Token 限制，最大长度设为基准长度的 1.1 倍（$e=1.1$），并测试了 $1.0$ 到 $1.5$ 的不同扩张比例。
*   **对比基准（Benchmarks）**：

*   **评估指标**：语法正确率（Syntax）、模式符合率（Schema）、精确匹配率（Exact-match）。

### 6. 主要结论与发现
*   **语法鲁棒性**：在严格的 Token 约束下，现有方法（Outlines, SynCode 等）的语法正确率仅为 1%-36%，而 **TruncProof 达到了 100% 的语法正确率**。
*   **语义准确性**：虽然 TruncProof 保证了语法，但简单的贪婪搜索可能导致语义偏差。结合 **MCTS** 后，精确匹配率显著提升（从 21% 提升至 58%），证明了该方法与高级搜索算法结合的潜力。
*   **困惑度分析**：TruncProof 避免了模型陷入“局部最优但最终截断”的路径，迫使模型选择能在限制内完成的路径。

### 7. 优点
*   **严格保证**：是首个能在预设 Token 限制下严格保证 JSON 语法完整性的方法。
*   **通用性强**：作为 Logit 修改器，兼容多种分词器、模型和解码算法。
*   **高效估算**：通过预计算和 LL(1) 解析，实现了运行时的快速成本校验。

### 8. 不足与局限
*   **延迟问题**：结合 MCTS 等高级搜索策略时，计算开销较大，可能不适用于对延迟极度敏感的实时场景。

*   **语法限制**：目前主要针对 LL(1) 文法（如 JSON），对于更复杂的编程语言文法（如 Python 的完整文法）可能需要进一步扩展。
