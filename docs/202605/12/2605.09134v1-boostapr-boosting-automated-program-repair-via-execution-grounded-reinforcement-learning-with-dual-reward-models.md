## 摘要
用于程序修复的强化学习受到稀疏的执行反馈和粗粒度的序列级奖励的阻碍，这些奖励掩盖了哪些编辑实际上修复了错误。我们提出了 BoostAPR，这是一个解决这些挑战的三阶段框架：(1) 在带有推理轨迹且经过执行验证的示例上进行有监督微调；(2) 根据执行结果训练双奖励模型——一个序列级评估器和一个行级信用分配器；(3) 进行 PPO 优化，其中行级模型将奖励重新分配到关键的编辑区域。这种行级信用分配在一种自然适合代码更改的中间粒度上运行。BoostAPR 在 SWE-Gym 上进行训练，并在四个基准测试上进行了评估，在 SWE-bench Verified 上达到了 40.7%（比基础模型提高了 22.9 个百分点），在 Defects4J（Python 到 Java 的迁移）上达到了 24.8%，在 HumanEval-Java 上达到了 84.5%，在 QuixBugs 上达到了 95.0%，在开源模型中取得了具有竞争力的结果，并展现出强大的跨语言泛化能力。

## Abstract
Reinforcement learning for program repair is hindered by sparse execution feedback and coarse sequence-level rewards that obscure which edits actually fix bugs. We present BoostAPR, a three-stage framework addressing these challenges: (1) supervised fine-tuning on execution-verified demonstrations with reasoning traces, (2) training dual reward models--a sequence-level assessor and a line-level credit allocator--from execution outcomes, and (3) PPO optimization where the line-level model redistributes rewards to critical edit regions. This line-level credit assignment operates at an intermediate granularity naturally suited to code changes. Trained on SWE-Gym and evaluated on four benchmarks, BoostAPR achieves 40.7% on SWE-bench Verified (+22.9pp over base model), 24.8% on Defects4J (Python-to-Java transfer), 84.5% on HumanEval-Java, and 95.0% on QuixBugs, achieving competitive results among open-source models with strong cross-language generalization.

### 1. 核心问题与整体含义（研究动机和背景）
自动程序修复（APR）旨在利用 AI 自动修复软件漏洞。尽管大语言模型（LLM）在此领域取得了进展，但利用强化学习（RL）优化 APR 仍面临三大挑战：
*   **执行反馈稀疏：** 修复结果通常是二元的（通过或不通过所有测试），缺乏中间状态的引导。
*   **信用分配困难（Credit Assignment）：** 奖励通常在序列级别（整个补丁）给出。对于包含多行修改的补丁，模型无法得知具体哪一行的修改是有效的，哪一行是有害的，导致梯度估计方差大、学习效率低。
*   **分布偏移：** 模型在特定数据集上训练后，难以泛化到真实世界中复杂且多样的 bug 模式。

### 2. 方法论
BoostAPR 提出了一个三阶段训练框架，核心思想是**引入行级（Line-level）信用分配机制**来细化奖励信号。

*   **阶段 I：执行验证的推理迁移（SFT）**
    *   使用强模型（如 Claude 3.5）生成包含“诊断推理轨迹”和“补丁”的示例。
    *   **关键点：** 只有通过严格执行验证（通过所有测试）的示例才会被用于监督微调（SFT），确保学生模型学习到正确的逻辑。
*   **阶段 II：双奖励模型训练（Dual Reward Models）**
    *   **序列级奖励模型 ($R_{seq}$)：** 评估补丁的整体质量。采用回归（绝对分数）与偏好学习（相对排序）的混合目标函数。
    *   **行级信用分配器 ($R_{line}$)：** 学习识别补丁中哪些编辑行（edit-line spans）对修复起到了关键作用。
    *   **监督信号来源：** 利用执行失败时的堆栈轨迹（Stack-trace）进行启发式标注。如果某行修改出现在失败路径上，则赋予负反馈；若补丁成功，则所有修改行均为正反馈。
*   **阶段 III：在线 PPO 优化**
    *   使用 PPO 算法进行策略优化。
    *   **奖励塑造：** 将 $R_{seq}$ 给出的总分，根据 $R_{line}$ 计算出的权重，重新分配到具体的 Token 上。这种“行级”粒度避开了 Token 级的过度细碎和序列级的过度粗糙。

    *   **仓库级（Python）：** SWE-bench Verified（500个真实漏洞）。
    *   **仓库级（Java）：** Defects4J v2.0（835个漏洞，用于测试跨语言泛化）。

    *   **RL 方法：** SWE-RL, CodeRL, 以及同基座下的 GRPO 和拒绝采样 RL（RS-RL）。

*   **消融实验：** 详细对比了 SFT、PPO+序列奖励、PPO+双奖励的效果，验证了 $R_{line}$ 的增量贡献。

### 6. 主要结论与发现
*   **性能飞跃：** BoostAPR 在 SWE-bench Verified 上达到 **40.7%** 的解决率，比基础模型提升了 **22.9%**。
*   **跨语言泛化：** 尽管只在 Python 数据上训练，但在 Java 基准测试（Defects4J）上性能翻倍（11.3% -> 24.8%），证明模型学到了通用的修复模式而非简单的文本匹配。
*   **$R_{line}$ 的作用：** 虽然序列级奖励贡献了大部分涨幅，但行级信用分配进一步提升了 2.4%~5.6% 的性能，并显著提高了训练的梯度信噪比（提升 29%）和收敛速度。
*   **输入模式：** 实验发现奖励模型仅看补丁（Patch-only）比看“补丁+问题描述”效果更好，因为后者容易让模型学习到虚假相关性（如倾向于给简单问题高分）。

### 7. 优点
*   **中间粒度创新：** 提出的“行级编辑跨度（edit-line spans）”信用分配既符合代码差异（Diff）的结构，又比 Token 级更稳定，比 Hunk 级更精细。
*   **执行驱动：** 整个流程高度依赖真实的执行反馈，而非仅仅是文本相似度（如 BLEU 分数），这更符合程序修复的本质。
*   **训练效率：** 相比于动辄使用 70B 或更大模型的系统，BoostAPR 在 32B 模型上通过算法优化达到了顶尖性能，具有较好的参考价值。

### 8. 不足与局限
*   **启发式标签噪声：** $R_{line}$ 的监督信号依赖于堆栈轨迹的解析，这是一种启发式方法，存在约 18% 的标注不一致率（根据论文审计结果）。
*   **复杂漏洞挑战：** 对于涉及多个文件、具有隐式依赖关系的复杂 bug，模型仍表现出定位不全的问题。
*   **格式依赖：** 信用分配机制高度依赖于统一差异格式（Unified Diff）的解析，如果模型生成的格式轻微受损，分配机制可能失效。
