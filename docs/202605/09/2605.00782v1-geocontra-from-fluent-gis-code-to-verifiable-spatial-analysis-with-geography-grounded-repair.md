## 摘要
GIS 科学中可靠的空间分析需要保持坐标语义、拓扑、单位和地理合理性。目前基于大语言模型（LLM）的 GIS 系统能够生成流畅的脚本，但很少在大规模范围内强制执行这些地理规则。我们提出了 GeoContra，这是一个针对 LLM 驱动的 Python GIS 工作流的验证与修复框架。它将每个任务表示为一个可执行的地理空间合约，包括自然语言问题、模式（schemas）、CRS 元数据、预期输出、空间谓词、拓扑、指标、所需操作以及禁止的捷径。生成的程序会经过静态规则检查、运行时验证和语义验证，违规行为会被反馈到有界修复循环中。在涵盖波士顿地区 15 个区域、9 个任务族和 11 个开源模型的 7,079 个真实地理空间任务（每个模型运行 600 次）的评估中，GeoContra 将闭源模型的空间正确性从 DeepSeek-V4 的 47.6% 提升至 77.5%，Kimi-K2.5 从 57.7% 提升至 81.5%。在 11 个开源模型中，平均正确率提升了 26.6%。GeoContra 将流畅的代码生成转变为可验证的空间分析，能够捕捉到负旅行时间、CRS/字段模式违规、缺失谓词以及脆弱的输出转换，而这些问题在其他情况下可能会产生可执行但地理上无效的结果。

## Abstract
Reliable spatial analysis in GIScience requires preserving coordinate semantics, topology, units, and geographic plausibility. Current LLM-based GIS systems generate fluent scripts but rarely enforce these geographic rules at scale. We present GeoContra, a verification and repair framework for LLM-driven Python GIS workflows. It represents each task as an executable geospatial contract-including natural-language questions, schemas, CRS metadata, expected outputs, spatial predicates, topology, metrics, required operations, and forbidden shortcuts. Generated programs undergo static rule inspection, runtime validation, and semantic verification, with violations fed back into a bounded repair loop. Evaluated on 7,079 real geospatial tasks across 15 Boston-area zones, 9 task families, and 11 open-source models (600 runs each), GeoContra improves spatial correctness on closed models from 47.6% to 77.5% for DeepSeek-V4 and from 57.7% to 81.5% for Kimi-K2.5. Across 11 open models, average correctness rises by 26.6%. GeoContra turns fluent code production into verifiable spatial analysis, catching negative travel times, CRS/field-schema violations, missing predicates, and brittle output casts that otherwise yield executable but geographically invalid results.

### 1. 核心问题与整体含义（研究动机和背景）
*   **核心问题**：LLM 生成的 GIS 脚本虽然在语法上往往是正确的（可运行），但经常违反地理空间规则。例如：在经纬度坐标系下进行距离缓冲区计算（单位错误）、忽略坐标参考系统（CRS）转换、产生负值的旅行时间、或者在网络分析任务中使用欧氏距离捷径。
*   **研究背景**：现有的 LLM-GIS 研究多关注工具调用和代码生成能力，缺乏对地理规则的显式验证。这种“地理上无效”的分析会误导城市规划、环境正义等关键决策。

### 2. 方法论：核心思想与技术细节
GeoContra 的核心思想是将 GIS 任务转化为**可验证的地理空间合约（Geospatial Contract）**，并通过“验证-修复”循环确保结果的地理有效性。
*   **地理空间合约**：每个任务被定义为一个 JSON 对象，包含：自然语言查询、数据集模式（Schema）、CRS 元数据、预期输出约束（如非负值、行数一致性）、空间谓词语义、所需 GIS 操作及禁止的捷径方法。
*   **多层验证引擎**：
    1.  **静态验证器 (Static Verifier)**：在代码运行前，通过抽象语法树（AST）检查 CRS 处理、空间谓词方向、字段引用是否符合 Schema。
    2.  **运行时验证器 (Runtime Verifier)**：执行代码并检查生成的工件（Artifact），验证输出文件是否存在、数据格式是否正确、数值是否在地理合理范围内（如面积比率在 0-1 之间）。
    3.  **语义验证器 (Semantic Verifier)**：检查代码逻辑是否真正执行了任务要求的 GIS 操作（如缓冲区任务是否真的调用了 `buffer` 函数）。
*   **有界修复循环**：如果验证失败，系统会将具体的地理违规信息（而非通用的报错）反馈给模型，引导其进行针对性修复，直到达到最大尝试次数。
