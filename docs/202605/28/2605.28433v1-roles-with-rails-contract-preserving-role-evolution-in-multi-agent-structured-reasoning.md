## 摘要
基于角色的LLM多智能体系统需要自适应的角色池，然而自适应这样的系统不仅仅是提示优化的问题：角色往往承担结构性的义务，包括能力覆盖、消息兼容性、验证、最终答案聚合以及解析器兼容的输出协议。现有系统要么固定角色清单而失去自适应性，要么允许无约束生成导致角色漂移，移除结构上必要的角色并破坏答案合同。我们将此表述为合同保持的角色演化，要求每次提交的编辑都保持五个结构合同（能力、通信、验证、聚合、输出协议）。我们在SERO（自我演化角色编排框架）中实例化了这一表述，该框架通过信用引导检索、带有受保护终端聚合器和条件验证器修复的信用排序通信DAG，以及一个上下文赌博机控制器来演化类型化的角色卡片池，其中LLM提出的编辑仅在保持合同并提高任务分数时提交。在三个LLM骨干网上的真实世界推理基准实验证实了合同保持角色演化的价值。

## Abstract
Role-based LLM multi-agent systems need adaptive role pools, yet adapting such systems is not merely a matter of prompt optimization: roles often carry structural obligations, including capability coverage, message compatibility, validation, final-answer aggregation, and parser-compatible output protocols. Existing systems either fix the role inventory and lose adaptivity, or allow unconstrained generation to induce role drift, removing structurally necessary roles and breaking answer contracts. We formulate this as contract-preserving role evolution, requiring every committed edit to preserve five structural contracts (capability, communication, validation, aggregation, output protocol). We instantiate this formulation in SERO, a Self-Evolving Role Orchestration framework that evolves a typed role-card pool through credit-guided retrieval, a credit-ranked communication DAG with a protected terminal aggregator and conditional validator repair, and a contextual-bandit controller whose LLM-proposed edits are committed only when they preserve the contracts and improve task score. Experiments on real-world reasoning benchmarks across three LLM backbones confirm the value of contract-preserving role evolution.

### 论文提出的方法论

- **核心思想：契约保持的角色演化 (Contract-Preserving Role Evolution)**。
    - 将角色池的每一次编辑视为一次“提案”，该提案必须通过“五大约束”的检查，并且在实际任务中证明其性能提升后，才能被“提交”成为永久性修改。
    - **五大约束**：
        1.  **能力合约**：确保关键能力家族始终有至少一个角色覆盖。
        2.  **通信合约**：确保演化后的角色间沟通仍是兼容的、有向无环图（DAG）结构。
        3.  **验证合约**：确保错误检测和修复能力不会被移除。
        4.  **聚合合约**：确保存在一个受保护的、负责最终决策的终端角色。
        5.  **输出协议合约**：确保最终输出格式与评测基准或应用接口兼容。
- **SERO (Self-Evolving Role Orchestration) 框架**：具体实现上述思想的系统。
    - **1. 类型化角色卡片 (Typed Role Cards)**：每个角色是一个结构化对象（`role card`），包含能力标签、通信协议、角色类型、保护标志等信息，使得合约检查可以在结构化字段上进行，而非原始提示词。
    - **2. 信用估计 (Credit Estimation)**：通过三个层面的信号评估角色对任务的贡献：
        - **快速信用**：任务内的低成本代理，衡量角色输出与任务和团队共识的相似度。
        - **精确信用**：定期使用留一法（Leave-One-Out）评估角色的因果贡献。
        - **历史信用**：使用指数移动平均（EMA）平滑精确信用，避免单次噪音干扰。
    - **3. 推理过程 (Inference Procedure)**：
        - 根据信用和相关性检索一个小型活动团队。
        - 构建一个基于信用排序的有向无环图（Credit-Ranked DAG），确保可靠角色优先发言，验证器后置。
        - 经过一个受保护的终端聚合器生成最终答案，并可选地经过验证器修复。
    - **4. 受控演化循环 (Guarded Evolution Loop)**：
        - **控制器 (Controller)**：一个上下文赌博机（Contextual-Bandit）策略网络，观察当前任务、团队状态和信用统计，提出 `ADD`、`REMOVE` 或 `NOOP` 的编辑提案。
        - **候选生成器 (Generator)**：根据控制器提案生成一个合约兼容的候选角色池。
        - **提交规则 (Commitment Rule)**：只有当候选角色池满足五大约束，并且在当前任务上的得分优于原角色池时 (`ΔScore > 0`)，才提交该编辑。否则，完全回滚。
