## 摘要
单步检索增强生成 (RAG) 为简单问答任务提供了一种整合外部信息的有效方式，但在处理复杂问题时表现不佳。智能体 RAG (Agentic RAG) 扩展了这一范式，将单步检索替换为多步过程，其中大语言模型 (LLM) 充当搜索智能体，生成中间思维和子查询，以迭代地与检索系统交互。由于冗长思维和子查询的自回归生成，这种迭代过程会产生显著的延迟。为了解决这一局限性，我们提出了 LatentRAG，这是一个将推理和检索从离散语言空间转移到连续潜空间的新型框架。与现有的逐标记生成自然语言思维或子查询的显式方法不同，LatentRAG 在单次前向传播中直接从隐藏状态生成思维和子查询的潜标记 (latent tokens)。我们在潜空间中将 LLM 与稠密检索模型对齐，实现了基于潜子查询标记的检索，并支持端到端的联合优化。为了提高透明度并促进具有语义意义的潜表示，我们引入了并行潜解码机制，将潜标记翻译回自然语言。在七个基准数据集上的广泛实验表明，LatentRAG 达到了与显式智能体 RAG 方法相当的性能，同时将推理延迟降低了约 90%，大幅缩小了与传统单步 RAG 的延迟差距。

## Abstract
Single-step retrieval-augmented generation (RAG) provides an efficient way to incorporate external information for simple question answering tasks but struggles with complex questions. Agentic RAG extends this paradigm by replacing single-step retrieval with a multi-step process, in which the large language model (LLM) acts as a search agent that generates intermediate thoughts and subqueries to iteratively interact with the retrieval system. This iterative process incurs substantial latency due to the autoregressive generation of lengthy thoughts and subqueries. To address this limitation, we propose LatentRAG, a novel framework that shifts both reasoning and retrieval from discrete language space to continuous latent space. Unlike existing explicit methods that generate natural language thoughts or subqueries token-by-token, LatentRAG produces latent tokens for thoughts and subqueries directly from the hidden states in a single forward pass. We align LLMs with dense retrieval models in the latent space, enabling retrieval over latent subquery tokens and supporting end-to-end joint optimization. To improve transparency and encourage semantically meaningful latent representations, we incorporate a parallel latent decoding mechanism that translates latent tokens back into natural language. Extensive experiments on seven benchmark datasets show that LatentRAG achieves performance comparable to explicit agentic RAG methods while reducing inference latency by approximately 90%, substantially narrowing the latency gap with traditional single-step RAG.

#### 1. 核心问题与整体含义
*   **研究动机**：传统的智能体检索增强生成（Agentic RAG）虽然在处理复杂多跳问题上表现优异，但存在严重的**高延迟**问题。其瓶颈在于 LLM 需要以自回归（逐标记）的方式生成冗长的中间思维（Thoughts）和子查询（Subqueries），这通常导致其推理时间比普通单步 RAG 慢 16-22 倍。
*   **核心目标**：在不显著损失性能的前提下，大幅降低 Agentic RAG 的推理延迟，使其效率接近单步 RAG。

#### 2. 方法论
*   **核心思想**：将推理和检索过程从“离散语言空间”转移到“连续潜空间”。不再生成文本，而是直接利用 LLM 的隐藏状态（Hidden States）进行推理和触发检索。
*   **关键技术细节**：
    *   **潜标记生成（Latent Generation）**：引入特殊的思维标记（`<think>`）和查询标记（`<query>`）。LLM 在单次前向传播中处理这些标记，利用其最后一层的隐藏状态作为“潜思维”和“潜子查询”，避免了逐标记生成的开销。
    *   **潜空间检索（Latent Retrieval）**：通过一个轻量级的投影器（Projector）将 LLM 的潜子查询标记映射到检索模型的嵌入空间。
    *   **对齐目标（Alignment Objective）**：使用 **KL 散度损失**。由于缺乏中间子查询的地面真值（Ground-truth），模型通过模仿预训练检索模型在自然语言查询下的文档分布来进行对齐，实现端到端优化。
    *   **并行潜解码（Parallel Latent Decoding）**：为了保持透明度，设计了一个可选的解码机制，将潜标记重新翻译为自然语言。由于各步骤的潜标记已生成，解码过程可以跨步骤并行执行。

*   **数据集**：涵盖 7 个基准数据集，包括通用问答（NQ, TriviaQA, PopQA）和复杂多跳问答（HotpotQA, 2Wiki, Musique, Bamboogle）。

*   **对比方法（Baselines）**：

    *   基于提示的智能体 RAG（Iter-RetGen, Search-o1）。
    *   基于训练的智能体 RAG（Search-R1, AutoRefine, DeepRAG, ZeroSearch 等）。
*   **评估指标**：准确率（EM Score）和平均推理延迟（Latency）。

*   **消融实验**：针对检索损失函数（KL vs Cosine vs InfoNCE）、是否有预训练检索器、是否有解码损失等关键设计进行了详细对比。
*   **扩展性分析**：研究了 LLM 规模（3B, 7B, 14B）和检索模型规模（0.6B, 4B, 8B）对性能和延迟的影响。
*   **充分性评价**：实验设计非常全面，不仅验证了效率提升，还通过 LogitLens 技术深入分析了潜标记的语义含义，实验结果客观且具有说服力。

#### 6. 主要结论
*   **极高的效率提升**：LatentRAG 在保持与最强显式 Agentic RAG（如 Search-R1）性能差距小于 5% 的情况下，**推理延迟降低了约 90%**。
*   **延迟差距缩小**：LatentRAG 的总推理时间已接近传统的单步 Naive RAG。
*   **潜空间可行性**：证明了 LLM 的隐藏状态可以有效地充当检索查询，且通过 KL 散度对齐可以克服标注数据稀缺的问题。
*   **并行化优势**：即使开启透明化解码，由于支持跨步并行，其速度仍比显式方法快 40%-60%。

#### 7. 优点
*   **创新性强**：首次将潜空间推理（Latent Reasoning）引入智能体 RAG 领域，并解决了潜空间检索的对齐难题。
*   **端到端优化**：支持 LLM 与检索器的联合微调，比单纯的提示工程更具潜力。
*   **平衡透明度与效率**：通过可选的并行解码机制，解决了潜空间方法常见的“黑盒”问题。

#### 8. 不足与局限
*   **对训练轨迹的依赖**：模型训练依赖于现有显式 Agentic RAG 生成的轨迹数据（SFT），其性能上限受限于教师模型的质量。
*   **检索模型敏感性**：实验发现该方法在某些各向异性（Anisotropy）严重的检索模型（如 e5-base-v2）上表现较差，说明对检索空间的几何特性有一定要求。
*   **词汇精确度风险**：在某些需要极高词汇精确度的场景下，潜表示可能导致微小的拼写错误（如将 Chilcot 误写为 Chilcott），从而影响 Exact Match 评分。
