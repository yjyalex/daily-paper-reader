# ABSTRACT 

Single-agent systems (SAS) have become the default pattern for LLM-driven scientific workflows, but routing planning, tool use, and synthesis through a single context window comes with a well-known cost: as tool specifications and observational traces accumulate, the effective context available for each decision shrinks, and end-to-end reliability suffers. We present a multi-agent system (MAS) prototype for hydrodynamics in which specialized agents are coordinated through a Layer Execution Graph (LEG). A planner agent constructs query-specific execution topologies from natural-language routing heuristics that capture domain knowledge without hard-coding it as rigid control logic; specialist agents operate under strict tool allowlists and occupy complementary data-class roles. Between layers, consolidator agents fuse parallel outputs into concise briefs, and a reporter agent synthesizes the final response, while the runtime logs provenance for every tool invocation to support auditability. All benchmarks, ablations, and stress tests use Claude Sonnet 4.6 as the backbone model for both specialist and general-purpose agents. Evaluated on 37 queries spanning six complexity categories, the prototype achieves 93.6% factual precision with a 100% pass rate. Accuracy remains above 90% across runs from single-threaded to five independent parallel tracks, and under simulated loss of individual data sources the system degrades gracefully, still returning substantive partial answers. Together, these results suggest that planner-guided, graph-structured multi-agent orchestration can meaningfully alleviate the context-saturation bottlenecks that constrain monolithic single-agent architectures.

# 1 Introduction 

In the landscape of autonomous scientific workflows, single-agent systems (SAS) currently represent the foundational and most widely deployed paradigm for executing user-defined objectives. Effectively, a single-agent architecture relies on a solitary large language model (LLM) serving as the central cognitive engine, routing all planning, reasoning, memory retrieval, and tool execution through one unified context window [ 1 ]. Equipped with iterative reasoning frameworks, these monolithic agents independently decompose tasks, interface with external software or APIs, and synthesize outputs in a strictly sequential loop [ 2]. This architecture naturally excels in tightly bounded tasks where maintaining strict contextual coherence across sequential steps is paramount. However, as the complexity of scientific inquiry scales, SAS architectures frequently encounter structural and cognitive ceilings. Recent evaluations indicate that single-agent performance degrades sharply when parameterized with a high volume of tools, as the compounding influx

An extreme example of this is small language models, which exhibit diminished tool-calling capabilities and are more prone to hallucination compared to frontier LLMs [ 5]. Furthermore, routing all logic through a single LLM node creates inherent operational bottlenecks; if the agent falters on a sub-task or struggles to integrate competing domains of expertise, the entire analytical pipeline stalls. Consequently, while SAS offers streamlined execution without the latency of inter-model communication, their vulnerability to cognitive overload in multifaceted, heterogeneous data environments exposes the need for alternative architectural strategies. To address the cognitive bottlenecks and context degradation inherent to SAS, multi-agent systems (MAS) have emerged as a highly effective structural alternative for complex scientific workflows. Rather than forcing a single model to manage an entire analytical pipeline, MAS architectures decompose overarching problems into distinct, manageable sub-tasks distributed across a network of specialized agents [ 6 ]. By assigning specific roles such as a “planner” to outline methodology, a “coder” to execute geospatial data retrieval, and a “reviewer” to debug statistical logic, MAS mirrors the multidisciplinary collaboration of human research teams. This division of labor provides a critical computational advantage: it isolates state and tool execution across multiple context windows, preventing any single agent from becoming overwhelmed by heterogeneous data or exhaustive tool schemas [ 7]. Furthermore, MAS frameworks intrinsically support iterative self-reflection and peer critique without prompt pollution, which has been shown to drastically reduce hallucinations and compounding logical errors in chained scientific problem-solving [ 8]. However, deploying a MAS is not universally superior; it introduces new complexities regarding inter-agent communication protocols, latency, and token costs [ 9 ]. Consequently, MAS should be viewed not as a blanket improvement over SAS, but as a viable strategy for high-complexity, multi-scale inquiries such as geoscientific modeling where the benefits of specialized, collaborative reasoning might outweigh the overhead of system coordination. The efficacy of any MAS ultimately hinges upon its underlying coordination strategy. This is the mechanism by which specialized agents communicate, share memory, and resolve internal conflicts to achieve a unified goal. These approaches are not mutually exclusive as complex frameworks frequently hybridize three foundational architectural patterns: hierarchical oversight, iterative debate, and sequential pipelines. Hierarchical structures rely on a central "supervisor" to delegate sub-tasks. For instance, the recently developed PANGAEA-GPT framework utilizes a centralized planner to autonomously route complex geoscientific data retrieval requests to specialized worker nodes, successfully overcoming the semantic limitations of standard search engines [ 10 ]. Conversely, iterative debate patterns simulate structured discourse among peer models rather than relying on top-down control. In these interactions, agents actively critique and refine each other’s intermediate outputs to reach a consensus, a methodology proven to significantly reduce hallucinations and enhance the rigor of multi-step scientific reasoning [ 11 ]. Finally, sequential pipelines establish a strict division of labor where the discrete output of one agent serves as the immediate, isolated input for the next. A prominent demonstration of this sequential flow is found in AgentMD, a framework that cleanly segregates the upstream curation of biomedical tools from their downstream diagnostic execution to prevent cognitive overload [12]. In recent decades, scientific workflow management systems have relied heavily on graphs to formalize task depen-dencies, track data provenance, and automate the execution of complex computational pipelines [ 13 ]. By treating discrete analytical steps as nodes and data flow as directed edges, graph-based architectures naturally prevent circular dependencies while optimizing parallel processing capabilities [ 14 ]. Building upon this established paradigm, herein, we explore using something akin to the ubiquitous directed acyclic graph (DAG), a layer execution graph (LEG), to orchestrate MAS both efficiently and transparently for hydrodynamics. Note that a LEG groups work into synchronized stages (layers) that run in parallel then merge at barriers, whereas a DAG is any acyclic dependency graph with no required notion of global layer barriers or stage-wise synchronization. Graphs in agentic technologies are rapidly maturing. For example, emerging AI orchestration frameworks like LangGraph [ 15 ] and multi-agent architectures such as GPTSwarm [ 16 ] explicitly model agentic interactions as computational graphs to maintain state isolation and prevent the runaway reasoning loops common to unconstrained large language models [ 16 ]. In these graph-based multi-agent systems, specific LLM roles and tool executions are compartmentalized into distinct nodes. An upstream agent tasked with, for instance, retrieving and formatting raw environmental datasets must successfully fulfill its node’s execution criteria before data flows along the directed edge to trigger a downstream agent responsible for statistical synthesis. This deterministic routing not only significantly reduces redundant token overhead by limiting the context required at each step, but it guarantees the strict execution order necessary to maintain absolute data provenance across complex, multifaceted scientific inquiries. This level or orchestration is necessary for the field of hydrodynamics, an inherently multi-source discipline. Char-acterizing coastal flood hazard alone requires reconciling observational data of fundamentally different provenance: tide gauge records referenced to station-specific vertical datums, bathymetric and topographic surveys with varying horizontal resolution and acquisition epochs, remotely sensed inundation extents derived from synthetic aperture radar or multispectral imagery, and outputs from unstructured-grid hydrodynamic models like ADCIRC or SCHISM that resolve complex coastal geometries across millions of computational nodes. These datasets differ not only in format 2Towards Multi-Agent Autonomous Reasoning in Hydrodynamics and resolution, but in the physical quantities they represent. Compounding this, operational hydrodynamics draws simultaneously from atmospheric forcing fields, oceanic boundary conditions, river discharge records, and sediment transport data, each governed by separate agency mandates, archival conventions, and update cycles. The result is a data landscape that is deeply fragmented, vertically and temporally heterogeneous, and resistant to unified querying under any single schema. Herein, we detail the architecture and performance of a MAS prototype for hydrodynamics. While this prototype is fairly lightweight (i.e., it does not run models, but relies heavily on APIs for data), it is extendable for more complex workflows. In particular, it encompasses Python-based tool calling and image understanding to accommodate a range of spatiotemporal data including point-based time series and high-resolution contour maps. First, we detail its architecture including its agents and routing capabilities. We then demonstrate the prototype on a range of queries, focusing specifically on tool-calling and the cascading logic. Thereafter, we assess its performance using end-to-end benchmarks, ablations, and stress tests. Finally, we provide concluding remarks including ways to mature the prototype further. 

# 2 A MAS Prototype 

The MAS prototype considered in this study was tailored to predict hurricane-induced storm surge from data proceeding from historical, hypothetical, and forecast products. It is comprised of several agents which are dynamically selected and arranged in a LEG during run-time to respond to a user query of varying complexity. While agent interactions are completely autonomous, during the graph architecture phase, the user must sign-off on the execution plan presented to them to avoid unexpected API costs. Functionally, the MAS prototype calls models using Anthropic and OpenAI API protocols, meaning that it supports popular LLMs like Claude Opus 4.6 and GPT-5.5. The MAS prototype is composed of multiple “specialist” agents. These include: • NHC : Supplies official tropical-cyclone guidance from the National Hurricane Center (NHC) including best tracks and current and archived advisories. Some of the tools in this agent were adapted from OceanMCP (nhc-mcp)[17]. • NOAA CO-OPS : Supplies observed and predicted (viz., tide) water levels at stations situated along the US coast operated by the National Oceanic and Atmospheric Administration’s (NOAA) Center for Operational Oceanographic Products and Services (CO-OPS). Some of the tools in this agent were adapted from OceanMCP (coops-mcp) [17]. • USGS : Supplies survey-based high-water marks and Short-Term Network event data from the United States Geological Survey (USGS). • FEMA : Supplies official flood insurance rate maps (FIRMs) and related digital products such as the National Flood Hazard Layer (NFHL) that show where inundation is regulated; mapped special flood hazard areas (SFHAs) identify locations subject to the base flood (often called the 1% annual-chance flood), and base flood elevations (BFEs) give the reference flood height used on those maps for compliance and insurance rating. This agent calls the Federal Emergency Management Agency’s (FEMA) ArcGIS REST MapServer. • STOFS : Supplies contour plots of NOAA’s Global Storm Surge and Tide Operational Forecast System (STOFS-2D-Global) for a requested forecast cycle. STOFS data are sourced from NOAA’s S3 bucket on AWS housing operational outputs. Because STOFS output files are relatively large, Kerchunk is used by this agent to selectively pull the relevant time / spatial data without having to download the entire STOFS file, expediting processing on even modest internet connections. The primary output format of this agent is the .png file. • OSM : Supplies Open Street Map (OSM) maps via OSM’s tile server. Both satellite and non-satellite maps can be sourced. The primary output format of this agent is the .png file. The prototype is also composed of five general-purpose agents: • Graph Architect : Receives the user query and constructs a LEG complete with specialist and other general-purpose agents. This agent cannot participate in LEGs themselves, only constructs them. • Consolidator : Consolidates the outputs of multiple specialist agents, summarizing them for forthcoming agents. This agent always appears directly downstream of multi-agent layers. Downstream agents receive the consolidated outputs and not each upstream specialist agent’s raw output. This is a context compaction strategy meant to decrease burdens on context windows. 3Towards Multi-Agent Autonomous Reasoning in Hydrodynamics • Cross-Track Merge : Consolidates the outputs of multiple upstream parallel tracks. This agent always appears directly downstream of two or more parallel tracks. • Image : Analyzes the images output by specialist agents like STOFS and OSM . This agent utilizes the image understanding capabilities of frontier LLMs. Consequently, the primary function of this agent is to route images to the underlying LLM for interpretation. This agent always appears directly downstream of specialist agents that produce images (viz., STOFS and OSM ). • Reporter : Generates a concise response addressing the particulars of the query. At the heart of the MAS prototype is the registry , a manifest of sorts. The fields below determine agent identity, how the Graph Architect selects and connects specialists in the LEG, and the allowlisted tool surface the runtime enforces: • ID - a stable key or alias • kind - a string indicating whether the agent is a specialist (e.g., a tool caller) or general-purpose .• graph_data_class - an additional designation for specialist agents indicating whether they are experts either in observation , hypothetical , or forecast - centric queries. • system_prompt - long-form instructions for agent (e.g., what it is, what APIs it may reason about, how it should behave, tool-calling logic) • router_capabilities - condensed version of system_prompt that the Graph Architect (described below) uses to ascertain agent capabilities for proper routing 

• tool_names and openai_tools — jointly define the allowlisted tool surface: openai_tools supplies the schema the LLM sees, while tool_names is what the server enforces at execution time before routing a call to the appropriate dispatch backend. The registry is exercised to create the LEG. An agent called the Graph Architect maps the user’s natural-language request along with a specialist catalog derived from the registry to a LEG. The architect chooses one of two topologies for the LEG: (i) a single pipeline, i.e. an ordered sequence of layers, wherein each layer contains one or more specialists or a single consolidator; or (ii) parallel tracks, where n largely independent sub-questions receive n distinct pipelines (each with its own ordered layers and per-track goal text), after which the Cross-Track Merge agent integrates the lanes. Downstream code validates this specification, optionally rewrites it with deterministic routing heuristics which are pattern- and policy-driven, and compiles it into the LEG the user sees for sign-off before autonomous execution. Examples of routing heuristics include sourcing best track data for a particular storm before sourcing the associated storm surge from USGS and NOAA CO-OPS and never assigning dissimilar agents (e.g., observed and hypothetical specialists) to the same agent layer. While we attempt to limit the number of routing heuristics that Graph Architect 

sees to maintain its agility over a range of queries, we must declare certain agent-agent relationships to force recognized and efficient cascade logic. A user may sign-off on the suggested LEG or choose to modify it with a new prompt. If the LEG is approved, the prototype shifts from graph architecture to automated discussion. Each track runs as its own pipeline wherein execution moves downstage from one horizontal layer to the next. A layer may hold a single specialist or several running in parallel against the same accumulated context for that track. When only one specialist occupies a layer, its output becomes the track’s running brief and flows straight into the next stage. When more than one specialist shares a layer, the track does not jump immediately to the next specialist layer; instead, the runtime routes those parallel completions through an automatic “consolidator”. The Consolidator agent is a non-specialist agent that fuses the outputs of the previous layer’s agents into one coherent markdown block. This block subsumes the prior context for everything downstream. After every track has finished its pipeline, the runtime performs a cross-track merge that folds the per-track upstream briefs into one shared consolidated block, and the reporter then takes that merged material and the original user goal to author the final integrated response. A diagram of the prototype’s operation is given in Figure 1. During automated discussion, the runtime keeps a structured audit trail alongside the conversational flow. Each graph node is annotated with its execution inputs and outputs, and every specialist tool call is recorded under a shared trace identifier including arguments, timing, content hashes and bounded text previews, and URLs harvested from results. This audit trail is designed to support provenance-related queries after automated discussion. Together with the LLM transcript, which might convey rationale for tool selection, this audit trail provides a ledger of sorts for those scrutinizing the provenance of reported statements and quantities. 4Towards Multi-Agent Autonomous Reasoning in Hydrodynamics

# 4 Performance Assessment 

We next quantify the prototype, turning from the qualitative demonstrations above to end-to-end benchmarks, component ablations, stress tests, and cost analysis. We assess the MAS prototype along three axes: end-to-end accuracy against authoritative federal ground truth, component-wise ablation, and robustness under stress. All experiments used Claude Sonnet 4.6 as the backbone LLM for both the specialist and general-purpose agents. It is noteworthy that no model code was modified for any ablation; all variants were implemented via runtime function interception that bypassed the targeted component while leaving the remainder of the pipeline intact.

4.3 Ablation Study 

To determine whether each architectural component contributed measurably, we ran three ablations on our seven-query representative subset (one per complexity category, listed in Table A.4). In each variant, we disabled exactly one component and left everything else running normally. 

4.3.1 Fixed Graph 

We replaced the Graph Architect with a static LEG that sent every query through the same pipeline: [NHC] → [NOAA CO-OPS, USGS] → [FEMA] . This is essentially how a rigid, programmatic scientific workflow would behave. Factual Precision dropped by 28.7 pp as the static topology could not decompose multi-part queries into parallel tracks, so agents ended up juggling disparate data simultaneously. Additionally, irrelevant specialists (e.g., FEMA for a pure surge query) diluted the context available to the Reporter . Agent F1 fell by 20.1 pp, and, understandably, Topology Selection dropped by 28.6 pp. This confirmed that adaptive, query-specific LEG planning is the single most consequential architectural decision. 

4.3.2 No Consolidation 

The LEG was scheduled normally, but we replaced per-layer and cross-track context compaction with raw text concatenation. The Reporter , therefore, received unfiltered specialist outputs. Factual Precision fell by 4.6 pp, meaning the Consolidator ’s structured merge genuinely helped the Reporter extract correct values from multi-source data. 4.4 Stress Test 

We designed four complementary stress tests, reported in Table 3, to probe the robustness boundaries of the MAS prototype beyond the conditions covered by the end-to-end benchmark. 

4.4.1 Complexity Scaling 

We wanted to assess how accuracy and latency change as a single query grows more complex. Specifically, we increased the number of independent sub-questions from one to five. Data for one through three sub-questions came from the end-to-end benchmark ( n = 25 , 7, 5); we constructed two additional queries for four- and five-track LEGs. Accuracy held above 90% across all complexity levels (range: 90.6–95.0%), and latency scaled at roughly ∼30 s per additional track, consistent with parallel specialist execution plus fixed cross-track merge overhead. In short, the LEG architecture did not introduce compounding degradation as query complexity grew. 

4.4.2 Paraphrase Robustness 

We suspected the Graph Architect might be sensitive to how a query is phrased, so we rephrased each of four queries two additional ways ( 3× per group, 12 runs total), Table A.5. Topology was unaffected: every paraphrase produced the same LEG structure. The specialist set was more brittle as only 50% (2/4) of paraphrases triggered the complementary-agent expansion heuristic, so terse phrasings sometimes omitted a specialist. Despite this, the factual score standard deviation across paraphrases was only σ = 1 .2% , so answer quality remained stable even when the specialist set varied slightly. 

4.4.3 Adversarial Robustness 

We tested whether the system would hallucinate answers to impossible or misleading queries. We ran five adversarial cases: a storm–location mismatch (Hurricane Sandy at Galveston), an inland location (Denver), a nonexistent storm 11 Towards Multi-Agent Autonomous Reasoning in Hydrodynamics (Hurricane Bob in 2008), an incorrect category (Category 5 Ike), and an inland FEMA query (Omaha). These queries are listed in Table A.6. The system never fabricated data for impossible scenarios (5/5 hallucination-free). It detected 4 of the 5 problematic premises; for Denver it returned a minimal response rather than an explicit rejection, but generated no spurious data. Regarding specifying the incorrect category, the prototype noted “Ike made landfall as a Category 2. . . ” and then still answered the underlying surge question. Here, the system reasoned about factual consistency rather than blindly executing the query as stated. 

4.4.4 Fault Tolerance 

How does the prototype behave when a federal API goes dark? We simulated outages by forcing all tool calls for one specialist to return structured error messages. All three scenarios (NOAA CO-OPS down, USGS down, FEMA down) produced a substantive partial answer (3/3) without crashing (3/3), drawing on whatever data sources remained. In two of three cases, the system explicitly noted the missing data in its response. This graceful degradation stems directly from the LEG architecture: each specialist operates within its own tool allowlist and execution context, so one agent’s failure does not propagate to the others. 

4.5 Computational Cost 

We also examined the token consumption and API cost of the system, using the same seven-query representative subset from the ablation study (Table 2), measured at published Claude Sonnet 4.6 rates ($3.00 per million input tokens, $15.00 per million output tokens). They are given in Table 4. Single-agent queries cost just 15–17K tokens at $0.06 each, while multi-agent and parallel-track queries ran from 56K to 136K tokens ($0.19–$0.45). The costliest query was L03 (Katrina at Grand Isle), which involved a gauge data gap that forced the NOAA CO-OPS specialist into extra tool rounds searching for alternative stations and date ranges. Extrapolating to the full 37-query benchmark, we estimate a total cost of $9.85 and roughly 2.8M tokens. We then considered how tokens were distributed across pipeline stages (Table 5). Specialist agents dominated at 92.2% of all tokens which is not surprising given their tool-calling responsibilities. The Graph Architect consumed only 4.4%, and the Consolidator and Reporter together took just 3.4%. In other words, the LEG orchestration overhead is minimal relative to the specialist work it coordinates. 

# 5 Conclusions 

Herein, we have proposed a MAS prototype for hydrodynamics, one that automates the planning and execution of scientific workflows. While our prototype is relatively simple, it can be readily extended to a range of subdomains in the geosciences invoking observation, hypothetical, and/or forecast spatiotemporal data. Benchmarks, ablations, and stress tests indicate that the MAS is both accurate and resilient, and we expect this performance to carry over into other applications beyond storm surge. The MAS prototype is a shift away from programmatic workflow management strategies. The Graph Architect 

agent is charged with creating the LEG by invoking routing heuristics which encode best practices. They include splitting mixed-data-class layers, reordering observation specialist agents relative to storm track context, and expanding under-specified queries with complementary data sources. Rather than encoding these decisions as hard-coded rules in a workflow engine, the heuristics are expressed as natural-language guidance that the LLM interprets and applies in context. This allows the same pipeline infrastructure to produce different execution graphs depending on the query. Moreover, it gives the Graph Architect agent the ability to deviate from naive orderings when the query demands it, while still constraining outputs to valid structures. In addition to routing heuristics, our experience indicates that thoughtful and concise agent-based system prompts are necessary to achieve accurate and agile tool calling. However, we emphasize that these prompts are simply cues. By over-prompting the agents during development, we noted that they became less resilient tool callers because they were seemingly apprehensive to break from the prompted guidance. Consequently, we advocate for a “less is more” approach to system prompts. Moreover, data provenance cannot be overlooked as the architecture of MAS acts as a double-edged sword. On one hand, distributing context across multiple specialized agents is a distinct advantage that significantly reduces the likelihood of hallucination. On the other hand, because these systems invoke numerous agents wherein each is an expert tool-caller in its respective domain, they may chain or call tools in highly unexpected ways. While these execution paths might be technically correct, the underlying logic can be opaque and confusing to the end user. This is precisely why robust data provenance is essential: it provides the necessary transparency to trace how and why these independent agents arrived at their conclusions. For the MAS prototype, we are apprehensive to claim that the provenance logging strategy is optimal. Future studies should consider data provenance in MAS more thoroughly, specifically developing a “Goldilocks” level of provenance that works synergistically with agents to maximally infer context and enhance the explainability of the MAS output. This is motivated by our ablation studies wherein we demonstrated that the prototype’s consolidation scheme (i.e., context compaction) was perhaps too aggressive and compromised the

Finally, the MAS prototype lacks agentic oversight and consequently redundancy. If a specialist agent struggles in any way, neither is its output flagged for downstream agents nor is it given a chance to re-fire. For example, if an upstream agent does not provide necessary data for a downstream agent, the downstream agent will likely not be productive, and this will be apparent in its output. An “overseer” agent could monitor the output of agents, and in the case of a struggling downstream agent, could force the upstream agent to re-fire (with the added context of its past output and that of the downstream agent, as well). This feedback mechanism is currently missing from the prototype, but it could certainly be incorporated as a maturation activity to enhance redundancy.
