## 摘要
随着大语言模型（LLMs）越来越多地辅助安全软件开发，它们满足 Rust 程序验证严苛需求的能力仍不明确。现有的评估将 Rust 验证视为黑盒，仅通过证明提示（proof hints）的二元通过或失败结果来评估模型。这掩盖了模型是否真正理解验证非平凡 Rust 代码所需的逻辑推导。为了弥补这一差距，我们引入了 VCoT-Lift，这是一个将底层求解器推理提升为高层、人类可读验证步骤的框架。通过将求解器层面的推理暴露为显式的验证思维链（Verification Chain-of-Thought），VCoT-Lift 为细粒度评估提供了具体的基准真相。利用 VCoT-Lift，我们推出了 VCoT-Bench，这是一个包含 1,988 个 VCoT 补全任务的综合基准，用于严格评估 LLMs 对整个验证过程的理解。VCoT-Bench 从三个正交维度衡量性能：对不同程度证明缺失的鲁棒性、对不同证明类型的胜任力，以及对证明位置的敏感性。对十个最先进模型的评估揭示了严重的脆弱性，表明当前的 LLMs 远未达到自动定理证明器所展现的推理能力。

## Abstract
As Large Language Models (LLMs) increasingly assist secure software development, their ability to meet the rigorous demands of Rust program verification remains unclear. Existing evaluations treat Rust verification as a black box, assessing models only by binary pass or fail outcomes for proof hints. This obscures whether models truly understand the logical deductions required for verifying nontrivial Rust code. To bridge this gap, we introduce VCoT-Lift, a framework that lifts low-level solver reasoning into high-level, human-readable verification steps. By exposing solver-level reasoning as an explicit Verification Chain-of-Thought, VCoT-Lift provides a concrete ground truth for fine-grained evaluation. Leveraging VCoT-Lift, we introduce VCoT-Bench, a comprehensive benchmark of 1,988 VCoT completion tasks for rigorously evaluating LLMs' understanding of the entire verification process. VCoT-Bench measures performance along three orthogonal dimensions: robustness to varying degrees of missing proofs, competence across different proof types, and sensitivity to the proof locations. Evaluation of ten state-of-the-art models reveals severe fragility, indicating that current LLMs fall well short of the reasoning capabilities exhibited by automated theorem provers.

#### 2. 方法论：VCoT-Lift 框架
VCoT-Lift 是一个将底层 Z3 证明轨迹提升（Lift）为高层 Verus 验证步骤的框架，包含四个阶段：
*   **证明转换器（Proof Transformer）**：利用 LLM 将 Z3 的底层规则（如单位归结、重写等）转换为 Verus 级别的断言、引理和循环不变式。为了处理长上下文，引入了 **Z3 规则分级体系**（高、中、低三层重要性）。
*   **证明检查器（Proof Checker）**：通过“转换器-检查器循环”确保完整性。检查器将 Z3 证明划分为语义块，并要求 LLM 建立 Z3 步骤与 Verus 步骤之间的显式映射。
*   **证明剪枝器（Proof Pruner）**：移除冗余和琐碎的推理步骤（如 $x=x$ 或重复断言），以提高证明的简洁性。
*   **证明修复器（Proof Repair）**：通过与 Verus 验证器交互，利用编译器反馈的错误信息自动修复语法和语义错误，确保最终生成的 VCoT 是可验证的。

#### 3. 实验设计
*   **数据集**：基于 Verus-Bench（包含 150 个经过验证的 Rust 程序），通过 VCoT-Lift 生成了包含丰富推理细节的 **VCoT-Bench-Org**。

*   **评估维度**：

#### 6. 主要结论与发现
*   **推理脆弱性**：LLM 表现出严重的脆弱性。当缺失 10% 证明时表现尚可，但当缺失比例达到 40% 时，逻辑连续性崩溃，性能大幅下降。
*   **依赖脚手架**：LLM 并非从第一性原理出发进行推理，而是高度依赖现有的语法“脚手架”和局部模式匹配。
*   **中间推理最难**：模型在处理证明“中间部分”的连接性推理时表现最差，难以追踪程序状态的演变。
*   **证明类型差异**：断言（Assertion）的补全最难，因为其对状态特定逻辑的要求极高；循环不变式虽然相对容易，但最能区分模型能力的优劣。
*   **规模并非万能**：虽然大模型通常更强，但在中间步骤推理上，较小的模型（如 Gemini 3 Flash）有时反而优于更大的模型。

#### 7. 优点
*   **视角创新**：首次提出“验证思维链”（VCoT）概念，将 Rust 验证从结果导向转为过程导向。
*   **工具实用**：开发了 VCoT-Lift 框架，解决了 Z3 证明不可读的问题，为形式化验证领域提供了高质量的数据生成工具。
*   **维度全面**：VCoT-Bench 的三个评估维度（比例、类型、位置）能够非常细致地刻画 LLM 推理能力的短板。

#### 8. 不足与局限
*   **语言局限性**：Verus 作为一个相对较新的框架，其公开训练数据较少，可能限制了 LLM 的原生表现。
*   **小程序偏差**：在 100% 缺失比例下，准确率略有回升，这是因为该子集中包含较多简单的小程序，存在一定的统计偏差。
