Abstract 

Multi-step tool orchestration, where LLMs must invoke multiple dependent APIs in the correct order while propagating intermediate outputs, re-mains challenging. State-of-the-art models fre-quently fail on full sequence execution, with pa-rameter value errors accounting for a significant portion of failures. Training models to handle such workflows faces two obstacles: existing en-vironments focus on simple per-turn function calls with simulated data, and binary rewards provide no signal for partial correctness. We present a framework addressing both chal-lenges. First, we construct a reinforcement learn-ing environment backed by a large-scale cache of real API responses, enabling a data synthe-sis pipeline that samples valid multi-step orches-tration traces with controllable complexity and significantly higher generation efficiency than un-constrained methods. Second, we propose a grad-uated reward design that decomposes correctness into atomic validity (individual function call cor-rectness at increasing granularity) and orchestra-tion (correct tool sequencing with dependency respect). On ComplexFuncBench, our approach demonstrates substantial improvements in turn accuracy. Ablation studies confirm both reward components are essential: using either alone sig-nificantly degrades performance.

While signifi-cant progress has been made on single-step tool invocation, real-world applications often require multi-step tool orches-tration, i.e., invoking a sequence of APIs in the correct order while propagating outputs from earlier calls as inputs to subsequent ones. Recent benchmarks reveal that even state-of-the-art mod-els struggle with multi-step orchestration. On Complex-FuncBench (Zhong et al., 2025), which averages 3.26 steps per query, parameter value errors dominate failure modes (up to 78.8% for Qwen2.5-72B), indicating that models often select the correct function but fail to infer precise pa-rameter values. Similarly, NESTFUL (Basu et al., 2025) reports that the best models achieve only 28% full sequence accuracy on nested function calls, with performance degrad-ing sharply at depth ≥ 2.These findings suggest two key insights: (1) models need fine-grained feedback that rewards partial correctness rather than the standard all-or-nothing evaluation; and (2) training on multi-step workflows with explicit dependency handling is essential. Existing RL-based tool-use frameworks, such as ToolRL (Qian et al., 2025) and PARL-MT (Chai et al., 2025), primarily rely on terminal success signals or binary execution feedback. This sparsity is particularly problematic in orchestration tasks, where a single incorrect parameter in a multi-step sequence leads to total failure, providing no signal for the model to learn from partially correct reasoning paths. Applying reinforcement learning (RL) to multi-step tool orchestration faces two fundamental obstacles. Chal-lenge 1 concerns the lack of suitable training environments. While execution environments exist for tool learning (e.g., BFCL (Patil et al., 2025), tau-bench), they primarily target multi-user-turn scenarios with simple per-turn function calls rather than complex multi-step orchestration where a single user query requires multiple dependent API calls. Moreover, these environments often rely on simulated or synthetic data rather than real API responses, which can produce inconsis-tent dependency chains (e.g., a hotel ID returned in step 1 may not yield valid results when queried in step 2). 2.1. Problem Formulation 

We formulate multi-step tool orchestration as a sequential decision-making problem. Given a user query q and a set of available tools F, the agent must generate a sequence of actions y = ( y1, . . . , y T ), where each action yt = ( ft, θt)

consists of a function selection ft ∈ F and parameter as-signment θt.The execution environment is deterministic. Executing yt

yields an observation ot = Exec (ft, θt). Crucially, a subse-quent action yt′ (where t′ > t ) may possess a dependency 

on previous outputs. This is modeled as θt′ ← ϕ(o1: t′−1),where ϕ represents the ground truth logic for extracting and propagating values (e.g., extracting a hotel id to query room details). The agent must learn to approximate this dependency logic. Our objective is to learn a policy πθ (y|q) that maximizes a reward R(y, y∗), where y∗ represents the ground truth workflow. 2.2. Deterministic Training Environment 

Training LLMs for multi-step tool orchestration via RL requires a suitable execution environment. Existing ap-proaches like BFCL (Patil et al., 2025) or τ -bench (Yao et al., 2024) often rely on simulated responses or lack com-plex intra-turn dependencies. We address this by construct-ing a deterministic environment grounded in real data. Our construction process follows three steps: defining workflow logic, collecting targeted data, and enabling constrained synthesis. • Workflow Template Curation. Before collecting data, we must define valid orchestration patterns. We formalize recurring patterns as Workflow Templates that capture: or-dered function sequences (f1, . . . , f T ), dependency struc-tures specifying how θt depends on previous observations 

o<t , and field paths for value extraction (implementing 

ϕ). We curate over 100 templates spanning five domains (a concrete JSON example is provided in Section C). • Workflow-Aware Cache Collection. To enable deter-ministic RL training without the latency and instability of live API calls, we build a cache-based environment that implements ot = Exec (ft, θt). Crucially, we em-ploy workflow-aware collection : rather than caching ran-dom independent API calls, we execute complete action sequences (y1, . . . , y T ) from the templates against live Booking.com APIs. This ensures that for every cached observation ot, the corresponding dependent observations 

ot+1 , . . . that rely on values extracted from ot are also present. We collected 100k+ real API responses covering 40 func-tions in F. Responses are indexed by hash of (f, θ),enabling O(1) deterministic lookups. We implement com-prehensive input validation with 8 specialized validators (date formats, coordinate ranges, enum values) to return structured error messages matching real API behavior. • Constrained Data Synthesis. Rather than caching func-tion responses independently, we execute complete work-flow chains ( workflow-aware cache expansion ) to ensure cached responses form valid multi-step traces. The key insight is that freely exploring function chains leads to low data generation efficiency, as arbitrary combinations often violate dependency constraints or produce unsat-isfiable parameter bindings. Instead, we use workflow templates (pre-validated on live APIs) as skeletons, then sample cache entries step-by-step following the depen-dency structure, and finally prompt an LLM to synthesize a query q that would elicit the sampled action sequence

Two-Component Reward Structure. We decompose cor-rectness into two complementary components that capture different aspects: • Ratomic evaluates each predicted action yt = ( ft, θt),i.e., whether the model generates syntactically and semantically valid function calls. • Rorch evaluates workflow completion against y∗, i.e., whether all required steps are executed in the correct order with dependencies respected. 

Ratomic : Atomic Validity. For each predicted call, we assess correctness through two complementary validations: 

Abstract Syntax Tree (AST) Validation. We check syntactic correctness via three-level graduated scoring: • Level 1 ( α1): Correct function name • Level 2 ( α2): Correct parameter structure and types • Level 3 ( α3): Exact parameter value matching For a predicted call ˆy = ( ˆf , ˆθ) matched against ground truth y∗ = ( f ∗, θ∗):

RAST (ˆ y, y ∗) = α1 · 1[ ˆf = f ∗] + α2 · sstruct (ˆθ, θ∗)+ α3 · 1[ˆθ = θ∗] (1) where sstruct measures parameter overlap and type accuracy (detailed definitions in Section B). 

Semantic Validation. We execute the call through our de-terministic environment and check for success: Rsem (ˆ y) = 

1[execution succeeds ]. Concretely, we verify three condi-tions: (1) the call was executed and produced a response, (2) the response contains no error indicators (e.g., error fields or failure status), and (3) the response has valid struc-ture (non-empty with expected data fields). Note that this is execution-based validation rather than output matching: we reward successful execution regardless of whether the returned data exactly matches ground truth. This catches errors that pass AST checks but fail at runtime (e.g., invalid date ranges, non-existent IDs).

We conduct ablation studies to understand the contribution of each reward component (Section 3.4) and analyze how performance scales with data quantity (Section 3.5). We evaluate on ComplexFuncBench (Zhong et al., 2025), a benchmark specifically designed to assess multi-step tool orchestration. The benchmark comprises 1,000 test samples spanning five API domains (hotels, flights, car rentals, attrac-tions, and taxis), with an average of 3.26 steps per query. Zero-shot refers to the original model without any task-specific training: the model receives a system prompt with tool descriptions and the user query, without in-context ex-amples or fine-tuning.

Before evaluating generalization, we first verify that our cache-based RL environment can effectively improve model performance. This requires isolating the effect of the execu-tion environment from confounding factors such as distribu-tion shift between training and test data. 

Oracle Setting. We design an “oracle” experiment where models are trained and evaluated on the same 100 queries sampled from ComplexFuncBench. This setup eliminates generalization requirements: if the model fails to improve, the issue lies with the RL environment or reward signal rather than data coverage. Conversely, substantial improve-ment validates that the environment provides useful learning signals. These results demonstrate that our deter-ministic cache-based environment provides the consistent feedback necessary for effective policy optimization. 3.3. Effectiveness of Constrained Data Synthesis 

Having validated that the RL environment enables learning (Section 3.2), we now evaluate generalization : can models trained on synthetic data solve unseen queries? 

Generalization Setting. Unlike the oracle experiment, training and evaluation sets are now disjoint . We gener-ate synthetic training data using the constrained synthesis pipeline (Section 2.2), varying the number of samples per workflow template (3 or 10). Models are evaluated on the full 1000-sample ComplexFuncBench test set. We compare three training strategies: supervised fine-tuning (SFT) on ground-truth trajectories, GRPO with our graduated rewards, and SFT followed by GRPO. 

Results. Table 2 shows several observations: (1) SFT alone yields negligible gains or even degradation. Our error analysis (Table 5) reveals the root cause: SFT increases func-tion selection errors at the sequence level (12.9% →28.3% samples stopped after function error) and worsens query parameter extraction (14.7% →15.8% error rate), suggest-ing that imitation learning on limited templates does not generalize well to novel query phrasings. Per-domain analy-sis (Table 7) corroborates this: SFT overfits to a subset of domains while degrading on others. (2) GRPO with fewer samples per template (GRPO-3) sometimes outperforms more samples (GRPO-10), though results vary across mod-els and learning rates. (3) Qwen2.5-7B shows the largest gain with SFT+GRPO-3 combination (+13.7% over base-line). These results suggest that for strong base models, direct RL training may be more effective than SFT when data is scarce.

3.8. Error Analysis 

To understand where errors occur, we categorize failures at three levels: (1) call-level —among function calls the model produces, what fraction have incorrect function names vs. in-correct parameters; (2) parameter-level —among calls with correct function names, comparing errors in query-derived parameters vs. dependency-propagated parameters; and (3)

4. Related Work 

Tool Learning Benchmarks. Early benchmarks primar-ily focused on API retrieval and single-step function call-ing. Recently, comprehensive evaluations like BFCL (Patil et al., 2025) and APIBench/Gorilla (Patil et al., 2024) have standardized tool-use assessment. However, these often tar-get multi-turn conversational scenarios with simple per-turn calls. Our work specifically targets multi-step tool orchestra-tion, where a single query necessitates complex, dependent API sequences. We evaluate on ComplexFuncBench (Zhong et al., 2025), which identifies parameter value errors as the dominant failure mode—a finding that motivates our gradu-ated reward design. Additionally, NESTFUL (Basu et al., 2025) and Seal-Tools (Wu et al., 2024) highlight the sharp performance degradation in nested function calls (depth 

≥ 2), further validating the need for the specialized depen-dency handling we propose.

We extend this philosophy to the training phase by constructing a deterministic, cache-based environment with 100k+ real responses, enabling the intensive trial-and-error exploration required for RL training on complex multi-step trajectories. 

Training Data Generation. Generating high-quality multi-step traces is a significant bottleneck. ToolAl-paca (Tang et al., 2023) utilized multi-agent simulation, while ToolLLM (Qin et al., 2023) scaled to 16k APIs. More recent approaches like APIGen-MT (Prabhakar et al., 2025) employ LLM committees for multi-stage verification. Our constrained data synthesis pipeline achieves comparable suc-cess rates using a single LLM call by leveraging workflow templates as structural skeletons. 

Reinforcement Learning for Tool Use. Standard RL ap-proaches suffer from sparse, binary rewards. ToolRL (Qian et al., 2025) demonstrated PPO for single-step selection; ReTool (Feng et al., 2025) applied iterative fine-tuning but focused on math/code rather than API orchestration; RA-GEN (Wang et al., 2025) and PARL-MT (Chai et al., 2025) identify the sparse reward issue but remain reliant on binary success signals. Our approach addresses this by introduc-ing a graduated reward design, decomposing rewards into 

Ratomic (structural validity) and Rorch (dependency fulfill-ment) to provide dense learning signals. 

5. Conclusion 

We presented a framework for training LLMs on multi-step tool orchestration: a deterministic cache-based environment with 100k+ real API responses, a workflow-aware data syn-thesis pipeline, and a graduated reward design combining 

Ratomic and Rorch . Experiments demonstrate substantial im-provements in turn accuracy, with ablations confirming both reward components are essential. 8Training LLMs for Multi-Step Tool Orchestration 

Impact Statement 

This paper improves the reliability of LLMs in multi-step tool orchestration. As LLMs are integrated into autonomous agents executing real-world actions, ensuring they adhere to dependency constraints and parameter validity is crucial for safety. While our work reduces risks of incorrect API calls, autonomous agents carry inherent societal implications. Our constrained data synthesis and graduated rewards aim to provide more robust safeguards, contributing to safer agentic systems.
