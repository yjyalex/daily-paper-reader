# EMPIRICAL COMPARISON OF AGENT COMMUNICATION 

# PROTOCOLS FOR TASK ORCHESTRATION 

Dobrovolskyi I. – MS in AI & ML Engineering, Staff Software Engineer, Walmart Global Tech, Sunnyvale, CA, USA. ORCID: https://orcid.org/0009-0005-6938-0384. 

ABSTRACT 

Context. Nowadays, artificial intelligence agent systems are transforming from single-tool interactions to complex multi-agent orchestrations. As a result, two competing communication protocols have emerged: a tool integration protocol that standardizes how agents invoke external tools, and an inter-agent delegation protocol that enables autonomous agents to discover and delegate tasks to one another. Despite widespread industry adoption by dozens of enterprise partners, no empirical comparison of these protocols exists in the literature. 

Objective. The goal of this work is to develop the first systematic benchmark comparing tool-integration-only, multi-agent delegation, and hybrid architectures across standardized queries at three complexity levels, and to quantify the trade-offs in response time, context window consumption, monetary cost, error recovery, and implementation complexity. 

Method. An open-source health analyzer application was developed to query four public data sources for repository statistics, package downloads, vulnerability data, and community metrics. Thirty queries at three complexity tiers were each executed five times per architecture, yielding several hundred total executions with zero failures. All runs used the same large language model. Statistical analysis employed bootstrap confidence intervals, the Mann–Whitney test with Bonferroni correction, and Cliff’s delta effect sizes. 

Results. A statistically significant crossover effect was discovered: the tool integration protocol is faster for simple queries, but the multi-agent protocol outperforms on complex multi-project comparisons while consuming substantially fewer tokens due to distributed context windows, resulting in markedly lower cost. The hybrid architecture tracks the winner at each complexity level with near-optimal performance. 

Conclusions. The scientific novelty lies in the first empirical demonstration that the two protocols exhibit a complexity-dependent performance crossover, proving they are complementary rather than competing. For the first time, the phenomenon of context window bloat in single-agent architectures has been quantified. The practical significance is a validated decision framework: use the tool integration protocol for single-source queries, the multi-agent protocol for complex orchestration, and hybrid routing when complexity is unknown at runtime. Directions for further research include extending the benchmark to additional models, distributed deployments, and additional application domains. 

KEYWORDS: agent communication protocols, Model Context Protocol, Agent-to-Agent protocol, multi-agent systems, large language model orchestration, benchmark, tool use.

INTRODUCTION 

The rapid evolution of large language model (LLM) 

agent systems has created a fundamental architectural 

question: how should agents communicate with tools and 

with each other? Two protocols have emerged as de facto 

standards in 2025–2026. 

Model Context Protocol (MCP), introduced by 

Anthropic [1], standardizes how LLM agents invoke 

external tools. An agent connects to MCP servers via 

stdio or HTTP transport, discovers available tools, and 

calls them within a single context window. As of March 

2026, MCP SDKs have surpassed 97 million monthly 

downloads [2]. 

Agent-to-Agent Protocol (A2A), introduced by 

Google [3], enables autonomous agents to discover, 

communicate with, and delegate tasks to other agents over 

HTTP using JSON-RPC. Each agent publishes an Agent 

Card at /.well-known/agent.json for capability discovery. 

A2A has been adopted by over 50 enterprise partners [4]. 

Despite widespread adoption, the only existing 

comparison is a theoretical analysis by Chen et al. [5], 

which does not include empirical measurements. 

Practitioners choosing between these protocols must rely 

on intuition rather than data. 

The object of study is the process of communication 

between LLM agents and external tools or other agents 

during task orchestration. The subject of study is the comparative performance 

characteristics of MCP, A2A, and Hybrid communication 

architectures across varying levels of query complexity. 

The purpose of the work is to develop the first 

empirical benchmark for comparing agent communication 

protocols and to identify a decision framework for 

protocol selection. To achieve this goal, the following 

tasks were solved: to design three equivalent architectures 

(MCP-only, A2A multi-agent, Hybrid) implementing the 

same application logic; to define 30 standardized queries 

at three complexity levels; to execute 450 controlled 

experiments and collect six performance metrics; to apply 

nonparametric statistical tests to quantify the significance 

and magnitude of observed differences; and to formulate a 

practical protocol selection framework based on query 

complexity. 

1 PROBLEM STATEMENT 

Given a set of natural language queries Q = { q₁, q₂, …, 

q₃₀ } classified into three complexity levels (simple, 

medium, complex), and three architectural configurations 

A ∈ {MCP, A2A, Hybrid}, the objective is to determine 

for each complexity level which architecture minimizes a 

composite performance vector P = (latency, tokens, cost) 

subject to correctness constraints. 

Formally, for each complexity tier c ∈ {simple, 

medium, complex}, the goal is to identify A* = argminₐ 

f(Pₐ, c), where f is a weighted objective function. The 

constraints require zero execution failures (reliability = 

1.0) and zero hallucinations in the output. 

The input variables are the 30 standardized queries 

and the four public API data sources (GitHub, npm, 

OSV.dev, StackOverflow). The output variables are six 

metrics: wall-clock latency, total LLM tokens, cost in 

USD, LLM API calls, data source API calls, and code 

complexity. The quality evaluation criteria are statistical 

significance (Mann–Whitney U test) and effect size 

magnitude (Cliff’s delta). 

2 REVIEW OF THE LITERATURE 

Tool augmentation of LLMs was formalized by Schick 

et al. [6] with Toolformer, which demonstrated that 

language models can learn to invoke external tools during 

text generation. Patil et al. [7] extended this with Gorilla, 

enabling API-level tool invocation with improved 

accuracy. Yao et al. [8] introduced ReAct, which 

demonstrated the reasoning-action loop that underpins 

modern agent frameworks. These foundational works 

establish tool use capabilities but do not address 

inter-agent communication protocols. 

Multi-agent LLM orchestration has been explored 

through several frameworks. Wu et al. [9] proposed 

AutoGen for conversable multi-agent systems. Moura 

[10] introduced CrewAI for role-based agent 

orchestration. Hong et al. [11] developed MetaGPT for 

collaborative software engineering through multi-agent 

coordination. All three frameworks implement their own 

proprietary communication mechanisms but do not use 

standardized protocols such as MCP or A2A. 

MCP was released by Anthropic in November 2024 

[1] as an open standard for tool integration. Google 

introduced A2A in April 2025 [3] as an open protocol for 

inter-agent delegation. Chen et al. [5] provided the first 

theoretical comparison of the two protocols, noting their 

complementary nature, but did not include any empirical 

measurements. No previous study has benchmarked these 

protocols head-to-head with controlled experiments. The 

unsolved part of the problem is the absence of empirical 

data to guide practitioners in selecting between MCP and 

A2A for specific use cases. The present work addresses 

this gap. 

3 MATERIALS AND METHODS 

An Open Source Health Analyzer was built as the 

experimental application. It answers natural-language 

queries about open-source project health by pulling data 

from four public APIs: the GitHub REST API for 

repository statistics, the npm Registry for package 

downloads, OSV.dev for vulnerability data, and Stack 

Overflow for community metrics. The same application 

logic was implemented in three architectures described 

below. 

In Architecture A (MCP-Only), a single LLM agent 

connects to four MCP tool servers via stdio transport. The 

agent decides which tools to call, executes them within its 

tool-use loop, and accumulates all tool responses in a 

single context window (Fig. 1). 

Figure 1 – MCP-Only architecture: single agent with four 

stdio tool servers 

In Architecture B (A2A Multi-Agent), a coordinator 

agent delegates subtasks to four specialist agents over 

HTTP using the A2A protocol. Each specialist runs as an 

independent FastAPI server, publishes an Agent Card at 

/.well-known/agent.json, and processes tasks via 

JSON-RPC. The coordinator discovers agents, plans a 

delegation, dispatches them in parallel, and synthesizes 

the results. Each specialist maintains its own LLM 

context window (Fig. 2). Figure 2 – A2A Multi-Agent architecture: coordinator 

delegates to four independent HTTP agent servers 

In Architecture C (Hybrid), a routing layer classifies 

query complexity using keyword-based heuristics at zero 

LLM cost. Simple queries are routed directly through 

MCP; complex queries are delegated via A2A (Fig. 3). 

Figure 3 – Hybrid architecture: heuristic router selects 

protocol per query 

Thirty queries were defined across three complexity 

levels: ten simple queries (single-source, single-project), 

ten medium queries (multi-source, single-project), and ten 

complex queries (multi-source, multi-project 

comparison). Each query was executed 5 times per 

architecture, yielding a total of 450 executions. All 

executions used Claude Sonnet 

(claude-sonnet-4-20250514) to control for model 

variation. 

Six metrics were recorded for each execution: 

wall-clock latency in milliseconds, total LLM tokens 

(prompt plus completion), cost in USD at the rate of $3 

per million input tokens and $15 per million output 

tokens, number of LLM API calls, number of data source 

API calls, and code complexity measured in lines of code 

and cyclomatic complexity via AST analysis. 

For statistical analysis, 95% bootstrap confidence 

intervals were computed using 10,000 resamples for all 

means. Pairwise comparisons used the Mann–Whitney U 

test [13] with Bonferroni correction for nine simultaneous 

tests. Effect sizes were computed via Cliff’s delta [12], 

interpreted as negligible (|δ| < 0.147), small (< 0.33), 

medium (< 0.474), and large (≥ 0.474). 

4 EXPERIMENTS 

All experiments were conducted on a single machine 

with all A2A agents deployed on localhost to eliminate 

network variability. Each of the 30 queries was executed 

five times per architecture (MCP, A2A, Hybrid), 

producing 450 total executions. The four data source APIs 

(GitHub, npm, OSV.dev, StackOverflow) were queried 

live during each run; no caching was employed. 

Additionally, fault injection was applied by randomly 

turning off one API during selected runs to test error 

recovery. A hallucination detection module verified that 

all factual claims in the output could be traced to actual 

API response data. All 450 executions completed 

successfully with zero failures.

Figure 4 shows the latency crossover, which is the 

central finding. For simple queries, MCP (8.9 s) is 

significantly faster than A2A (19.8 s) with p < 0.0001 and 

a large effect (δ = −0.90). For medium queries, MCP 

(30.3 s) retains its advantage ( p = 0.0001, δ = −0.51). For 

complex queries, A2A (45.1 s) outperforms MCP (51.8 s) 

with a small-to-medium effect (δ = +0.29). The Hybrid 

architecture tracks the winner at each complexity level. 

Figure 4 – Latency crossover: MCP wins on simple and 

medium queries; A2A wins on complex queries (95% 

bootstrap CI shown as error bars) Token consumption explains the crossover. For 

complex queries, MCP consumes 34,959 tokens, which is 

3.1 times more than A2A (11,318 tokens; p < 0.0001, δ = 

+0.84). In the MCP single-agent architecture, every tool 

response is added to a single growing context window. 

For complex queries that require more than 22 API calls, 

this accumulation becomes the dominant cost driver. A2A 

distributes context across specialist agents, keeping each 

agent’s context lean. 

Cost follows token usage. All three architectures cost 

approximately $0.017 for simple queries. For complex 

queries, MCP costs $0.130 while A2A costs $0.079, 

representing a 39% reduction. One thousand complex 

queries would cost $130 with MCP versus $79 with A2A 

at enterprise scale. 

All pairwise comparisons at the simple and complex 

tiers show large effects (|δ| ≥ 0.474), confirming that the 

observed performance differences are not artifacts of 

sampling variability. 

Regarding code complexity, MCP requires 706 LOC 

across 11 files. A2A requires 1,530 LOC across 15 files, 

which is 2.2 times more. The Hybrid architecture requires 

1,899 LOC. The A2A overhead comes from HTTP server 

infrastructure (291 LOC) and per-agent boilerplate. 

Average cyclomatic complexity is 7.9 for MCP and 10.7 

for A2A. 

6 DISCUSSION 

Based on the findings, the following decision 

framework is proposed (Fig. 5). For queries involving a 

single data source or simple lookups, MCP should be used 

due to its zero-overhead advantage: no HTTP, no agent 

discovery, no delegation LLM call. For complex queries 

involving multiple projects and multiple data sources, 

A2A should be used because it distributes context across 

specialist agents, keeping each at approximately 2,263 

tokens versus 34,958 tokens in MCP’s single window. 

When query complexity is unknown at runtime, the 

Hybrid architecture with a lightweight heuristic router 

achieves near-optimal performance across tiers. 

Figure 5 – Decision framework for protocol selection 

based on query complexity 

Two opposing forces drive the crossover. MCP’s 

advantage on simple queries comes from zero overhead: 

no HTTP connections, no agent discovery, and no 

delegation LLM call. A2A’s advantage on complex 

queries comes from context isolation: MCP accumulates 

all tokens in a single window, whereas A2A distributes 

them across independent agents. 

These results imply that MCP and A2A are 

complementary rather than competing. MCP excels as a 

tool integration layer (agent-to-tool), while A2A excels as 

an orchestration layer (agent-to-agent). The A2A 

specialist agents in this study use direct API calls 

internally; in production, they could use MCP for their 

tool connections, combining both protocols. 

Comparing with the theoretical analysis by Chen et al. 

[5], which hypothesized a complementary relationship 

between the protocols, the present work confirms this 

hypothesis with empirical data and further identifies the 

specific crossover point driven by context window 

accumulation. 

Several limitations should be noted. All experiments 

use a single LLM model (Claude Sonnet); results may not 

generalize to other models. The A2A agents run on 

localhost; real network latency would increase A2A 

overhead and potentially shift the crossover point. Only 

one application domain (open-source project analysis) 

was tested. Five runs per query (50 per cell) provides 

good statistical power for large effects but limits the 

detection of small effects. Different system prompts could 

affect relative performance. 

CONCLUSIONS 

The urgent scientific problem of empirically 

comparing agent communication protocols, which was 

previously addressed only theoretically, is now solved. 

The scientific novelty of the obtained results is that a 

statistically significant, complexity-dependent crossover 

effect between MCP and A2A is first empirically 

demonstrated: MCP is faster for simple queries ( p <

0.0001), while A2A consumes 3.1 times fewer tokens for 

complex orchestrations ( p < 0.0001), reducing cost by 

39%. This allows practitioners to make evidence-based 

decisions about protocol selection rather than relying on 

intuition. The context window bloat phenomenon in 

single-agent architectures, previously discussed only 

qualitatively, was further developed: it was quantified for 

the first time as a 3.1-fold token overhead, increasing cost 

by 39% on complex queries. This allows architects to 

predict cost implications of protocol choice. 

The practical significance of the obtained results is a 

decision framework validated by 450 executions: use 

MCP for single-source queries, A2A for complex 

multi-project orchestration, and Hybrid routing for 

unknown complexity at runtime. The Hybrid architecture 

validates automatic routing with near-optimal 

performance at each complexity level. All code, 

benchmark data, and analysis tools are available at 

https://github.com/IvanDobrovolsky/mcp-vs-a2a-bench 

under the MIT license. 

Prospects for further research are to extend the 

benchmark to additional LLM models, to test on real 

distributed deployments with network latency, to evaluate 

additional application domains, and to develop adaptive 

routing that learns the crossover point from data.

АНОТАЦІЯ 

Актуальність. З розвитком агентних систем на основі великих мовних моделей виникли два конкуруючі протоколи комунікації: протокол інтеграції інструментів (MCP) та протокол міжагентної делегації (A2A). Незважаючи на широке впровадження в індустрії десятками партнерів, емпіричне порівняння цих протоколів відсутнє в літературі. 

Мета роботи – розробка першого систематичного бенчмарку для порівняння архітектур MCP, A2A та гібридної на стандартизованих запитах трьох рівнів складності та кількісна оцінка компромісів за часом відгуку, споживанням токенів, вартістю, відновленням після помилок та складністю реалізації. 

Метод. Побудовано додаток для аналізу стану проєктів з відкритим кодом, що звертається до чотирьох публічних джерел даних. Тридцять запитів трьох рівнів складності виконувалися по п’ять разів на кожну архітектуру, що дало кілька сотень виконань без жодного збою. Усі запуски використовували одну й ту ж мовну модель. Статистичний аналіз включав бутстреп-довірчі інтервали, критерій Манна–Вітні з корекцією Бонферроні та розмір ефекту дельта Кліффа. 

Результати. Виявлено статистично значущий перехресний ефект: протокол інтеграції інструментів швидший для простих запитів, але міжагентний протокол переважає на складних порівняннях, споживаючи значно менше токенів завдяки розподіленим контекстним вікнам, що призводить до нижчої вартості. Гібридна архітектура досягає майже оптимальної продуктивності на кожному рівні складності. 

Висновки. Наукова новизна полягає в першій емпіричній демонстрації того, що два протоколи демонструють залежний від складності перехресний ефект, доводячи їхню комплементарність. Практичне значення – валідована схема прийняття рішень: MCP для простих запитів, A2A для складної оркестрації, гібридна маршрутизація при невідомій складності. Перспективи подальших досліджень включають розширення бенчмарку на інші моделі, розподілені розгортання та додаткові предметні області.
