## 摘要
背景：如今，人工智能智能体系统正从单一工具交互向复杂的多智能体编排转型。因此，出现了两种竞争性的通信协议：一种是规范智能体如何调用外部工具的工具集成协议，另一种是使自主智能体能够相互发现并委派任务的智能体间委派协议。尽管数十家企业合作伙伴已在工业界广泛采用这些协议，但文献中尚不存在对这些协议的实证比较。目标：本研究旨在开发首个系统性基准测试，在三个复杂度级别的标准化查询中比较仅工具集成、多智能体委派以及混合架构，并量化在响应时间、上下文窗口消耗、货币成本、错误恢复和实现复杂度方面的权衡。

## Abstract
Context. Nowadays, artificial intelligence agent systems are transforming from single-tool interactions to complex multi-agent orchestrations. As a result, two competing communication protocols have emerged: a tool integration protocol that standardizes how agents invoke external tools, and an inter-agent delegation protocol that enables autonomous agents to discover and delegate tasks to one another. Despite widespread industry adoption by dozens of enterprise partners, no empirical comparison of these protocols exists in the literature. Objective. The goal of this work is to develop the first systematic benchmark comparing tool-integration-only, multi-agent delegation, and hybrid architectures across standardized queries at three complexity levels, and to quantify the trade-offs in response time, context window consumption, monetary cost, error recovery, and implementation complexity.

## 论文详细总结（自动生成）

这篇论文对当前大语言模型（LLM）领域两种主流的通信协议——**Model Context Protocol (MCP)** 和 **Agent-to-Agent Protocol (A2A)** 进行了首次系统性的实证比较。

### 1. 核心问题与研究动机
随着 AI 智能体从简单的单工具交互转向复杂的多智能体编排，开发者面临架构选择的难题：是让一个智能体通过工具集成协议（如 Anthropic 的 MCP）直接调用所有工具，还是让多个智能体通过委派协议（如 Google 的 A2A）相互协作？
尽管这两种协议在工业界已被广泛采用，但学术界此前仅有理论分析，缺乏量化的实验数据。本研究旨在填补这一空白，通过实证研究量化不同架构在响应时间、Token 消耗、成本及复杂度方面的权衡。

### 2. 论文提出的方法论
研究者构建了一个名为“开源健康分析器（Open Source Health Analyzer）”的实验应用，并实现了三种架构：
*   **架构 A (仅 MCP)：** 单个智能体通过 stdio 传输协议连接到 4 个 MCP 工具服务器。所有工具返回的数据都累积在同一个上下文窗口中。
*   **架构 B (A2A 多智能体)：** 一个协调智能体通过 HTTP/JSON-RPC 协议将子任务委派给 4 个独立的专家智能体。每个专家智能体拥有独立的上下文窗口。
*   **架构 C (混合架构)：** 使用基于关键词的轻量级启发式路由层。简单查询走 MCP，复杂查询走 A2A。
*   **核心逻辑：** 实验通过对比这三种架构在处理相同逻辑时的性能表现，寻找性能拐点。

### 3. 实验设计
*   **数据集/场景：** 针对 GitHub（仓库统计）、npm（下载量）、OSV.dev（漏洞）和 StackOverflow（社区指标）四个公共 API 进行实时查询。
*   **Benchmark：** 设计了 30 个标准化查询，分为三个复杂度等级：
    *   **简单：** 单一数据源，单一项目。
    *   **中等：** 多数据源，单一项目。
    *   **复杂：** 多数据源，多项目对比。
*   **对比指标：** 端到端延迟、总 Token 消耗、货币成本（美元）、API 调用次数、代码行数（LOC）及循环复杂度。

### 4. 资源与算力
*   **模型：** 所有实验均统一使用 `claude-sonnet-4-20250514` 模型，以消除模型差异。
*   **算力环境：** 实验在一台单机上运行，所有 A2A 智能体部署在 `localhost` 以消除网络波动干扰。
*   **训练时长：** 本研究属于推理侧的基准测试，不涉及模型训练。

### 5. 实验数量与充分性
*   **实验规模：** 每个架构对 30 个查询各执行 5 次，总计 **450 次受控实验执行**。
*   **统计严谨性：** 采用了 95% 布特斯特拉普（Bootstrap）置信区间、曼-惠特尼 U 检验（带博恩费罗尼校正）以及 Cliff's delta 效应量分析。
*   **充分性评价：** 实验设计较为充分，通过多次重复执行和非参数统计检验确保了结果的客观性，且在实验中加入了故障注入测试以验证错误恢复能力。

### 6. 主要结论与发现
*   **性能交叉效应（Crossover Effect）：** 
    *   在**简单查询**中，MCP 显著更快（8.9s vs 19.8s），因为其没有 HTTP 通信和委派开销。
    *   在**复杂查询**中，A2A 反超 MCP（45.1s vs 51.8s）。
*   **上下文膨胀（Context Window Bloat）：** 在复杂任务中，MCP 的单窗口模式导致 Token 消耗是 A2A 的 **3.1 倍**。
*   **成本优势：** 对于复杂任务，A2A 相比 MCP 可降低约 **39% 的成本**，因为它通过分布式上下文保持了每个智能体窗口的精简。
*   **实现复杂度：** MCP 的代码量（706 LOC）远低于 A2A（1530 LOC），A2A 存在明显的样板代码和基础设施开销。

### 7. 优点与亮点
*   **首次量化：** 第一次通过实验证明了 MCP 和 A2A 是**互补关系**而非竞争关系（MCP 适合工具集成，A2A 适合任务编排）。
*   **决策框架：** 提出了一个实用的决策流程图，指导开发者根据查询复杂度选择协议。
*   **开源贡献：** 提供了完整的基准测试工具和代码库，具有很高的工程参考价值。

### 8. 不足与局限
*   **模型单一：** 仅测试了 Claude Sonnet，结果可能无法完全推广到其他模型（如 GPT-4 或开源模型）。
*   **网络环境理想化：** 实验在 localhost 运行，忽略了真实生产环境中的网络延迟，这可能会进一步推迟 A2A 的性能优势拐点。
*   **领域局限：** 仅测试了开源项目分析这一特定领域，其他领域（如长文本处理或多模态任务）的表现尚不明确。
*   **样本量：** 虽然 450 次执行对于统计显著性足够，但对于捕捉极小概率的异常或长尾效应仍显不足。
