## 摘要
当多个基于大语言模型（LLM）的代码智能体独立实现同一个类的不同部分时，即使规范说明（specification）中未明确给出这些选择，它们也必须在共享的内部表示上达成一致。我们通过 51 个类生成任务研究了这一协作问题，逐步剥离规范细节（从完整的文档字符串 L0 到仅剩函数签名 L3），并引入相反的结构偏差（列表与字典）以对集成进行压力测试。研究得出三个发现：首先，存在持久的规范差距，随着细节的移除，双智能体集成准确率从 58% 下降到 25%，而单智能体基准的下降则更为平缓（从 89% 到 56%），导致了 25-39 个百分点的协作差距，且该结果在两个 Claude 模型（Sonnet、Haiku）和三次独立运行中保持一致。其次，一个基于 AST 的冲突检测器在最弱的规范级别下无需额外 LLM 调用即可达到 97% 的精确度，但析因恢复实验表明，仅恢复完整的规范说明就能达到单智能体的性能上限（89%），而提供冲突报告并无显著收益。第三，将差距分解为协作成本（+16 pp）和信息不对称（+11 pp）表明这两种效应是独立的且具有近似可加性。该差距不仅是隐藏信息的结果，更反映了在缺乏共享决策时生成兼容代码的难度。这些结果支持了多智能体代码生成的“规范优先”观点：更丰富的规范说明既是主要的协作机制，也是充分的恢复手段。

## Abstract
When multiple LLM-based code agents independently implement parts of the same class, they must agree on shared internal representations, even when the specification leaves those choices implicit. We study this coordination problem across 51 class-generation tasks, progressively stripping specification detail from full docstrings (L0) to bare signatures (L3), and introducing opposing structural biases (lists vs. dictionaries) to stress-test integration. Three findings emerge. First, a persistent specification gap: two-agent integration accuracy drops from 58% to 25% as detail is removed, while a single-agent baseline degrades more gracefully (89% to 56%), leaving a 25--39 pp coordination gap that is consistent across two Claude models (Sonnet, Haiku) and three independent runs. Second, an AST-based conflict detector achieves 97% precision at the weakest specification level without additional LLM calls, yet a factorial recovery experiment shows that restoring the full specification alone recovers the single-agent ceiling (89%), while providing conflict reports adds no measurable benefit. Third, decomposing the gap into coordination cost (+16 pp) and information asymmetry (+11 pp) suggests that the two effects are independent and approximately additive. The gap is not merely a consequence of hidden information, but reflects the difficulty of producing compatible code without shared decisions. These results support a specification-first view of multi-agent code generation: richer specifications are both the primary coordination mechanism and the sufficient recovery instrument.

### 论文总结：规范差距——代码智能体在部分知识下的协作失败

#### 1. 核心问题与整体含义
论文探讨了**多智能体代码生成中的协作失效问题**。当多个 LLM 智能体独立实现同一个类的不同方法时，它们必须对共享的内部状态（如构造函数中的数据结构）达成一致。然而，如果需求规范（Specification）不够明确，智能体往往会根据各自的偏好做出不兼容的选择（例如一个用列表，另一个用字典），导致集成失败。这种由于知识不完整导致的性能损失被称为“**规范差距（Specification Gap）**”。

#### 2. 方法论
研究采用受控实验方法，通过操纵“规范完整性”这一独立变量来观察其对集成成功率的影响：
*   **规范分级（L0-L3）：** 从包含完整文档字符串、示例和数据结构引用的 L0，逐步剥离信息，直到仅剩函数签名的 L3。
*   **偏差注入：** 为模拟现实中开发者的风格差异，给智能体 A 注入“偏好列表”的偏差，给智能体 B 注入“偏好字典”的偏差。
*   **AST 冲突检测器：** 开发了一种基于抽象语法树（AST）的检测器，无需额外 LLM 调用即可识别类型冲突（如 `append` vs `key-value` 赋值）和状态访问冲突。
*   **恢复实验（2x2 析因设计）：** 测试在修复集成错误时，是“提供更详细的规范”有效，还是“提供冲突检测报告”有效。

*   **数据集：** 基于 **ClassEval** 构建了 **AmbigClass** 基准测试，包含 51 个复杂的类生成任务（每个类至少 3 个方法）。

*   **对比模型：** 使用了 Claude 3.5 Sonnet 和 Claude 3.5 Haiku 两个模型进行交叉验证。

*   **实验规模：** 涵盖 51 个任务，每个任务有 4 个规范等级，并在两个模型上运行。为了验证稳定性，Haiku 模型进行了 3 次独立运行。

#### 6. 主要结论与发现
*   **规范决定成败：** 随着规范细节减少，多智能体集成准确率从 58% 骤降至 25%，而单智能体仅从 89% 降至 56%。
*   **持久的协作税：** 即使在最详细的规范下，多智能体比单智能体仍有约 25-39 个百分点的性能差距，且这种差距不随规范改善而消失。
*   **冲突检测的局限：** AST 检测器虽然能以 97% 的精度识别冲突，但将冲突报告提供给负责合并的智能体并不能提升修复效果。
*   **规范是唯一的救星：** 实验证明，**仅恢复完整规范**就足以让合并后的性能达到单智能体水平（89%），而冲突报告在有规范的情况下反而可能引入干扰。

#### 7. 优点与亮点
*   **系统性的方法论：** 提出了一套评估多智能体协作能力的标准化流程（规范消融 + 偏差注入 + 差距分解）。
*   **零成本诊断：** 证明了 AST 分析在诊断智能体协作问题上的高效性，无需昂贵的 LLM 推理。
*   **理论联系实际：** 将经典软件工程理论（Parnas 的信息隐藏、Meyer 的契约式设计）应用于现代 AI 智能体研究，具有很强的启发性。

#### 8. 不足与局限
*   **偏差形式单一：** 实验主要聚焦于“列表 vs 字典”的结构冲突，可能无法涵盖所有类型的协作错误（如复杂的逻辑依赖）。
*   **语言局限：** 实验仅针对 Python 语言，不同编程语言的类型系统（如强类型的 Java）可能会改变冲突检测的效果。

*   **人为注入偏差：** 实验中的偏差是通过 Prompt 强制注入的，虽然模拟了现实风险，但与智能体自发产生的风格差异可能存在细微区别。
