## 摘要
LLM多智能体系统通常通过自然语言对话或松散结构的共享内存进行协调，使得中间状态难以验证、归因和审计。我们提出PatchBoard，一种基于模式的协作架构，它用对共享结构化状态的经过验证的JSON Patch变异取代了智能体间的对话。一个架构智能体构建任务特定的模式和流程规则，而一个确定性内核在事务性地提交每个提议的状态变异之前，根据模式约束、角色特定的写入契约和运行时不变式对其进行验证。在630个匹配的ALFWorld剧集中，PatchBoard实现了84.6%的成功率，相比之下LangGraph为30.8%，Flock为61.6%，同时每个成功任务的令牌数减少到45.5k，而它们分别为368.3k和64.2k。

## Abstract
LLM multi-agent systems often coordinate through natural-language dialogue or loosely structured shared memory, making intermediate state difficult to validate, attribute, and audit. We introduce PatchBoard, a schema-grounded collaboration architecture that replaces inter-agent dialogue with validated JSON Patch mutations over a shared structured state. An Architect agent constructs a task-specific schema and workflow rules, while a deterministic kernel validates each proposed state mutation against schema constraints, role-specific write contracts, and runtime invariants before committing it transactionally. On 630 matched ALFWorld episodes, PatchBoard achieves an 84.6% success rate, compared with 30.8% for LangGraph and 61.6% for Flock, while reducing tokens per successful task to 45.5k, compared with 368.3k and 64.2k, respectively.

    - 若通过则事务性提交，更新全局状态 \(S_{t+1}\)；否则拒绝并记录（不影响已提交状态）。
    - 记录事务日志，支持重放和审计。

### 6. 论文的主要结论与发现

- **成功率**：PatchBoard在630个ALFWorld episodes中成功率84.6%（533/630），远超LangGraph（30.8%）和Flock（61.6%）。
- **token效率**：每个成功任务仅需45.5k token，远低于LangGraph（368.3k）和Flock（64.2k）。
- **黑板控制**：仅添加共享内存（Plain Blackboard或Structured JSON Blackboard）不足以达到相同成功率，PatchBoard的验证和写合约提供了额外收益。
- **消融**：移除补丁/模式接口和上下文切片导致最大效果下降（成功率降15%，token成本翻倍）。
- **灵敏度**：较小上下文预算（1k字符）表现更好；生成的（任务特定）模式优于固定模式。
- **故障鲁棒性**：PatchBoard对无效JSON、错误路径/类型、未授权写入的污染率为0%，循环终止率96%；但虚假声明（语义错误）仍可能通过模式验证（率43%）。
- **HotpotQA诊断**：所有系统准确率相近，但PatchBoard的未支持声明率非最低，说明模式验证保障结构性而非语义正确性。

### 7. 优点

- **结构化验证与审计**：将智能体输出强制为路径寻址的JSON Patch，所有更新经模式、授权、运行时不变式检查后提交，确保状态完整性。
- **确定性内核**：拒绝的补丁不污染共享状态，事务日志完全可重放，支持调试和归因。
- **上下文管理**：基于角色读合约和有界窗口，避免给工人不必要信息，减少token消耗和角色泄露。
- **故障隔离**：内置断路器检测循环、无效补丁等，可自动终止异常轨迹。
- **广泛实验**：不仅对比基线，还进行黑板控制、消融、灵敏度、故障注入、诊断，全面揭示设计贡献。
- **高效率**：每个成功任务token成本最低，推理调用更高效。

### 8. 不足与局限

- **语义正确性不保证**：模式验证仅保证结构合法，一个满足模式的声明仍可能是错误、不完整或无证据支持的。HotpotQA中未支持声明率仍较高。
- **蓝图依赖**：系统性能高度依赖Architect生成的蓝图质量。稀疏模式会阻碍进展，而过度宽松模式会削弱角色隔离。
- **任务限制**：最强证据来自ALFWorld（明确定义状态和成功条件），更开放的任务（如软件工程、科学发现）可能需要更复杂的模式与验证。
- **工程开销**：需要维护模式定义、上下文切片、补丁验证和日志，适用于对可审计性和状态完整性要求高的场景，简单任务可能过度设计。
- **伦理风险**：可审计性可能造成虚假安全感，用户不应将模式有效性等同于事实正确性。事务日志可能包含敏感信息，需访问控制和保密措施。
- **实验局限**：未在多个模型上验证（仅Qwen-plus），未评估长流程（如超过20步）或更大规模智能体团队的扩展性。
