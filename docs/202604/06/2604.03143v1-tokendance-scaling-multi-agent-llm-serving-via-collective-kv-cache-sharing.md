## 摘要
多智能体大语言模型（LLM）应用通常以同步轮次的方式组织执行，由中央调度器收集所有智能体的输出并重新分发合并后的上下文。这种全收集（All-Gather）通信模式产生了海量的 KV 缓存冗余，因为每个智能体的提示词都包含相同的共享输出块，而现有的重用方法无法高效地利用这一特性。我们提出了 TokenDance，该系统通过利用全收集模式进行集合式 KV 缓存共享，从而扩展了并发智能体的数量。TokenDance 的 KV 收集器（KV Collector）通过单个集合步骤在整个轮次内执行 KV 缓存重用，因此无论智能体数量多少，重用共享块的开销仅需支付一次。其差异感知存储（Diff-Aware Storage）将同级缓存编码为相对于单个主副本的块稀疏差异，在代表性工作负载上实现了 11-17 倍的压缩。在 GenerativeAgents 和 AgentSociety 上的评估表明，在满足服务水平目标（SLO）要求的前提下，TokenDance 支持的并发智能体数量比带有前缀缓存的 vLLM 高出 2.7 倍，将单个智能体的 KV 缓存存储降低了高达 17.5 倍，并比针对单个请求的位置无关缓存实现了高达 1.9 倍的预填充加速。

## Abstract
Multi-agent LLM applications organize execution in synchronized rounds where a central scheduler gathers outputs from all agents and redistributes the combined context. This All-Gather communication pattern creates massive KV Cache redundancy, because every agent's prompt contains the same shared output blocks, yet existing reuse methods fail to exploit it efficiently. We present TokenDance, a system that scales the number of concurrent agents by exploiting the All-Gather pattern for collective KV Cache sharing. TokenDance's KV Collector performs KV Cache reuse over the full round in one collective step, so the cost of reusing a shared block is paid once regardless of agent count. Its Diff-Aware Storage encodes sibling caches as block-sparse diffs against a single master copy, achieving 11-17x compression on representative workloads. Evaluation on GenerativeAgents and AgentSociety shows that TokenDance supports up to 2.7x more concurrent agents than vLLM with prefix caching under SLO requirement, reduces per-agent KV Cache storage by up to 17.5x, and achieves up to 1.9x prefill speedup over per-request position-independent caching.

### TokenDance：通过集合式 KV 缓存共享扩展多智能体 LLM 服务

#### 1. 核心问题与整体含义
论文针对多智能体 LLM 应用（如社会模拟、协作编程）中的 **KV Cache 冗余问题** 进行了深入研究。在这些应用中，通常遵循 **All-Gather（全收集）模式**：每一轮对话中，调度器收集所有智能体的输出，并将其作为共享上下文分发给下一轮的所有智能体。
*   **研究动机**：由于每个智能体拥有不同的私有历史（Private History），导致相同的共享输出块在不同请求中的绝对位置不同。现有的前缀缓存（Prefix Caching）因位置偏移失效，而位置无关缓存（PIC）虽能重用但需为每个请求重复计算，导致 GPU 显存迅速耗尽，限制了并发智能体的扩展规模。

#### 2. 方法论
TokenDance 提出将优化粒度从“单个请求”提升到“多智能体轮次”，核心技术包括：
*   **轮次感知提示词接口（Round-Aware Prompt Interface）**：在逻辑块之间插入特殊的隔离符 `<TTSEP>`，使系统能识别块边界，并采用基于段（Segment）的哈希索引，而非固定大小的块哈希，从而在私有历史长度不同时仍能识别共享块。
*   **集合式 KV 缓存重用（Collective KV Cache Reuse）**：将同一轮次内的 $N$ 个智能体请求分组。系统仅执行一次批处理的 RoPE 旋转和一次共享的“重要位置”选择分析，将重用开销从 $O(N)$ 降低到接近 $O(1)$。
*   **差异感知存储（Diff-Aware Storage）**：采用 **Master-Mirror（主从）布局**。仅存储一个完整的“主副本（Master）”，其余智能体的缓存则编码为相对于主副本的“块稀疏差异（Sparse Diffs）”。
*   **融合差异恢复（Fused Diff Restore）**：在数据从存储传输到 GPU 显存的流水线中，利用 Ping-pong 缓冲技术直接应用稀疏差异，避免了额外的内存读写开销。

#### 3. 实验设计
*   **数据集/场景**：使用了两个代表性的多智能体社会模拟框架：**GenerativeAgents**（斯坦福虚拟小镇）和 **AgentSociety**。
*   **模型**：Qwen2.5-7B 和 Qwen2.5-14B。
*   **Benchmark 与对比方法**：
    *   **vLLM**：带前缀缓存的工业级推理引擎。
    *   **CacheBlend (Ordinary)**：不带跨前缀重用的 CPU 侧缓存方案。
    *   **CacheBlend (Full PIC)**：目前最先进的、针对单请求的位置无关缓存重用方法。
*   **评估指标**：轮次延迟（Round Latency）、SLO 下的最大并发智能体数、KV Cache 存储占用、预填充加速比。

#### 5. 实验数量与充分性
*   **实验规模**：涵盖了不同模型规模（7B/14B）、不同工作负载（短历史 vs 长历史）、不同并发压力（1-20 个智能体）以及不同请求频率（QPS 1-16）。
*   **充分性**：实验设计较为全面，不仅验证了端到端的系统容量提升，还通过消融实验分别验证了计算侧（集合重用）和存储侧（差异存储）的收益，并进行了准确性（模型输出一致性）测试。对比基准涵盖了从工业标准到学术前沿的方法，实验结果具有较强的说服力和客观性。

#### 6. 主要结论与发现
*   **容量扩展**：在满足延迟 SLO 的前提下，TokenDance 支持的并发智能体数量比 vLLM 提升了 **2.7 倍**。
*   **存储压缩**：相比于为每个智能体存储独立缓存，TokenDance 将单个智能体的存储开销降低了 **94%**（在 14B 模型上实现 17.5 倍压缩）。
*   **性能加速**：预填充阶段相比于单请求 PIC 方法实现了高达 **1.9 倍** 的加速，端到端延迟降低达 **2.3 倍**。
*   **准确性**：TokenDance 不会引入超出底层 PIC 方法之外的额外精度损失，在多个场景下与基准模型输出保持高度一致。

#### 7. 优点
*   **应用模式驱动**：敏锐捕捉到多智能体应用中特有的 All-Gather 通信模式，并将其转化为系统层面的优化机会。
*   **计算与存储双重优化**：不仅解决了显存容量瓶颈，还通过集合计算降低了预填充的计算开销。
*   **高效的恢复机制**：Fused Restore 设计巧妙，避免了压缩数据解压带来的二次性能损失。

#### 8. 不足与局限
*   **模式依赖性**：该系统高度依赖于 All-Gather 这种同步轮次模式。如果多智能体应用采用完全异步或点对点的通信模式，其优化效果将大打折扣。
*   **精度风险**：由于底层依赖 PIC 技术（选择性重计算），在某些对数值精度极其敏感的任务中，可能会因为微小的 KV 差异导致长链推理后的结果漂移。
*   **硬件限制**：目前主要针对单机单卡环境，尚未详细讨论在多机分布式推理环境下的扩展表现。
