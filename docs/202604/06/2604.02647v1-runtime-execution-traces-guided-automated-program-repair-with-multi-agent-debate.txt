## Abstract 

Automated Program Repair (APR) aims to resolve software bugs without human intervention. However, handling complex logic errors, especially silent failures that produce incorrect outputs without any explicit crash signals, remains a significant challenge. Current LLM-based APR approaches attempt to address this issue through conversational feedback, retrieval-augmented generation, or static analysis tools. However, these methods are fundamentally static, relying only on source code and basic test outputs, and thus struggle to capture complex runtime behaviors and dynamic data dependencies accurately. Inspired by human debugging practice, incorporating runtime evidence into APR provides direct access to program behavior through execution traces, which expose concrete state transitions and data dependencies. However, even with such runtime evidence, a single LLM interpreting it in isolation tends to commit to a specific repair hypothesis, leading to test overfitting and producing patches that satisfy the observed traces and test suite by coincidence rather than correct logic. This observation suggests that runtime evidence should not be treated merely as additional input, but as objective constraints that candidate patches must satisfy. Building on this insight, we propose TraceRepair ,a multi-agent framework that leverages runtime facts as shared constraints for patch validation. A probe agent captures execu-tion snapshots of critical variables, forming an objective basis for repair, while a committee of specialized agents cross-verifies candi-date patches to expose inconsistencies and iteratively refine them. ## 1 Introduction 

Automated Program Repair (APR) aims to automatically transform buggy programs into correct ones without human intervention, thereby mitigating the high cost of manual debugging [ 21 , 29 ]. Traditional APR approaches mainly rely on heuristic search [ 9,14 , 21 , 26 , 32 , 36 , 40 , 45 , 53 ], constraint solving [ 1, 7, 20 , 27 , 28 ,31 , 46 , 48 ], or predefined templates [ 11 , 14 , 18 , 19 , 24 , 35 , 55 ] to generate patches by mutating code, synthesizing logic, or applying fix patterns. Although these methods have laid the foundation for the field, they are often limited by restricted search spaces and a dependence on surface-level syntactic manipulations or manually crafted patterns [29, 33]. In recent years, the adoption of Large Language Models (LLMs) has transformed the landscape of APR, shifting the dominant para-digm from heuristic search to neural generation [ 42 , 43 ]. Trained on massive-scale code corpora, LLMs demonstrate strong capabilities

in code comprehension and logical reasoning [ 4, 10 , 23 , 34 ]. State-of-the-art approaches exploit these models through diverse strate-gies: ChatRepair [ 44 ] utilizes conversational feedback for patch refinement; ThinkRepair [ 52 ] and REINFIX [ 54 ] enhance repair per-formance via chain-of-thought reasoning and the retrieval of repair ingredients, respectively; while recent agentic frameworks, such as RepairAgent [ 2 ] and AdverIntent-Agent [ 51 ], automate the repair process by actively invoking external tools or utilizing adversarial feedback. These methods have shown substantial improvements over traditional techniques on standard benchmarks in both the number of fixed bugs and patch quality. Despite recent advancements, current LLM-based approaches still face significant limitations. First, most existing methods rely primarily on static code analysis or coarse-grained dynamic feed-back. While feedback-driven approaches, such as ChatRepair [ 44 ]and Self-Debugging [ 5 ], utilize standard error streams (stderr) or test outputs to guide the repair process, such information is of-ten insufficient for complex bugs. These signals typically indicate where a failure occurred but fail to reveal why it happened, as they lack visibility into the intermediate execution values and data flow leading up to the error. Unlike human developers who inspect run-time states to trace the logic flaw, models operating without this detailed execution history often struggle to resolve silent failures. Second, relying on a single agent or simple feedback loops is often insufficient for robust repair. Research indicates that LLMs operating in isolation are prone to hallucination and struggle to self-correct erroneous reasoning [ 12 , 13 ]. One common approach is large-scale sampling: generating and ranking multiple candidates to maximize the probability of finding a correct patch [ 42 , 43 ]. How-ever, this approach prioritizes quantity over quality, incurring high computational costs while increasing the risk of test-overfitting. While multi-agent debate has shown effectiveness in improving reasoning through cross-verification in other Natural Language Pro-cessing domains [ 3 , 6], its potential as a logical filter for scrutinizing patches in APR remains largely unexplored. Motivated by these observations, we propose TraceRepair , an agentic framework that combines runtime trace collection with multi-agent debate. Unlike approaches that rely solely on static code analysis, TraceRepair grounds the repair process in run-time evidence. Specifically, the framework employs an autonomous Probe Agent to inject logging statements into the buggy program, capturing detailed variable snapshots to pinpoint the root cause. Guided by these traces, TraceRepair initializes repair agents with defensive, causal, and semantic reasoning perspectives, enabling the generation of diverse fix candidates. These agents engage in an iterative debate where the captured execution data serves as objec-tive evidence to filter out hallucinations; proposals contradicted by the runtime state are challenged by peer agents. Finally, a Judge Agent arbitrates the debate and synthesizes the validated insights into a definitive fix. In short, TraceRepair transforms execution traces from passive observations into active logical filters, grounding the entire repair process in what the program actually did rather than what static analysis assumes it did. This fundamentally shifts the reasoning basis of repair agents: instead of inferring error propagation paths from source code, agents reason directly from runtime evidence to identify the true root cause. Meanwhile, the multi-agent debate mechanism ensures that every patch candidate must remain con-sistent with the observed execution facts, filtering out patches that appear logically sound in isolation but contradict actual program behavior. We evaluated TraceRepair on the widely-used Defects4J bench-mark (v1.2 and v2.0) [ 17 ] and a newly constructed dataset Recent-Java . Experimental results demonstrate that TraceRepair estab-lishes a new state-of-the-art in repair effectiveness, while substan-tially reducing token consumption compared to existing conversa-tional and agentic baselines. In summary, this paper makes the following contributions: 

Novel Framework: We propose TraceRepair , a framework that injects diagnostic probes into buggy programs to collect runtime execution traces, then uses these traces as constraints in a structured multi-agent debate. Repair agents with distinct strategies generate and cross-validate patch candidates against the observed runtime state, preventing false consensus on patches that contradict actual program behavior. 

Extensive Evaluation: We conduct an extensive evaluation on the Defects4J benchmark, where TraceRepair correctly fixes 392 bugs, outperforming existing LLM-based APR methods. To assess robustness against data leakage, we further construct a new benchmark, Recent-Java , and show that TraceRepair maintains strong performance on unseen defects beyond model training data. 

Open Science: We release the full implementation of Trac-eRepair , evaluation scripts, and all experimental data to support reproducibility and follow on research. The replication package is anonymously available at: https://doi.org/10.5281/zenodo.19252356. 

## 2 Motivation 

A core limitation of current APR tools lies in their reliance on static analysis, which is often ineffective when bugs produce no explicit crash signals [ 29 , 33 , 39 ]. We illustrate this through a real-world defect from Defects4J, showing where static diagnosis fails and how runtime instrumentation exposes the true fault. 

## 2.1 The Challenge of Silent Failures 

A fundamental challenge in APR is handling silent failures, which are logic errors where the program executes successfully but pro-duces incorrect results [ 38 ]. In contrast to crashing bugs, where the stack trace restricts the search space to a specific execution path or line, silent failures often provide only a test assertion error (e.g., “Expected X, but got Y”). Without visibility into intermediate variable states or control flow decisions, the model must rely solely on the source code and the final failure message, often targeting code regions unrelated to the actual fault [8].

## 3 Methodology 

We propose TraceRepair , a framework that augments static code reasoning with runtime execution evidence. As shown in Figure 2, the system comprises a Trace-Guided Debate System operating over a Universal Execution Sandbox. Unlike approaches that rely solely on static code generation, 

TraceRepair actively queries the program state through dynamic instrumentation. The workflow proceeds in three phases: (1) Phase 1: Diagnostic Probing. The Probe Agent identifies the failure context and executes an instrumentation strategy. By injecting logging statements, it captures a structured Run-time Trace containing variable values and branch decisions. (2) Phase 2: Multi-Strategy Debate. We initialize agents with distinct repair strategies: Defensive, Causal, and Semantic. These agents operate within a Generate, Critique and Update loop, utilizing the captured trace as a hard constraint to validate and filter patch candidates. (3) Phase 3: Arbitration and Synthesis. Finally, a Judge Agent analyzes the debate history to resolve conflicts between strategies and synthesize a final patch, which is verified within the sandbox.

To ensure the instrumentation process itself does not introduce secondary errors (e.g., compilation failures), the agent ad-heres to a strict safety protocol: all injected probes are encapsulated within defensive try-catch blocks. If an injected probe causes a compilation error, the agent automatically reverts the code to its original state and retries with adjusted parameters, ensuring the build remains stable. 

Trace Volume Management. Runtime execution traces, par-ticularly those emerging from loops or deep recursion, can rapidly exceed the context window limits of LLMs. To mitigate this without losing diagnostic fidelity, TraceRepair implements a Tail-Biased Truncation strategy. We enforce a layered containment protocol. First, at the exe-cution level, the instrumented probe includes a guard clause that halts logging if the file size exceeds 10MB; this prevents storage ex-haustion during infinite loops. ## 3.3 Phase 2: Multi-Strategy Debate 

To avoid convergence on incorrect but plausible solutions, TraceRe-pair does not rely on a monolithic generator. Instead, it orchestrates a committee of agents that engage in a structured debate. This phase proceeds in two stages: strategy initialization and the trace-guided debate loop. 

Strategy Initialization. We initialize a diverse hypothesis space (H ) by employing agents equipped with distinct reasoning strate-gies. Conditioned on the buggy code ( 𝑃 bug ) and the runtime trace ( 𝜏 ), each agent generates candidate patches from a distinct perspective: 

• Defensive Strategy. Prioritizes system robustness. Guided by anomalies in the trace, it assumes environmental instabil-ity and injects preconditions to prevent state corruption. 

• Causal Strategy. Focuses on error propagation. It analyzes the divergence point in the trace to identify the algorithmic root cause and repairs the underlying control flow logic. 

• Semantic Strategy. Ensures specification adherence. It scru-tinizes the code against API contracts and domain-specific constraints. To seed the debate with high-quality candidates, we employ a Best of N Selection mechanism. Candidates undergo pre-evaluation in the sandbox, and the best-performing ones (Pass > Compile >

Fail) are selected as initial proposals. 

The Trace-Guided Debate Loop. In traditional multi-agent frameworks, agents often reach a false consensus on incorrect patches. TraceRepair disrupts this by enforcing that every critique must be grounded in the objective evidence provided by the trace 

𝜏 . The debate iterates between Proposers and Critics. For every candidate patch ℎdef (Defender), an opposing agent 

ℎatk (Attacker) performs a verification against the trace artifact 𝜏 .We formally define the critique generation function Fcrit as:

## 3.4 Phase 3: Arbitration and Synthesis 

This phase is invoked exclusively when the debate loop reaches the maximum iteration limit ( 𝑅 max ) without yielding a verified patch. To resolve the impasse, the Judge Agent intervenes as a global arbitrator. 

Retrospective Synthesis. Unlike the repair agents that refine patches through iterative local updates, the Judge Agent performs a one-shot analysis of the entire debate trajectory. Its input consists of three components: the complete proposal history submitted by all three strategy agents across every round, the critique records accumulated at each iteration, and the runtime trace 𝜏 .This global view allows the Judge to identify complementary signals that no individual agent could recognize in isolation. During the debate, different strategies tend to capture distinct facets of the fault: the Defensive agent may correctly identify where a boundary guard is needed, while the Causal agent accurately locates the root defect in the control flow, yet neither produces a complete fix within its own iteration. The Judge’s role is to consolidate these scattered partial corrections into a coherent patch candidate 𝑃 final .This consolidation is not a naive concatenation of proposals. The Judge uses the runtime trace 𝜏 as a consistency baseline, cross-validating all candidate logic against the observed execution state and discarding any inference that contradicts the recorded facts. The result is a repair that simultaneously satisfies all constraints surfaced during the debate.

To evaluate TraceRepair , we design our experiments around four research questions: 

• RQ1: Can the framework outperform existing base-lines in generating correct patches? 

We evaluate on Defects4J (v1.2 and v2.0) under both Per-fect Fault Localization (PFL) and Method-Level Fault Lo-calization (MFL) settings, comparing against the current best-performing baselines to comprehensively assess the framework’s repair capability. 

• RQ2: How much does each component contribute to the overall repair performance? 

We conduct an ablation study by removing runtime traces and multi-agent debate separately, measuring the individual contribution of each component to the final repair results. 

• RQ3: Can the framework generalize to defects beyond its training distribution? 

We evaluate on a newly constructed dataset comprising bugs collected from repositories active after the training cutoff of the underlying models, assessing whether the framework’s repair capability stems from genuine dynamic reasoning rather than memorization of previously seen fixes. Although modest in size, such datasets are inherently difficult to construct under strict leakage-free constraints, and are widely used to evaluate generalization beyond memorized patterns. 

## 4.3 Baselines 

To evaluate the effectiveness of TraceRepair , we compare it against ten state-of-the-art LLM-based APR approaches, covering five rep-resentative categories: (1) feedback-driven methods, including Cha-tRepair [ 44 ] and D4C [ 47 ]; (2) retrieval-augmented methods, includ-ing REINFIX [ 54 ]; (3) agentic and reasoning-based methods, includ-ing RepairAgent [ 2 ], ThinkRepair [ 52 ], and AdverIntentAgent [ 51 ]; (4) template-based and hybrid methods, including GiantRepair [ 22 ]and FitRepair [ 41 ]; and (5) direct generative methods, including GAMMA [55] and Codex [4]. 

## 4.4 Evaluation Metrics 

We evaluate performance using two standard metrics: plausible patches , which satisfy the full test suite (passing both 𝑇 fail and 𝑇 reg ), and correct patches , which are semantically equivalent to the devel-oper’s ground truth. Mere test passage does not guarantee correctness due to the over-fitting phenomenon, where patches exploit weak test specifications without fixing the underlying logic [ 37 ]. To rigorously mitigate this threat, we adopted the manual verification protocol established in prior studies [ 15 , 50 , 56 ]. Specifically, two authors independently inspected every plausible patch to determine semantic equivalence. Cases of disagreement were resolved through an arbitration session to reach consensus, ensuring that our correctness metrics reflect genuine semantic equivalence rather than artifacts of insufficient test coverage. 

## 4.5 Implementation Details 

TraceRepair is implemented in Python and interacts with LLMs via the standard OpenAI API interface. The framework is model-agnostic and can be seamlessly integrated with any LLM backend. To evaluate its generalizability across different model families, we instantiate it with two representative backbones: DeepSeek-V3.2 and GPT-3.5-Turbo, hereafter referred to as TraceRepair 𝐷𝑆𝑉 3.2 and 

TraceRepair 𝐺𝑃𝑇 3.5, respectively. To promote search space diver-sity, we configured the generation agents with a temperature of 1.0. We employed a sampling strategy with a maximum budget of 40 LLM invocations per bug, incorporating an early-stopping mecha-nism that terminates generation once a plausible patch is identified. We enforced a strict 300-second timeout for all test executions to prevent non-terminating executions caused by faulty patches. The multi-agent debate is limited to 3 rounds.

## 6 Threats to Validity 

We identify three categories of threats to the validity of our findings. 

Internal Validity. A key concern in LLM-based APR is data leakage, where the model recalls training data rather than rea-soning about the bug. We address this through the Recent-Java 

benchmark, which contains defects that post-date the training cut-off of our backbone models. The framework’s ability to resolve 10 of these unseen bugs (Section 5.3) suggests that performance reflects dynamic reasoning rather than memorization. To guard against overfitted patches, two authors independently verified ev-ery generated patch for semantic equivalence to the developer’s fix. 

External Validity. The diagnostic instrumentation strategy of 

TraceRepair is built around runtime observation rather than any particular language feature, which positions it well for adaptation beyond Java. We selected Java as the primary evaluation language because Defects4J is the most mature and widely adopted bench-mark in the APR community, enabling direct comparison with prior work. Extending the Probe Agent to other languages remains a natural direction for future work. 

Construct Validity. TraceRepair depends on test suites that exercise the buggy logic sufficiently to produce a meaningful trace. Under weak test coverage or flaky test conditions, the Probe Agent may fail to capture relevant runtime state, leaving the debate agents without grounded evidence. Our experiments with coarse-grained fault localization suggest that the multi-agent debate mechanism retains reasonable resilience under noisy trace data, though im-proving robustness under low coverage conditions remains future work. 

## 7 Related Work 

APR has evolved through three broad phases: heuristic search, neural generation, and agentic reasoning. We position TraceRepair 

within this trajectory. 

From Heuristics to Neural Translation. Early APR systems framed patch generation as a search problem guided by test suites. GenProg [ 21 ] and ARJA [ 53 ] applied genetic algorithms to mu-tate code, but frequently produced overfitting patches that pass tests without correcting the underlying logic [ 33 ]. Constraint-based methods such as Angelix [ 28 ] and S3 [ 20 ] offered greater precision through symbolic execution, while template-based tools such as TBar [ 24 ] and FixMiner [ 19 ] encoded expert fix patterns. The rise of deep learning then shifted the field towards Neural Machine Translation, with models such as CoCoNut [ 25 ], CURE [ 15 ], and Recoder [ 56 ] learning to map buggy code to correct code directly. However, these approaches remain bound by fixed vocabularies and historical training distributions, limiting their ability to handle unseen logic or project-specific identifiers. 

Generative Repair and Retrieval Augmentation. Large Lan-guage Models broadened the scope of APR by enabling repair with-out prior exposure to similar bugs. Models such as Codex [ 4], StarCoder [ 23 ], and DeepSeek-Coder [ 10 ] brought strong code reasoning to the task, and approaches like AlphaRepair [ 42 ] and GAMMA [ 55 ] exploited this capacity through cloze style gener-ation. To reduce hallucination in pure generation [ 13 ], retrieval-augmented frameworks such as REINFIX [ 54 ] and RetriGen [ 30 ]supply the model with relevant fix patterns at inference time. This strategy improves plausibility on common defect types, but de-grades on novel, state dependent faults where no similar historical fix exists. 

Agentic Debugging and Reasoning. Recent work has moved toward autonomous agents that interact with the codebase rather than passively generating patches. ChatRepair [ 44 ] and Self-De-bugging [ 5 ] feed test failure messages back to the model in a con-versational loop, while RepairAgent [ 2] and SWE-agent [ 49 ] extend this with tool use, allowing agents to navigate repositories and invoke static analyzers. Benchmarks such as SWE-Bench [ 16 ] have been introduced to evaluate these agents on real GitHub issues, targeting repository-level repair where the primary challenge lies in localizing faults across large codebases. Unlike SWE-Bench, De-fects4J operates at the function level, a granularity at which runtime traces provide the most direct and actionable diagnostic evidence. Extending the trace-guided mechanism to repository-level fault localization and repair remains an important direction for future work. 

## 8 Conclusion 

This paper presents TraceRepair , a multi-agent APR framework that incorporates runtime trace evidence into the patch generation process. The Probe Agent collects execution snapshots to guide subsequent debate among repair agents, producing more grounded and accurate fixes. Extensive experiments show that runtime traces consistently im-prove fault localization and patch correctness, while the multi-agent debate mechanism reduces hallucination by anchoring candidates to observed runtime facts. Evaluation on unseen defects confirms that the framework’s effectiveness stems from dynamic reasoning rather than memorization. We hope this work offers a useful per-spective on combining dynamic analysis with LLM-based repair, and encourages further exploration of execution-driven approaches.
