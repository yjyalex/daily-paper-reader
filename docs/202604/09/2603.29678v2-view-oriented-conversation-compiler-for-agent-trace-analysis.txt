# View-oriented Conversation Compiler for Agent Trace Analysis 

Lvmin Zhang and Maneesh Agrawala Stanford University 

Agent traces carry increasing analytical value in agentic systems and context engineering, yet most prior work treats conversation format as a trivial imple-mentation detail. Modern agent conversations, however, contain deeply structured content—nested tool calls and results, chain-of-thought reasoning blocks, sub-agent invocations, context-window compaction boundaries, and harness-injected system directives—whose complexity far exceeds that of simple user–assistant exchanges. Feeding such traces to a reflector or other analytical mechanism in plain text, JSON, YAML, or via grep can materially degrade analysis quality. This paper presents VCC (View-oriented Conversation Compiler), a compiler (lex → parse → IR →

lower → emit) that transforms raw agent JSONL logs into a family of structured views: a full view (lossless transcript serving as the canonical line-number coordi-nate system), a user-interface (UI) view (reconstructing the interaction as the user actually perceived it), and an adaptive view (a structure-preserving projection gov-erned by a relevance predicate). In a context-engineering experiment on AppWorld, replacing only the reflector’s input format—from raw JSONL to VCC-compiled views—leads to higher pass rates across all three model configurations tested, while cutting reflector token consumption by half to two-thirds and producing more concise learned memory. These results suggest that message format functions as infrastructure for context engineering, not as an incidental implementation choice. 

## 1 Introduction 

Context learning research has devoted considerable attention to upper-level architectural choices: role decomposition among generators, reflectors, and curators [ 60 ]; hierarchical memory organization and retrieval [ 32 , 34 ]; tool use [ 35 ]; and experience extraction and generalization from agent trajectories [ 9,42 , 38 ]. The format of the messages themselves—how an agent trace is represented and presented to an LLM for analysis—has largely been treated as a trivial implementation detail, with plain text, JSON, YAML, and grep serving as default conventions. For simple user–assistant exchanges, these conventions may suffice. Contemporary traces, however, carry a different kind of complexity. A single session from a coding agent such as Claude Code [ 2] or Codex [ 31 ] can produce a JSONL log exceeding ten thousand lines, in which tool-call parameters, tool-result bodies, chain-of-thought reasoning blocks [ 48 ], sub-agent invocations, context-compaction boundaries, base64-encoded media, and harness-injected system directives are interleaved. Tool-result bodies alone can account for the majority of the token budget, burying the actual decision points beneath volumes of file contents and terminal output. Prior empirical work supports the view that format is not neutral: identical content presented in different formats (plain text, Markdown, JSON, YAML) can cause LLM performance to vary by up to 40 percentage points on certain tasks [ 13 ], and 40–60% of tokens in agent trajectories have been identified as useless, redundant, or expired information whose removal does not degrade task performance [ 50 ]. The bottleneck, in other words, is not a lack of capable learning methods but the quality of their input. A generator agent executes tasks; a reflector agent analyzes the resulting trace and edits a shared memory file to accumulate rules. The sole experimen-tal variable is the reflector’s input format: VCC-compiled views versus raw JSONL. Across three model configurations (Opus, Sonnet, Haiku) and two test splits (168 and 416 tasks), VCC-trained memory leads to higher pass rates, while cutting reflector token consumption by half to two-thirds and producing more concise memory files. The details are presented in Section 3.2. 

## 2 VCC 

2.1 Three Views 

An agentic conversation trace is a temporally ordered sequence of heterogeneous records: user utterances, assistant responses (interleaving natural language, chain-of-thought reasoning, and tool invocations), tool results, and system directives. VCC formalizes this trace as a structured document and defines three views —projections of a shared intermediate representation that vary in resolution and coverage while maintaining referential consistency (Figure 1).

The full view serves as the canonical reference surface—all line numbers in other views resolve against it. 

User-Interface (UI) view. The UI view reconstructs the perceptual experience of the human user during the original interactive session. In a terminal-based coding agent, the user seldom sees raw tool inputs, tool outputs, or chain-of-thought reasoning; the user observes assistant text, one-line tool-call summaries, and the conversational turn structure. The UI view applies four transformations to the IR. First, internal content is elided : tool-call bodies, tool-result bodies, thinking blocks, and system messages are suppressed. Second, summary substitution collapses each tool call to a single-line summary ( e.g . * Read "src/config.py" (file.txt:19-21,24-34) ) annotated with line-range pointers into Vfull . Third, consecutive-turn merging combines adjacent assistant sections with no intervening user turn, reflecting the user’s perception of a single continuous response. Fourth, noise filtering strips markup injected by the harness (IDE selection contexts, system reminders, hook outputs), as these are not part of the user’s intentional input. 

Adaptive view. The adaptive view generalizes content-based retrieval by combining a relevance predicate with structure-preserving projection. Given a predicate ρ : line → { 0, 1}, the adaptive view retains the structural skeleton of the conversation—section headers, turn boundaries, block delimiters—while projecting each content block through ρ:

The predicate ρ is a parameter: regex matching is the simplest instantiation, but BM25 scoring, embedding-based similarity, or LLM-based relevance judgments slot into the same framework without altering the structural projection logic. The adaptive view admits two output modalities that stand in a transpose relationship. The document-oriented modality preserves the temporal structure of the conversation, organized section by section with non-matching blocks elided. The index-oriented modality discards structural scaffolding and emits a flat sequence of matching blocks, each tagged with its semantic role ( e.g . [tool_call] ,

[assistant] , [thinking] ). The underlying data is identical; only the iteration order differs—row-major by section versus column-major by match. 

Distinction from multi-level memory. Multi-level memory architectures (MemGPT [ 32 ], A-MEM [ 51 ], Mem0 [ 5 ], RAPTOR [ 34 ]) maintain static, precomputed artifacts—summaries, embed-dings, knowledge-graph nodes—connected by persistent edges. Querying such a structure means traversing a pre-existing topology; the levels exist independently of any particular query. VCC’s views, by contrast, are dynamic, query-time projections of a single flat IR. No precomputed summary exists; no persistent graph is maintained; no embedding is stored. Different predicates produce different views by recomputing the projection from scratch. More fundamentally, multi-level memory performs abstraction : generating new content (summaries, embeddings) not present in the source, introducing lossy compression whose faithfulness must be evaluated empirically. VCC performs 

projection : selecting and arranging existing content from the source, annotated with exact coordinates for recovering the unselected remainder. The system is lossless by construction—not in the sense that every view contains all information, but in the sense that every view contains a verifiable path back to the full information. 

Distinction from grep. A flat grep over a conversation trace returns a set of isolated matching lines. Each line is stripped of its conversational identity: the reader cannot determine whether a match occurred in a user utterance, an assistant’s chain-of-thought, a tool-call input, or a tool result. A match tagged [tool_call] Edit (an intention) carries different implications than a match tagged 

[tool_result] Read (an observation)—grep collapses intention, observation, and reasoning into a uniform bag of lines. The adaptive view preserves block-level role annotations and range pointers, enabling semantic distinctions that flat search cannot express. 2.2 Compiler Pipeline and Line-Number Consistency 

VCC follows a full compiler pipeline with four stages: (i) lexing tokenizes the JSONL source into typed records, one per source record; records carrying no conversational content ( queue-operation ,

file-history-snapshot , progress , api_error , among others) are filtered at this stage; (ii) 

parsing lowers records to a typed intermediate representation where each node carries a semantic type (user , assistant , tool_call , tool_result , thinking , etc .), its content lines, and structural metadata (section index, block index); (iii) line assignment performs a single sequential pass that assigns monotonically increasing line numbers to all IR nodes—this assignment is deterministic and occurs exactly once, before any view-specific lowering; (iv) view lowering projects the shared IR into each view: the full view emits all nodes; the UI view and adaptive view apply their respective transformations. No lowering pass modifies the line-number assignment—lowering may only select, truncate, or annotate content, never reorder or renumber it. Because all three views are projections of a single IR with a single line-number assignment, pointer consistency across views is a structural invariant of the pipeline rather than a property that must be validated post hoc. This is analogous to how a compiler’s static single-assignment form guarantees use-def consistency by construction rather than by testing. 

2.3 Compiler Transformations 

As with any compiler, the lexer and parser are specific to the source language—in this case, the Claude Code JSONL “language”. The IR, line-assignment, and view-lowering stages are language-independent; supporting a different agent runtime ( e.g . Codex, OpenClaw) requires writing a new lexer and parser while reusing the rest of the pipeline. We briefly describe the Claude Code lexer and parser to illustrate the scope. Tool-call parameters are escaped JSON strings ( e.g . {"file_path":"src/pets.py","content":"class Pet:\n def __init__..."} ); the lexer compiles them to YAML with block scalars ( |) for multi-line values. Read-tool results carry a digits →content prefix that the parser strips to recover the original source. Harness-injected XML markup ( <system-reminder> , <ide_opened_file> ,

<command-message> , among others) is removed, and user turns consisting entirely of such markup are elided. Internal tool calls ( TodoWrite , ToolSearch ), ANSI escape codes, and control characters are stripped. When context compaction splits an assistant message across two JSONL records with the same message.id , the parser reassembles them into one section. Record types with no conver-sational content ( queue-operation , file-history-snapshot , progress , api_error , among others) are discarded during lexing. Base64-encoded inline images are decoded to separate files and replaced by placeholders. 

2.4 Agent Workflow 

The three views enable a progressive-disclosure workflow. An agent (or human analyst) begins with the UI view to scan the overall session trajectory—the sequence of user requests, assistant actions, and outcomes—without wading through tool I/O or chain-of-thought internals. Targeted searches via the adaptive view then locate specific content blocks of interest, each annotated with its semantic role and a line-range pointer. Following a pointer into the full view recovers the complete, verbatim content of that block and its immediate surroundings:

3.1 Setup Benchmark. We use AppWorld [ 44 ], which provides a realistic API-interaction environment for evaluating autonomous agents. The benchmark offers three splits: train (90 tasks) for learning, 

test_normal (168 tasks) and test_challenge (416 tasks) for evaluation. Tasks span three difficulty levels (d1, d2, d3). 

Learning procedure. We adopt an epoch-based context-engineering protocol. In each epoch, every task in the train split is executed once by a generator agent, then evaluated by the AppWorld harness. A reflector agent reads the evaluation result to determine success or failure, analyzes the generator’s execution trace, and edits a shared memory file ( MEMORY.md ) to add rules. For failed tasks, the reflector writes [CRITICAL] rules identifying root causes and preventive measures; for successful tasks, the reflector writes rules flagging risky or suboptimal patterns. The process runs for two epochs (90 tasks × 2 = 180 analyses per condition), iteratively refining the rule set. Training runs 32 tasks in parallel; after each batch, workers’ memory edits are merged via an LLM-based diff-and-merge step. The learned MEMORY.md is then evaluated on the test splits in a single pass with no further memory updates. 

Experimental variable. The sole variable is the format of the trace presented to the reflector during analysis: In the VCC condition, the raw JSONL is pre-compiled into structured views; the reflector receives the UI view (for session-level orientation) and the full view (for line-level detail), navigating between them via line-number pointers. In the JSON condition, the reflector reads the raw JSONL directly, which is the default approach in existing trajectory-learning methods. 

Model configurations. Three generator–reflector pairings are tested: Opus+Opus, Sonnet+Sonnet, and Haiku+Sonnet. The original design called for same-model pairing throughout, but Haiku does not have reflector capability (unable to improve pass rates over any candidates in our experiments), and the Haiku reflector was replaced with Sonnet. 

Baselines. For each generator model, a baseline evaluation uses only the initial seed memory (no trained rules) on both test splits, establishing the lower bound for context-engineering gains. 

Metrics. We report AppWorld’s official metrics: task_goal_completion (whether the task’s primary objective is met) and scenario_goal_completion (a stricter measure requiring all scenario-level constraints—side effects, state consistency, edge cases—to be satisfied in addition to the primary goal), broken down by difficulty level (d1 = easy, d2 = medium, d3 = hard). We also report total reflector token consumption during training ( T tok ), per-analysis token cost ( T/a ), and the size of the learned memory file in characters ( Mem ).

Memory size. VCC-trained memory files are consistently smaller: Opus 12.0K vs. 12.6K characters, Sonnet 15.5K vs. 23.4K, Haiku 37.9K vs. 43.3K. Smaller memory correlates with higher pass rates in this experiment, suggesting that VCC enables the reflector to produce more focused rules rather than accumulating verbose or redundant entries. 

Weak-model benefit. The Haiku configuration shows the largest absolute gain from context engi-neering (baseline 39.9 → VCC 75.6 on test_normal task_goal, an increase of 35.7 percentage points) and the largest VCC-over-JSON advantage on test_normal task_goal (75.6 vs. 71.4, a difference of 4.2 percentage points). This is likely because a weaker generator produces more error-prone traces that are harder to analyze in raw form, making the structured views more beneficial for the reflector. 

JSON behaviors. JSON reports higher scores than VCC in certain per-difficulty cells—for instance, Haiku d2 on test_normal task_goal (79.2 vs. 75.0) and several scenario_goal cells in the Haiku configuration. These cases tend to occur in specific difficulty–split combinations rather than at the aggregate level. One possible explanation is that the JSON reflector, consuming more tokens per analysis, occasionally captures details that the VCC reflector’s pointer-based navigation misses, at the cost of a much larger overall token budget. 

Discussion. Higher pass rates, lower token consumption, and smaller memory files are not in-dependent outcomes. A plausible shared cause is that structured views let the reflector focus on decision-relevant content rather than navigating raw data. This produces rules that are both more precise (higher pass rates) and more concise (smaller memory), while the navigation structure itself reduces the volume of text the reflector must process (fewer tokens). The three metrics may be measuring the same underlying variable—the degree to which the reflector’s attention is directed at the right information. 

## 4 Related Work 

Format affects LLM performance. He et al . [ 13 ] show that formatting identical content as plain text, Markdown, JSON, or YAML causes performance to vary by up to 40 percentage points; Tam et al . [ 43 ] find that structured-output constraints degrade reasoning ability; Sui et al . [ 39 ] observe that table serialization format changes accuracy by approximately 7 points. On the length axis, Liu et al . [ 26 ] document a U-shaped attention pattern where mid-context information is systematically underused, and Du et al . [ 8] show that input length alone degrades performance by 13.9–85% 7even with perfect retrieval. Within agent trajectories, Xiao et al . [ 50 ] find that 40–60% of tokens are redundant, and Lindenbauer et al . [ 25 ] show that simple observation masking matches LLM-based summarization at half the cost. Performance degradation with increasing input length is also documented in [ 14 ]. Liu et al . [ 28 ] show that jointly optimizing prompt content and format outperforms content-only optimization. Yang et al . [ 52 ] demonstrate that agent-computer interface design affects performance as much as the underlying model; OpenHands [ 46 ] provides an open platform for evaluating such interfaces. Mei et al . [ 30 ] survey context engineering as an emerging discipline. 

Trajectory learning methods. A growing body of work extracts reusable experience from agent trajectories. ReAct [ 54 ] interleaves reasoning traces with actions; Reflexion [ 37 ] stores verbal self-reflections in episodic memory; ExpeL [ 61 ] contrasts successful and failed trajectories to extract rules. TIM [ 9] derives strategy and recovery tips from trajectories (+14.3 percentage points on App-World); ACE [ 60 ] evolves a playbook via generator/reflector/curator roles; Dynamic Cheatsheet [ 42 ]maintains a reusable strategy table at test time. ETO [ 38 ] learns from contrastive trajectory pairs via DPO; ECHO [ 15 ] generates counterfactual success trajectories via hindsight rewriting; RPMS [ 57 ]introduces rule–memory conflict arbitration. AutoGuide [ 12 ] generates conditional guidelines from offline experience; AutoManual [ 4] constructs instruction manuals through multi-agent interaction; Agent Workflow Memory [ 47 ] induces reusable workflows (+51.1% on WebArena); MACLA [ 11 ]distills trajectories into procedural memory with Bayesian reliability tracking. All of these methods consume traces in their raw format. VCC is a preprocessing layer that structures their input. 

Structured trace analysis. AgentRx [ 3] normalizes heterogeneous agent logs to a common IR and synthesizes executable constraints for failure diagnosis (+23.6 percentage points in failure localization). TraceSIR [ 53 ] uses a multi-agent framework with an LLM-based importance scorer for trace reduction. The Cognition Agent Trace specification [ 6] defines a structured JSON format for agent provenance. AgentStepper [ 17 ] represents trajectories as structured conversations and provides an interactive debugger with breakpoints and stepwise execution. TRAIL [ 7 ] benchmarks LLM trace-debugging ability on 148 human-annotated traces and reports that the best model scores 11%; AgenTracer [ 59 ] finds that current reasoning LLMs achieve below 10% accuracy on failure attribution in multi-agent traces. DoVer [ 29 ] augments hypothesis-based debugging with active intervention and replay, recovering 18–28% of failed trials. LOCA-bench [ 58 ] benchmarks agent performance under controllable context growth. The compiler metaphor has been applied elsewhere in LLM systems: DSPy [ 21 ] compiles prompt-pipeline parameters and SGLang [ 62 ] compiles LLM programs for efficient execution. 

Agent memory and context management. Multi-level memory systems—MemGPT [ 32 ], A-MEM [ 51 ], Mem0 [ 5], RAPTOR [ 34 ]—organize stored experience through hierarchical or graph-based structures, building on the memory-stream architecture of Park et al . [ 33 ]; Hu et al . [ 16 ] survey the broader landscape. Sumers et al . [ 40 ] propose CoALA, a cognitive architecture organizing agents around modular memory and structured action spaces. Context-compression methods include LLM-Lingua [ 18 ] (up to 20x prompt compression), AgentFold [ 56 ] and ACON [ 20 ] (runtime observation compression), and AgentRM [ 36 ] and AgentOCR [ 10 ] (learned compression). More recent work gives agents control over their own context: Focus [ 45 ] lets the agent decide when to consolidate and prune history, and Context-Folding [ 41 ] folds completed subtasks to reduce active context by 10x. Multi-agent frameworks such as AutoGen [ 49 ] produce particularly complex traces. Observability platforms (LangSmith [ 23 ], Langfuse [ 24 ], AgentTrace [ 1]) instrument running agents for structured telemetry. Memory-pointer approaches [ 22 ] for context-window overflow are conceptually related to VCC’s cross-view line-number pointers. 

## 5 Conclusion 

VCC compiles raw agent JSONL logs into semantically annotated, line-number-consistent, pointer-navigable structured views. Replacing only the reflector’s input format—from raw JSONL to VCC-compiled views—yields higher pass rates, lower token consumption, and more concise learned memory across three model configurations on AppWorld, with the three metrics improving simulta-neously rather than trading off. The experiment changes nothing about the learning algorithm, the model, or the task set; the only difference is how the trace is presented. That this single substitution 8moves all three metrics in the same direction suggests that message format is not an incidental implementation choice but a load-bearing component of context engineering infrastructure. The current evaluation is limited to AppWorld; extending to other agent benchmarks—SWE-bench [ 19 ], TAU-bench [ 55 ], WebArena [ 63 ], AgentBench [ 27 ]—and to additional trajectory-learning methods remains an open direction.
