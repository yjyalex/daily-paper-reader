## 摘要
训练有效的软件工程智能体需要大量的特定任务轨迹，这会产生高昂的数据构建成本。受数学推理中“少即是多”（Less-Is-More）假设的启发，我们研究了其在智能体场景中的扩展，并提出了一种端到端训练框架，利用更少但更高质量的训练轨迹实现了更优的智能体能力。这一目标通过 STITCH（滑动记忆轨迹推理与任务分块启发式）实现，这是一种由粗到精的机制，通过过滤低价值噪声并保留决策关键的 token，从而最大化训练信号的质量。我们在多个智能体框架（如 mini-SWE-agent、MSWE-agent）、模型规模（30B 到 355B）以及多语言设置（Python、Java 和 ArkTS）上进行了实验。在 SWE-bench Verified 上，使用 STITCH 训练的模型相比基础模型实现了高达 63.16% 的相对提升。在 Multi-SWE-bench (Java) 上，MiniMax-M2.5-STITCH 配合我们的 CodeArts Agent 脚手架达到了 43.75%（+16.67%）。在鸿蒙系统（ArkTS）上，GLM-4.7-STITCH 在不到 1000 条训练轨迹的情况下，将编译通过率提高到 61.31%（+43.34%）。我们的结果证实，“少即是多”范式可以有效地推广到跨不同语言和模型规模的复杂智能体任务中。

## Abstract
Training effective software engineering agents requires large volumes of task-specific trajectories, incurring substantial data construction costs. Inspired by the "Less-Is-More" hypothesis in mathematical reasoning, we investigate its extension to agentic scenarios and propose an end-to-end training framework that achieves superior agentic capabilities with fewer but higher-quality training trajectories. This is achieved via STITCH (Sliding-memory Trajectory Inference and Task Chunking Heuristic), a coarse-to-fine mechanism that filters low-value noise and retains decision-critical tokens to maximize training signal quality. We conduct experiments across multiple agent frameworks (e.g., mini-SWE-agent, MSWE-agent), model scales (30B to 355B), and multilingual settings (Python, Java, and ArkTS). On SWE-bench Verified, models trained with STITCH achieve up to 63.16% relative improvement over base models. On Multi-SWE-bench (Java), MiniMax-M2.5-STITCH achieves 43.75% with our CodeArts Agent scaffold (+16.67%). On HarmonyOS (ArkTS), GLM-4.7-STITCH improves the compilation pass rate to 61.31% (+43.34%) with less than 1K training trajectories. Our results confirm that the "Less-Is-More" paradigm generalizes effectively to complex agentic tasks across diverse languages and model scales.

*   **整体含义**：论文证明了“少即是多”范式在复杂的代码智能体场景中同样适用，并提供了一套从数据构建到精炼的端到端框架。

### 2. 方法论：核心思想与关键技术
论文提出了两个核心组件：**SandForge**（数据生产）和 **STITCH**（数据精炼）。

*   **SandForge (数据构建与评估)**：
    *   从 GitHub 真实 Issue 中提取任务，通过自动化流水线将其转化为可执行的沙箱任务。
    *   支持多语言（Python, Java, ArkTS）和多种智能体脚手架（Scaffold）。
    *   将执行过程中的所有信号（轨迹、工具调用、观察结果、奖励、补丁）结构化存储，作为训练素材。
*   **STITCH (由粗到精的轨迹精炼)**：
    *   **宏观预筛选 (Macro-Level)**：利用轻量级的逻辑回归（LR）模型，基于统计特征（如代码产量、工具使用频率、成功率、效率等）快速过滤掉低质量轨迹。
    *   **微观语义提取 (Micro-Level)**：针对长轨迹，采用 **Map-Reduce 范式结合滑动记忆机制（Sliding-memory）**。
        *   **Map 阶段**：将长轨迹切分为语义完整的块（利用启发式安全切分点），由 LLM 担任裁判进行评分，并携带“压缩记忆”以保持跨块的语义连贯。
        *   **Reduce 阶段**：汇总局部评分，提取高价值的子任务片段。即使整条轨迹最终失败，也能从中提取出局部正确的决策步骤。

### 3. 实验设计
*   **数据集/场景**：
    *   **Python**：使用 **SWE-bench Verified**（500 个案例）。
    *   **Java**：使用 **Multi-SWE-bench (Java)**（128 个真实 GitHub Issue）。
    *   **ArkTS (鸿蒙)**：自建基准测试（168 个需求），评估编译通过率和视觉预览通过率。
*   **对比方法**：
    *   **Base 模型**：未经针对性微调的基础模型。
    *   **RFT (Rejection Sampling Fine-Tuning)**：传统的拒绝采样微调（全量使用成功轨迹）。
    *   **STITCH (本文方法)**：经过精炼后的少量高质量数据微调。
*   **模型规模**：涵盖 30B 到 355B 的多种模型（如 Qwen3-Coder, GLM-4.5/4.7, MiniMax-M2.5）。

### 6. 主要结论与发现
*   **“少即是多”有效性**：在所有规模的模型上，STITCH 均优于 RFT。在 30B 模型上提升最显著（相对提升 63.16%）。
*   **低资源语言突破**：在 ArkTS（鸿蒙）场景下，仅用不到 1k 条数据，将编译通过率从 42.77% 提升至 61.31%。
*   **模型饱和度**：对于超大规模模型（>200B），虽然 STITCH 仍有提升，但由于模型本身能力已接近基准测试天花板，提升幅度相对较小。
*   **智能体范式偏好**：发现 Function-calling 格式的智能体在配合大规模模型时，通常优于 ReAct 风格的智能体。

### 7. 优点
*   **极高的数据效率**：通过精炼技术，用极小的数据量达到了甚至超过了海量数据训练的效果。
*   **创新的长轨迹处理**：STITCH 的滑动记忆机制解决了长轨迹在 LLM 评估时容易丢失上下文或被截断的问题。
*   **多语言普适性**：不仅在 Python 等高资源语言有效，在 ArkTS 等新兴低资源语言上也表现强劲。

### 8. 不足与局限
*   **对裁判模型的依赖**：微调过程中的微观筛选高度依赖于一个强大的“裁判模型”（如 Qwen3.5-122B），这在一定程度上增加了数据准备阶段的推理成本。
*   **大模型边际效应**：在 SOTA 级别的大模型上，该方法带来的绝对性能增益有所收窄。
*   **ArkTS 基准局限**：ArkTS 的评估部分依赖于多模态模型对 UI 截图的判断，可能存在一定的评估偏差。
