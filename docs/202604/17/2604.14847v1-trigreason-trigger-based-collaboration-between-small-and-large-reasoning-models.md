## 摘要
大推理模型 (LRM) 通过扩展的思维链在复杂任务上表现出强大的性能，但由于自回归推理，其推理延迟较高。近期研究探索了利用小推理模型 (SRM) 来加速 LRM 的推理。在本文中，我们系统地刻画了 SRM 的能力边界，并识别出三种常见的推理风险：(1) 路径偏离 (path divergence)，即 SRM 缺乏构建初始计划的策略能力，导致推理偏离最可能的路径；(2) 认知过载 (cognitive overload)，即 SRM 无法解决特别困难的步骤；(3) 恢复无能 (recovery inability)，即 SRM 缺乏鲁棒的自我反思和纠错机制。为了应对这些挑战，我们提出了 TrigReason，这是一个基于触发的协作推理框架，它用选择性干预取代了持续轮询。TrigReason 将大部分推理任务委托给 SRM，仅在必要时激活 LRM 干预——包括初始策略规划期间（策略引导触发器）、检测到异常过度自信时（认知卸载触发器），或当推理陷入无效循环时（干预请求触发器）。在 AIME24、AIME25 和 GPQA-D 上的评估结果表明，TrigReason 的准确率与全量 LRM 和 SpecReason 相当，同时向 SRM 卸载了多出 1.70 倍至 4.79 倍的推理步骤。在边缘-云环境下，TrigReason 将延迟降低了 43.9%，API 成本降低了 73.3%。我们的代码可在 https://github.com/QQQ-yi/TrigReason 获取。

## Abstract
Large Reasoning Models (LRMs) achieve strong performance on complex tasks through extended chains of thought but suffer from high inference latency due to autoregressive reasoning. Recent work explores using Small Reasoning Models (SRMs) to accelerate LRM inference. In this paper, we systematically characterize the capability boundaries of SRMs and identify three common types of reasoning risks: (1) path divergence, where SRMs lack the strategic ability to construct an initial plan, causing reasoning to deviate from the most probable path; (2) cognitive overload, where SRMs fail to solve particularly difficult steps; and (3) recovery inability, where SRMs lack robust self-reflection and error correction mechanisms. To address these challenges, we propose TrigReason, a trigger-based collaborative reasoning framework that replaces continuous polling with selective intervention. TrigReason delegates most reasoning to the SRM and activates LRM intervention only when necessary-during initial strategic planning (strategic priming trigger), upon detecting extraordinary overconfidence (cognitive offload trigger), or when reasoning falls into unproductive loops (intervention request trigger). The evaluation results on AIME24, AIME25, and GPQA-D indicate that TrigReason matches the accuracy of full LRMs and SpecReason, while offloading 1.70x - 4.79x more reasoning steps to SRMs. Under edge-cloud conditions, TrigReason reduces latency by 43.9\% and API cost by 73.3\%. Our code is available at \href{https://github.com/QQQ-yi/TrigReason}{https://github.com/QQQ-yi/TrigReason}

### TrigReason：小推理模型与大推理模型之间基于触发的协作

#### 1. 核心问题与整体含义（研究动机和背景）
大推理模型（LRM，如 OpenAI o1、DeepSeek-R1）通过长思维链（CoT）显著提升了复杂任务的解决能力，但其自回归生成数千个“思考 Token”的过程导致了极高的推理延迟和计算成本。
现有的加速方案（如 SpecReason）尝试让小推理模型（SRM）生成草稿并由 LRM 进行验证。然而，作者发现这种“持续轮询（Polling）”机制存在两大缺陷：
*   **LRM 判定不可靠**：由于模型偏见，LRM 往往无法客观评价 SRM 的中间步骤，甚至会否定自己生成的正确步骤。
*   **执行效率低下**：每一步都调用 LRM 会产生巨大的通信开销和计算冗余，尤其在“边缘-云”协作场景下，API 成本和网络延迟抵消了加速效果。
**研究动机**：通过深入分析 SRM 何时会失败，将“持续验证”转变为“精准触发干预”，以实现更高效的大小模型协作。

#### 2. 方法论：核心思想与关键技术
TrigReason 的核心思想是**基于事件触发的选择性干预**。它识别了 SRM 的三种核心风险并设计了相应的触发器：

*   **路径偏离风险（Path Divergence Risk）**：SRM 往往在初始阶段缺乏全局规划，容易走入死胡同。
    *   **策略引导触发器（Strategic Priming Trigger）**：在推理开始时，强制由 LRM 生成前 $n$ 个步骤（如前 20 步），为 SRM 奠定正确的战略方向。
*   **认知过载风险（Cognitive Overload Risk）**：SRM 在处理复杂算术或多跳逻辑时会失效。研究发现，SRM 失败前常伴随“异常过度自信”（Token 困惑度极低）。
    *   **认知卸载触发器（Cognitive Offload Trigger）**：实时监控 SRM 生成 Token 的困惑度（PPL）。当低困惑度 Token 的比例超过阈值 $\rho$ 时，认为 SRM 陷入机械模式，触发 LRM 重新生成该步骤。
*   **恢复无能风险（Recovery Inability Risk）**：SRM 发现错误后难以自我修正，常陷入循环。
    *   **干预请求触发器（Intervention Request Trigger）**：监控特定的“犹豫词”（如 "wait", "hmm", "alternatively"）。若 SRM 连续 $k$ 步表现出犹豫或停滞，则调用 LRM 接管 $m$ 步以重回正轨。

#### 6. 主要结论与发现
*   **精度持平**：TrigReason 在 AIME 和 GPQA 上的准确率达到了全量 LRM 的 99.6%~105.8%，显著优于 SRM-only。
*   **效率大幅提升**：相比 SpecReason，TrigReason 将 SRM 承担的推理步骤比例提升了 **1.70x 至 4.79x**。
*   **成本与延迟双降**：在边缘-云场景下，相比 SpecReason，TrigReason 减少了 **43.9% 的端到端延迟**和 **73.3% 的 API 成本**。
*   **发现**：SRM 的“过度自信”（低困惑度）是识别其认知失效的有效信号；LRM 的自我判定往往不如基于统计信号（PPL）的触发机制可靠。

#### 7. 优点（亮点）
*   **深刻的风险刻画**：不仅提出了框架，还系统总结了 SRM 的三种失败模式（路径偏离、过载、恢复无能），具有理论指导意义。
*   **触发机制巧妙**：利用 PPL 检测“过度自信”来识别认知过载，这是一个非常新颖且有效的启发式信号。
*   **实用性强**：弃用了高频通信的轮询机制，使其在网络受限的边缘计算场景中具有极高的落地价值。

#### 8. 不足与局限
*   **启发式依赖**：触发器的阈值（如 $\rho$）在不同模型对之间需要微调，缺乏完全自动化的自适应机制。
*   **内存开销**：虽然降低了延迟和 API 成本，但本地部署仍需同时加载 SRM 和 LRM（或至少维持 LRM 的连接），对本地显存有一定要求。
*   **犹豫词列表**：干预请求触发器依赖于预定义的犹豫词列表，虽然在数学领域有效，但在其他语言或文化语境下可能需要重新构建。
