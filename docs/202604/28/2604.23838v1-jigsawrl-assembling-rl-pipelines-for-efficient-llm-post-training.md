## Abstract
We present JigsawRL, a cost-efficient framework that explores Pipeline Multiplexing as a new dimension of RL parallelism. JigsawRL decomposes each pipeline into a Sub-Stage Graph that exposes the intra-stage and inter-worker imbalance hidden by stage-level systems. On this abstraction, JigsawRL resolves multiplexing interference through dynamic resource allocation, eliminates fragmented utilization by migrating long-tail rollouts across workers, and formulates their coordination as a graph scheduling problem solved with a look-ahead heuristic. On 4-64 H100/A100 GPUs across different agentic RL pipelines and models, JigsawRL achieves up to 1.85x throughput over Verl on synchronous RL, 1.54x over StreamRL and AReaL on asynchronous RL, and supports heterogeneous pipelines with moderate latency trade-off.

### 1. 论文的核心问题与整体含义
*   **研究动机**：强化学习（RL）已成为大语言模型（LLM）后训练的标准技术，但其**成本效率（Cost Efficiency）**极低。作者调研发现，现有框架（如 Verl）在增加 GPU 资源时，吞吐量的提升伴随着不成比例的资金成本增加，GPU 的模型算力利用率（MFU）通常低于 10%。
*   **核心问题**：RL 训练中存在严重的**负载不平衡**。Rollout（采样）阶段受长尾序列、多轮对话和外部工具调用影响，导致 GPU 在解码期间有效 Batch Size 迅速缩小，资源大量闲置。
*   **整体含义**：论文提出了 **JigsawRL**，通过探索“**流水线复用（Pipeline Multiplexing）**”这一新的并行维度，将多个 RL 任务交织在同一组硬件上，以填补资源空隙，提升整体吞吐量。

### 2. 论文提出的方法论
JigsawRL 的核心思想是将粗粒度的 RL 阶段拆解并重新组合，主要包含以下关键技术：
*   **子阶段图抽象（Sub-Stage Graph Abstraction）**：将传统的 Rollout、Training 等阶段进一步细化为具有统一资源特征的“子阶段”（如 Prefill 爆发、低利用率解码、工具等待等），从而暴露阶段内和工作节点间的不平衡。
*   **子阶段空间复用（Sub-Stage Multiplexing）**：利用不同子阶段对计算（SM）和内存的互补需求（例如：将计算密集型的训练与内存带宽受限的解码进行共存），通过 NVIDIA MPS 等技术实现动态资源分配。
*   **子阶段合并与长尾迁移（Sub-Stage Merging）**：监测不同数据并行（DP）工作节点上的进度，将分散的长尾采样任务迁移并聚合到少数节点上，释放其他节点的资源供其他流水线使用。
*   **前瞻性调度（Look-ahead Scheduling）**：将流水线执行建模为图调度问题，使用启发式算法预测不同组合下的减速比（Slowdown），在窗口内选择最优的子阶段组合和资源划分方案。

### 6. 主要结论与发现
*   **吞吐量提升**：相比同步基准 Verl，JigsawRL 实现了高达 **1.85倍** 的吞吐量提升；相比异步框架 StreamRL 和 AReaL，提升达 **1.54倍**。
*   **成本优化**：在相同的 GPU 资源下，JigsawRL 显著降低了单位 Token 的生成成本，且随着 GPU 规模扩大，其优势更加明显。
*   **异构支持**：JigsawRL 能够高效处理不同大小模型（如 0.6B 与 8B 模型）的混合训练，小模型可以灵活填补大模型采样留下的资源空隙。

### 7. 优点（亮点）
*   **新颖的并行维度**：跳出了单流水线优化的局限，通过“流水线复用”解决了 RL 训练中本质的资源利用率难题。
*   **精细化建模**：子阶段图抽象能够捕捉到传统系统无法察觉的微秒级资源闲置。
*   **动态性与自适应**：系统能够根据最近步骤的执行特征自动调整资源分配，无需繁琐的离线调优。

### 8. 不足与局限
*   **延迟权衡**：虽然总吞吐量增加，但单个 RL 任务的完成时间（Latency）会有所增加（约 1.14x - 1.48x），对于追求极速反馈的场景可能受限。
*   **模型架构覆盖**：目前主要针对稠密（Dense）模型进行了优化，尚未深入探讨混合专家模型（MoE）在复用时的专家冲突问题。
*   **内存压力**：在显存极度紧张的场景下，同时运行多个流水线的训练阶段可能会面临 OOM 风险，虽然文中提到了微批次调整，但极端情况下的稳定性仍需更多验证。
