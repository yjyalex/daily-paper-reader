## 摘要
智能体通信语言 (ACL) 使异构智能体能够在不同领域共享知识并进行协作。这种多样性要求具备可扩展性，但极具表现力的扩展机制可能会使输入语言超出完全验证可处理的复杂度类别。我们提出了 CBCL（通用商业通信语言），这是一种将所有消息（包括运行时语言扩展）约束在确定性上下文无关语言 (DCFL) 类别中的智能体通信语言。CBCL 允许智能体将特定领域的“方言”扩展作为一等消息进行定义、传输和采用；通过在 Lean 4 中进行机器检查并在 Rust 参考实现中强制执行的三个安全不变性 (R1--R3)，防止了无限制的扩展，应用了声明的资源限制，并保留了核心词汇。我们在 Lean 4 中对该语言及其安全属性进行了形式化，在 Rust 中实现了一个参考解析器和方言引擎，并进行了基于属性的测试和差异测试，最后提取了一个经过验证的解析器二进制文件。我们的结果表明，同像性 (homoiconic) 协议设计（即扩展定义与普通消息共享相同的表示形式）可以实现可证明的安全性。随着自主智能体越来越多地扩展其自身的通信能力，形式化地限制它们之间可以表达的内容是实施监管的前提条件。

## Abstract
Agent communication languages (ACLs) enable heterogeneous agents to share knowledge and coordinate across diverse domains. This diversity demands extensibility, but expressive extension mechanisms can push the input language beyond the complexity classes where full validation is tractable. We present CBCL (Common Business Communication Language), an agent communication language that constrains all messages, including runtime language extensions, to the deterministic context-free language (DCFL) class. CBCL allows agents to define, transmit, and adopt domain-specific "dialect" extensions as first-class messages; three safety invariants (R1--R3), machine-checked in Lean 4 and enforced in a Rust reference implementation, prevent unbounded expansion, applying declared resource limits, and preserving core vocabulary. We formalize the language and its safety properties in Lean 4, implement a reference parser and dialect engine in Rust with property-based and differential tests, and extract a verified parser binary. Our results demonstrate that homoiconic protocol design, where extension definitions share the same representation as ordinary messages, can be made provably safe. As autonomous agents increasingly extend their own communication capabilities, formally bounding what they can express to each other is a precondition for oversight.

*   **核心问题**：如何在允许智能体在运行时动态扩展通信协议（定义新方言）的同时，确保解析和验证过程在计算上是可处理的（Tractable）且安全的？

#### 2. 论文提出的方法论
论文提出了 **CBCL（Common Business Communication Language）**，其核心思想是利用语言理论安全（LangSec）原则，将协议复杂度严格限制在**确定性上下文无关语言（DCFL）**类别中。

*   **同像性自扩展（Homoiconic Self-Extension）**：方言（Dialect）的定义本身就是有效的 CBCL 消息（S-表达式），可以使用相同的解析机制进行处理。
*   **三项安全不变性（R1–R3）**：
    *   **R1（无递归）**：方言模板仅限声明式模式替换，禁止直接或循环递归，防止无限展开。
    *   **R2（资源限制）**：强制执行静态资源配额（如嵌套深度 ≤ 64，扩展大小 ≤ 8192 字符，验证时间 ≤ 1000ms）。
    *   **R3（核心保留）**：禁止重定义 8 个核心原语（如 `tell`, `ask`, `reply` 等），确保协议引导的不变性。
*   **形式化验证**：使用 **Lean 4** 对语言及其安全属性进行了形式化建模，证明了解析器的完备性、健全性以及“DCFL 保持性”（即安装新方言后，语言仍属于 DCFL 类）。

#### 6. 论文的主要结论与发现
*   **DCFL 足够且安全**：证明了 DCFL 复杂度足以表达嵌套的智能体通信结构，同时能自动消除歧义，防止解析器差异攻击。
*   **安全扩展可行性**：通过 R1-R3 约束，可以在不牺牲安全性的前提下实现运行时的协议自演化。
*   **性能优异**：Rust 实现表明，安全检查的开销极低（微秒级），相对于网络延迟几乎可以忽略不计。

#### 7. 优点
*   **理论扎实**：紧密结合语言理论安全（LangSec）原则，解决了 ACL 领域长期存在的扩展性与安全性失衡问题。
*   **端到端验证**：不仅有数学证明，还有高性能的工业级 Rust 实现，并实现了证明到代码的提取。
*   **同像性设计**：简化了系统架构，使得协议扩展不需要额外的元语言解析器，减少了攻击面。

#### 8. 不足与局限
*   **表达能力上限**：为了安全性，CBCL 限制了表达能力。它无法处理递归数据转换（如展平任意深度的列表）或跨字段的上下文敏感验证（如校验和计算）。
*   **语义一致性**：CBCL 保证了语法安全，但无法强制要求智能体在语义上达成一致（虽然提供了 `:examples` 机制作为辅助）。
*   **信任基础设施**：论文未详细定义密钥分发、撤销等信任管理机制，这些仍需依赖外部基础设施。
*   **应用限制**：对于需要极高性能或极简协议（如某些嵌入式传感器网络）的场景，S-表达式和方言扩展可能显得过于沉重。
