# Abstract 

Existing synthetic tool-use corpora are primarily designed for offline su-pervised fine-tuning, yet reinforcement learning (RL) requires executable environments that support reward-checkable online rollouts. We propose COVERT, a two-stage pipeline that first generates reliable base tool-use trajectories through self-evolving synthesis with multi-level validation, and then applies oracle-preserving augmentations that systematically increase environmental complexity. These augmentations introduce distractor tools, indirect or ambiguous user queries, and noisy, multi-format, or erroneous tool outputs, while strictly preserving oracle tool calls and final answers as ground truth. This design enables automatic reward computation via reference matching for standard cases and lightweight judge-assisted veri-fication for special behaviors such as error detection, supporting RL opti-mization of tool-calling policies. On Qwen2.5-Instruct-14B, COVERT-RL improves overall accuracy on BFCL v3 from 56.5 to 59.9 and on ACEBench from 53.0 to 59.3, with minimal regressions on general-ability benchmarks; when stacked on SFT, it further reaches 62.1 and 61.8, confirming additive gains. These results suggest that oracle-preserving synthetic environments offer a practical RL refinement stage, complementary to SFT, for improving tool-use robustness under ambiguity and unreliable tool feedback. 

# 1 Introduction 

Equipping large language models (LLMs) with tool-use capabilities enables dynamic interac-tion with external APIs, calculators, and code interpreters, extending LLM behavior beyond parametric knowledge (Patil et al., 2024; Qin et al., 2023; Tang et al., 2023; Basu et al., 2024). Recent work has incorporated tool-calling trajectories into both pre-training (DeepSeek-AI et al., 2024) and post-training (Liu et al., 2024a; Xu et al., 2025), substantially improving tool-calling accuracy. Yet practical deployment also demands robustness to ambiguity, distractor tools, noisy outputs, and tool failures—conditions poorly covered by current training sets. Real-world tool-calling logs are sparse, access-constrained, privacy-sensitive, and biased toward successful executions, making it difficult to cover diverse failure modes and edge cases at scale. Synthetic data pipelines (Liu et al., 2024b; Prabhakar et al., 2025; Sengupta et al., 2024; Tang et al., 2023; Qin et al., 2023; Basu et al., 2024; Xu et al., 2025) have emerged as a scalable alternative, constructing millions of tool-grounded examples across diverse APIs, schemas, and task types. However, these corpora are formulated for offline supervised fine-tuning (SFT): each sample consists of a fixed user query, a predefined tool invocation, and a static tool response. As a result, most synthetic tool-use data serves as static supervision rather than as an executable environment for policy optimization. Reinforcement learning with verifiable rewards (RLVR) offers a principled alternative, op-timizing policies through online rollouts with outcome-driven feedback (Bai et al., 2022; Stiennon et al., 2020). Goal. We aim to build an RL-ready tool-use data synthesis pipeline that (i) exposes LLMs to diverse tool-use scenarios at scale, (ii) supports reward computation for online rollouts within the constructed mock environment, and (iii) provides explicit control over several perturbation families, including ambiguous user queries, distractor tools, and noisy or erroneous tool outputs. We focus on robustness-oriented tool use under these controlled perturbations, rather than fully stateful long-horizon agentic planning. Rather than replacing SFT, we position COVERT as a complementary RL refinement stage. 

Our approach. We propose the COntrollable and VERifiable Tool-use data synthesis (COVERT) framework, a two-stage pipeline for constructing RL-ready synthetic tool-use environments. In the first stage, we generate reliable base trajectories through self-evolving synthesis with multi-level validation, each involving simple tools, unambiguous queries, and straight-forward outputs. In the second stage, we apply oracle-preserving augmentations to the environmental components—system prompts, user queries, and tool outputs—introducing distractor tools, indirect queries, and multi-format, noisy, or erroneous tool responses, while strictly preserving oracle tool calls and final answers. This design yields automatically computable rewards via reference matching for most cases, while a lightweight LLM judge handles a subset of special behaviors such as failure detection. We summarize the contributions of this paper as follows: 

• An oracle-preserving RL-ready synthesis pipeline for tool use. Starting from reliable base trajectories, we construct synthetic tool-use environments tailored to online RL, in which oracle tool calls and final answers are preserved as ground truth for reward computation. 

• Explicit augmentation families for robustness-oriented complexity. We introduce con-trolled perturbations over environmental components, including indirect or ambiguous queries, distractor tools, multi-format or noisy outputs, erroneous tool responses, and problematic queries requiring clarification, each targeting a specific robustness capability. 

• Empirical gains on tool-use benchmarks with largely preserved general abilities. On Qwen2.5-Instruct-14B, COVERT-RL improves overall accuracy on BFCL v3 Patil et al. # 2 Related Work 

Research on tool-augmented LLMs can be organized along three axes: (1) constructing synthetic corpora for supervised tool-use training, (2) applying RL in executable tool envi-ronments, and (3) building controllable, oracle-preserving environments for RL—the axis along which COVERT contributes. We survey (1) and (2) below and position our work relative to each. 

2.1 Tool-calling data synthesis Large-scale synthetic corpora for SFT. To overcome the limited volume and diversity of real-world tool-use logs, a growing body of work synthesizes function-calling trajectories at scale. ToolLLM Qin et al. (2023) and Gorilla Patil et al. (2024) collect massive sets of real-world APIs and generate function-calling trajectories over them; ToolAlpaca Tang et al. (2023) creates both APIs and their usages in a multi-agent simulation; and API-BLEND Basu et al. (2024) reformulates and blends data from multiple domains into a comprehensive 2Preprint.

Stage 2 applies oracle-preserving augmentations to produce complex multi-tool scenarios with noisy and multi-format outputs, while preserving oracle tool behaviors for reward computation during RL training. corpus. These efforts address coverage and schema diversity but produce fixed offline trajectories for supervised learning. 

Verification-enhanced synthesis. A second line of work adds construction-time quality checks to improve data fidelity. APIGen (Liu et al., 2024b) employs a hierarchical verification pipeline with format, execution, and semantic checks; ToolACE (Liu et al., 2024a) combines an evolutionary strategy for increasing trajectory diversity and complexity with dual-layer verification; and MAG-V (Sengupta et al., 2024) develops multi-agent pipelines for synthetic data generation and verification. Orthogonally, ToolVerifier (Mekala et al., 2024) proposes contrastive prompting and self-verification to help LLMs distinguish among similar tools. These methods improve example-level correctness at data-construction time, but do not define rollout-time reward signals usable during RL training. 

Multi-turn and MCP-grounded synthesis. More recent pipelines generate multi-turn or real-protocol-grounded trajectories. TOUCAN (Xu et al., 2025) constructs 1.5 million trajectories from nearly 500 real-world Model Context Protocols, and APIGen-MT (Prabhakar et al., 2025) introduces multi-turn data generation via simulated agent–human interactions. Although these pipelines substantially increase interaction realism, they still produce fixed trajectory data for supervised training rather than rewardable environments with explicit, capability-targeted perturbations.

2.2 RL for tool-use LLM training RL with real-world executors. A growing body of work optimizes tool-use policies via RL using real executable tools—search engines Jin et al. (2025); Li et al. (2025a), code interpreters Wang et al. (2024a); Gehring et al. (2024); Liao et al. (2024); Dong et al. (2024), 3Preprint. or combinations thereof Feng et al. (2025); Li et al. (2025b); Zhang et al. (2025b); Dong et al. (2025). These approaches benefit from genuine execution feedback, but each typically covers a narrow, domain-specific tool set with largely fixed schemas and limited failure modes, making it difficult to stress-test robustness across diverse tool-use scenarios. 

RL on pre-existing synthetic data. The closest prior work to ours applies RL over synthetic tool-use datasets originally constructed for SFT. ToolRL Qian et al. (2025) decomposes rewards into format and correctness components against ground-truth tool invocations, while ToolZero Zeng et al. (2025) dynamically shifts from lenient to strict correctness-based rewards during training, using pre-existing synthetic corpora such as those from ToolACE Liu et al. (2024a) and xLAM Zhang et al. (2025a). These methods demonstrate that RL can improve tool-calling behavior even on static datasets, but the underlying data are not designed for RL: tool schemas and outputs are pre-generated, and the complexity of tool interactions is not explicitly controllable. 

Positioning of COVERT. Prior synthesis work primarily improves scale, diversity, and example-level verification for supervised tool-use training, while prior RL work optimizes policies either in narrow real executable environments or over fixed synthetic datasets. COVERT targets a different design point: transforming reliable tool-use trajectories into oracle-preserving training environments, where query ambiguity, tool-set distractors, and tool-output perturbations are systematically controlled for rollout-time reward computation. # 3 Controllable and Verifiable Tool-Use Data Synthesis Pipeline 

Problem setup. We define a base tool-use instance as a tuple (s, u, T , c∗, o∗, a∗), where s is the system prompt, u the user query, T the available tool set, c∗ the oracle tool call, o∗ the oracle tool output, and a∗ the oracle final answer. Stage I of the COVERT pipeline produces a large pool of reliable base instances. Stage II applies a capability-targeted augmentation operator 

Ak indexed by capability type k (e.g., distractor tools, ambiguous queries, erroneous outputs) to transform the environmental components (s, u, T , o∗) while preserving the oracle fields needed for reward computation. Depending on the capability type, the preserved oracle metadata differs: 

• Exact-verifiable augmentations (e.g., distractor tools, query rewriting, multi-format or noisy outputs): both c∗ and a∗ are preserved, enabling deterministic reward computation via reference matching. 

• Judge-assisted augmentations (e.g., erroneous outputs, irrelevant queries, missing-parameter queries): the oracle tool call c∗ is preserved, but the expected final response may differ from a∗; a lightweight LLM judge evaluates correctness given the capability label 

k and the original oracle metadata. Each augmented instance carries a verifier metadata tag mk specifying the capability label and the verifier type (exact or judge-assisted), which Section 4 uses to compute rewards. Figure 1 illustrates the overall workflow. 

3.1 Stage I: Reliable base trajectory generation 

The goal of Stage I is to produce a large pool of base instances (s, u, T , c∗, o∗, a∗) that are simultaneously (i) scalable , (ii) diverse , and (iii) reliable . In Stage I each instance exposes a single callable tool and a single oracle tool-use objective; multi-tool ambiguity and richer interaction complexity are introduced in Stage II. Figure 2 illustrates the pipeline. 

(i) Self-evolving generation. We initialize a seed pool with a small set of manually curated tool-use trajectories. At each generation round, one trajectory is sampled from the pool as an in-context example. A structured prompt—comprising tool specifications, diversity instructions, and the selected example—is fed into an LLM generator to produce candidate trajectories. Candidates that pass all validation checks (described below) are added back to the seed pool, forming a self-evolving mechanism Wang et al. (2023) in which newly validated trajectories serve as in-context examples for subsequent rounds. This iterative bootstrapping scales the dataset without manual intervention. 

(ii) Multi-level validation. Each candidate trajectory passes through three checkers: 4Preprint. 

• Format Checker (deterministic): verifies that tool definitions and tool calls contain the required structural tags ( <tool_call> , </tool_call> ) and that all arguments are valid JSON. 

• Rule Checker (deterministic): enforces tool-schema constraints—required parameters, parameter types, enum membership, and valid argument ranges. 

• Semantic Checker (LLM-based): an LLM judge receives the full trajectory (tool definition, user query, tool call, tool output, final answer) and returns a binary accept/reject decision as-sessing whether the tool choice is contextually appropriate, the arguments are semantically grounded in the query, and the final answer is consistent with the tool output. Only trajecto-ries that pass all three checkers are retained. Retained trajectories are further deduplicated based on normalized tool-call signatures to prevent mode collapse in the self-evolving pool. This multi-level filtering substantially improves the reliability of generated base trajectories. 

Figure 3: Illustration of oracle-preserving augmentation. Starting from a simple and reliable base tool-calling trajectory (left), we apply controlled perturbations to the user query and tool environment (right), including indirect entity references, distractor tools, and injected tool-call failures. 

(iii) Diversity instructions. To promote structural and semantic diversity, we maintain a curated pool of heterogeneous prompt elements: tool domains (e.g., finance, hiring, edi-torial planning), parameter types (string, integer, float, boolean, array), parameter counts, and functional behaviors. In each generation round, elements are randomly sampled and incorporated into the structured prompt alongside the in-context example. This random composition ensures that successive prompts present distinct tool-use scenarios, systemati-cally increasing coverage over domains and parameter configurations.

3.2 Stage II: Oracle-preserving augmentation 

Given a pool of reliable base instances from Stage I, we synthesize more challenging tool-use scenarios by augmenting only the environmental components —user queries u, tool sets T , and 5Preprint. tool outputs o∗—while preserving the oracle fields required by the corresponding verifier (see Table 1). Each augmented instance targets a single capability label k, and carries the verifier metadata mk that specifies whether the instance is exact-verifiable or judge-assisted. System prompts s may also be modified to provide misleading or underspecified context as part of the perturbation. This strict separation between controlled perturbations and preserved oracle metadata is the key enabler for scalable reward computation during RL. 

Exact-verifiable augmentations. For augmentations where the oracle tool call c∗ and final answer a∗ remain valid after perturbation, reward can be computed by reference matching alone. Figure 3 (left → right, upper variants) illustrates this: starting from a base trajectory with oracle call get_stock_price(TSLA) , we introduce indirect entity references in the query and distractor tools in T . The oracle tool call and its correct final answer remain unchanged, enabling deterministic verification of whether the model (i) selects the correct tool and (ii) infers the correct arguments under ambiguity. 

Judge-assisted augmentations. For augmentations where the expected model response differs from the pre-augmentation answer a∗, a lightweight LLM judge evaluates correctness given the capability label and the original oracle metadata. For example, when the tool output is replaced with Connection error (Figure 3, lower variant), the desired behavior is to acknowledge the tool failure and avoid hallucinating a resolved answer. The preserved oracle tool call c∗ together with the explicit capability label substantially reduces the burden on the judge by constraining the verification target to a narrow, capability-specific decision. # 4 Reward Design and RL Training 

Each training instance produced by the COVERT pipeline (Section 3) consists of an aug-mented prompt context, the available tool set T , and verifier metadata mk specifying the capability label k and the verifier type (exact or judge-assisted). Together these define an automatically checkable training environment: most instances are verified by determin-istic reference matching, while a subset of capability-specific behaviors are assessed by a lightweight LLM judge. 

Capability-conditioned reward verification. For exact-verifiable capabilities (e.g., distractor tools, query rewriting, multi-format or noisy outputs), the oracle tool call c∗ and final answer 

a∗ remain valid after augmentation and serve directly as reference signals. The model’s tool call is compared against c∗ (with argument-order normalization and type coercion), and its final answer is compared against a∗.For judge-assisted capabilities (e.g., erroneous outputs, irrelevant queries, missing-parameter clarification), the expected model behavior differs from the pre-augmentation answer a∗. In these cases, an LLM judge receives as input: the capability label k, the aug-mented trajectory (including the perturbed tool output), the oracle tool call c∗, and the candidate model response. The judge returns a binary accept/reject decision evaluating whether the response satisfies the capability-specific behavioral target (e.g., acknowledging a tool failure rather than hallucinating a resolved answer). Because the preserved oracle metadata and the explicit capability label narrow the judge’s decision space to a single behavioral criterion, a lightweight judge model suffices (we use Qwen2.5-Instruct-7B; see Appendix A for details).

RL training. We adopt GRPO Shao et al. (2024) as the RL algorithm, which computes group-wise relative advantages from sampled rollouts and does not require a separate value model, making it well-suited for verifiable-reward settings. During rollout, the model 7Preprint. interacts with a mock tool environment that returns the pre-defined (possibly augmented) tool output for the oracle tool; if the model issues an incorrect tool call, the environment returns a generic error message, and the episode continues until the model produces a final response or reaches the maximum turn limit. We use the same minimal function-calling system prompt during RL training as during evaluation (Appendix A), ensuring that benchmark improvements reflect capability gains rather than prompt-format adaptation. Full training hyperparameters—including rollout batch size, learning rate, maximum turns, and sampling temperature—are reported in Appendix A. 

Scope of verification. We note that the current verification scheme is not uniformly exact: judge-assisted capabilities rely on an LLM judge whose decisions are not deterministic. The preserved oracle metadata substantially constrains the judge’s task, but we have not yet conducted a systematic human-agreement study. We leave systematic judge-reliability evaluation to future efforts. 

# 5 Experiment 

We design our experiments to answer four questions: (RQ1) Does COVERT-RL improve tool-use benchmark performance? (RQ2) Which capability categories benefit most, and where do gains remain limited? (RQ3) Does RL on synthetic tool environments preserve general-domain abilities? (RQ4) Are SFT and COVERT-RL complementary, and what are the boundaries of this complementarity? Table 5 provides headline results; Tables 2–3 give per-category breakdowns; and Table 4 checks for non-tool regressions. 

Evaluation protocol. Unless otherwise stated, all BFCL v3 and ACEBench results are re-ported under a uniform minimal function-calling prompt that exposes only tool signatures without additional behavioral guidance (e.g., no instructions for handling missing param-eters or irrelevant tools). This protocol constitutes a controlled stress test rather than a direct reproduction of leaderboard conditions, and all models—including baselines—are evaluated under the same prompt. Baseline results are obtained by evaluating officially released checkpoints. See Appendix A for the full prompt text and discussion. 

5.1 Benchmarks Tool-use benchmarks. We evaluate on BFCL v3 Patil et al. (2025) and ACEBench Chen et al. (2025), which assess function-calling capability across diverse domains and interaction patterns, including single-turn, multi-turn, and parallel tool calls, as well as scenarios involving irrelevant or distractor tools. The synthetic tools in our training pipeline are disjoint from the benchmark tool definitions in both schema and domain. 

General-ability benchmarks. To check whether RL on synthetic tool environments harms general capabilities, we evaluate on LiveCodeBench-v5 Jain et al. (2024) (code generation), AIME24/25 Mathematical Association of America (2025) and Math500 Hendrycks et al. (2021) (mathematical reasoning), GSM8K Cobbe et al. (2021) (arithmetic), IFEval Zhou et al. (2023) (instruction following), GPQA-Diamond Rein et al. (2023) and MMLU-Pro Wang et al. (2024b) (domain knowledge and scientific reasoning). 

5.2 Models, baselines, and training setup

We consider four model variants: (1) Qwen2.5-Instruct : base model without additional training. (2) TOUCAN-SFT : SFT on 1.5M real-world MCP-based tool trajectories (Xu et al., 2025), initialized from Qwen2.5-Instruct. 5.3 Results Tool-use benchmark gains (RQ1, RQ2). COVERT-RL improves overall accuracy on both BFCL v3 and ACEBench over the base model and the TOUCAN-SFT baseline (Tables 2, 3). On the 14B model, COVERT-RL raises BFCL v3 from 56.5 to 59.9 (+3.4) and ACEBench from 53.0 to 59.3 (+6.3); on the 7B model the gains are +3.1 and +8.6, respectively. The strongest improvements appear in categories that align with our augmentation targets: Irrelevance Detection on BFCL v3 (+11.9 on 14B), and the Special category on ACEBench (+14.2 on 7B), which covers noisy, multi-format, and erroneous tool outputs. However, gains on long-horizon categories remain limited: BFCL Multi-Turn shows no improvement (7B: 10.0 →8.8; 14B: 15.6 →14.9), and ACEBench Agent is similarly flat (7B: 10.2 →9.2; 14B: 14.0 →13.0). This is expected: as discussed in Section 3, the current mock environment targets short-to-moderate interactions and does not yet simulate tightly coupled stateful dependencies across many turns. 

Complementarity with SFT (RQ4). Applying COVERT-RL on top of TOUCAN-SFT yields consistent additive gains (Table 5): the 14B stacked model reaches 62.1 on BFCL v3 (+2.6 over TOUCAN-SFT) and 61.8 on ACEBench (+13.0 over TOUCAN-SFT); the 7B stacked model shows similar patterns (+2.1 / +12.1). Full per-category breakdowns are in Tables 7 and 8 (Appendix B), and training reward curves are in Figure 5. These results indicate that RL targets complementary behaviors—such as robustness to distractor tools, noisy outputs, and error detection—that static supervised data does not fully cover. We note that under our minimal evaluation prompt, TOUCAN-SFT shows unexpectedly large regressions on several categories, most notably ACEBench Special (14B: 68.6 →27.3). Because we evaluate all models under the same prompt, this likely reflects sensitivity to prompt format rather than a fundamental capability deficit; under richer prompts that include behavioral instructions, TOUCAN-SFT may recover some of this gap. 

Preservation of general capabilities (RQ3). Table 4 shows that COVERT-RL largely pre-serves general-domain performance. Across the eight non-tool benchmarks, the largest absolute drop relative to the base model is 0.3 points at the 7B scale (LiveCodeBench-v5) and 1.0 point at the 14B scale (GPQA-Diamond), with several benchmarks showing slight improvements (e.g., IFEval +0.7, GSM8K +0.1 on both scales). In contrast, TOUCAN-SFT exhibits larger regressions on multiple non-tool benchmarks under the same evaluation protocol (e.g., 14B AIME24: 20.0 →10.0; MMLU-Pro: 48.7 →42.0). These results confirm that RL on oracle-preserving synthetic tool environments does not materially degrade general-domain performance.

We presented COVERT, a two-stage pipeline that constructs RL-ready synthetic tool-use en-vironments: Stage I produces reliable base trajectories, and Stage II applies oracle-preserving augmentations—distractor tools, ambiguous queries, noisy or erroneous outputs—that tar-9Preprint. get specific robustness capabilities while preserving oracle fields for reward computation. On Qwen2.5-Instruct-14B, COVERT-RL raises BFCL v3 from 56.5 to 59.9 and ACEBench from 53.0 to 59.3 with minimal general-ability regression; stacked on TOUCAN-SFT it further reaches 62.1 and 61.8, confirming that oracle-preserving RL and SFT yield complementary gains for tool-use training.

# A Experimental Details 

Training setup. All RL experiments are conducted using VERL 1, an open-source RL frame-work for LLM training. We adopt Qwen2.5-Instruct-7B as the LLM judge for capability-conditioned reward verification; the preserved oracle metadata and explicit capability labels narrow the judge’s task to a single behavioral criterion, making a lightweight model suf-ficient (see Section 4). Training is executed on 8 × NVIDIA H200 GPUs, with a wall-clock time of approximately 28 hours per full RL run for the 7B model and ∼1.6 × longer for the 14B model. Both model scales share the same hyperparameter configuration. 

Evaluation protocol. Figure 4 shows the minimal function-calling system prompt used for both training and evaluation on BFCL v3 Patil et al. (2025) and ACEBench Chen et al. (2025). This prompt exposes only tool signatures and instructs the model to produce tool-call outputs in JSON format within <tool_call> tags, without any additional behavioral guidance—no instructions for handling missing parameters, irrelevant tools, or multi-turn dependency resolution. This design constitutes a controlled stress-test protocol: unlike enriched prompts that explicitly specify error-handling strategies (e.g., “If none of the functions can be used, point it out”), our prompt requires the model to infer these behaviors from its own capabilities. Benchmark scores under this protocol should not be compared directly with results obtained under richer prompting conditions. All baseline results are obtained by evaluating officially released checkpoints under the same prompt.

# D Case Studies 

We present two case studies comparing the base Qwen2.5-Instruct-7B model with its COVERT-RL counterpart. Raw transcripts are shown in Figures 9–10. 

Case 1: Parameter handling under incomplete queries. The user requests EC2 pricing but omits optional parameters. When optional parameters (e.g., region and operating system) are not specified by the user, the base model introduces implicit defaults in the tool calls, effectively over-specifying the request and risking unintended assumptions. In contrast, the COVERT-RL model strictly adheres to the required arguments, avoiding ungrounded parameter injection. This demonstrates more faithful interpretation of underspecified queries and improved robustness in tool usage, consistent with gains on ACEBench Special tasks. 

Case 2: Argument normalization. The user asks for weather in “Yosemite National Park, Mariposa, CA” with Celsius units. The base model passes the full descriptive string as the location argument, which may be brittle for API backends expecting standardized inputs. The COVERT-RL model normalizes the argument to “Mariposa, CA,” demonstrating improved grounding of user-provided entities into tool-compatible formats. 

Summary. Both cases illustrate that COVERT-RL improves robustness in argument con-struction and parameter handling—behaviors directly targeted by the oracle-preserving augmentations in our pipeline. 15 Preprint.
