## 摘要
大语言模型（LLMs）在多步结构化工作流中仍然表现脆弱，错误会在顺序转换、验证阶段以及 SQL 持久化等有状态操作中不断累积。我们提出了 PlanCompiler，这是一种用于结构化 LLM 流水线的编译架构，它通过类型化节点注册表、静态图验证和确定性编译将规划与执行分离。该系统不再依赖运行时的自回归链式调用，而是首先根据固定的原语注册表生成类型化 JSON 计划，根据显式的结构和类型约束验证该计划，并仅将验证通过的计划编译为可执行的 Python 代码。我们在一个包含 300 个任务的基准测试上评估了该方法，涵盖了不断增加的工作流深度、SQL 往返持久化以及以模式为主题的压力测试。在此设置下，PlanCompiler 在集合 A 和 B 上实现了 100% 的首轮成功率，在集合 C 上为 88%，在集合 D 上为 96%，在模式陷阱任务上为 88%，在 SQL 往返任务上为 84%，在六个基准测试集中的五个上优于 GPT-4.1 和 Claude Sonnet 的直接自由格式代码生成基准，总体实现了 278/300 的成功，而两个基准模型分别为 202/300 和 187/300。在整个测试套件中，规划成本约为 0.356 美元，而 GPT-4.1 为 2.140 美元，Claude 为 18.391 美元，同时保持了具有竞争力的端到端延迟。这些结果表明，对于受注册表约束的结构化数据工作流，与自由格式代码生成相比，确定性编译可以提高首轮可靠性和成本效率。残余失败集中在两类狭窄的范围内：聚合任务中的后期输出契约错误，以及 SQLite 持久化边界处的早期类型不匹配，这阐明了该方法的优势和当前的局限性。

## Abstract
Large language models (LLMs) remain brittle in multi-step structured workflows, where errors compound across sequential transformations, validation stages, and stateful operations such as SQL persistence. We present PlanCompiler, a compilation architecture for structured LLM pipelines that separates planning from execution through a typed node registry, static graph validation, and deterministic compilation. Instead of relying on autoregressive chaining at runtime, the system first produces a typed JSON plan over a fixed registry of primitives, validates that plan against explicit structural and type constraints, and compiles only validated plans into executable Python.   We evaluate the approach on a 300-task benchmark covering increasing workflow depth, SQL roundtrip persistence, and schema-themed stress tests. In this setting, PlanCompiler achieves 100% first-pass success on Sets A and B, 88% on Set C, 96% on Set D, 88% on schema-trap tasks, and 84% on SQL roundtrip tasks, outperforming direct free-form code-generation baselines from GPT-4.1 and Claude Sonnet on five of six benchmark sets and achieving 278/300 successes overall versus 202/300 and 187/300 for the two baselines, respectively. Across the full suite, planning cost is approximately \$0.356, compared with \$2.140 for GPT-4.1 and \$18.391 for Claude, while maintaining competitive end-to-end latency.   These results suggest that, for registry-constrained structured data workflows, deterministic compilation can improve first-pass reliability and cost efficiency relative to free-form code generation. Residual failures are concentrated in two narrow classes: late output-contract errors on aggregation tasks and early type mismatches at the SQLite persistence boundary, clarifying both the benefits and the current limits of the approach.

#### 1. 核心问题与整体含义（研究动机和背景）
大语言模型（LLM）在处理单步任务（如编写简单 SQL 或函数）时表现良好，但在涉及多个顺序步骤的**结构化工作流**（如数据摄取、转换、数据库持久化、导出）中表现脆弱。其核心问题在于**错误累积（Error Compounding）**：第二步的幻觉或命名不一致会传播到后续步骤，导致整个流水线失效。现有的框架（如 LangChain 或工具编排系统）通常在运行时处理失败，缺乏预执行的结构化保证。本文旨在通过一种“确定性编译”架构，将规划（Planning）与执行（Execution）严格分离，消除幻觉导入、命名漂移和布线错误。

#### 2. 方法论：核心思想与技术细节
PlanCompiler 的核心思想是**将 LLM 的角色限制为在受限注册表上的规划器**，而非直接生成代码。其架构包含五个阶段：
*   **类型化节点注册表（Typed Node Registry）：** 定义了一组固定的计算原语（Node），每个节点包含名称、输入/输出类型、必需参数和预定义的 Python 模板。
*   **规划器（Planner）：** LLM 接收任务描述和注册表信息，输出一个结构化的 JSON 计划（包含节点选择、参数绑定和边连接）。
*   **静态验证层（Static Validator）：** 对 JSON 计划执行 **7 项确定性检查**：
    1. 节点存在性；2. 边合法性；3. 类型兼容性（如 DataFrame 不能直接传给 DBHandle）；4. 无环性（Kahn 算法）；5. 无孤立节点；6. 输入入度（当前仅支持单流）；7. 必需参数完整性。
*   **确定性编译器（Compiler）：** 验证通过后，通过拓扑排序将节点模板组装成完整的 Python 程序。
*   **执行模型：** 在独立子进程中运行生成的代码，节点间通过函数返回值传递状态，无共享内存或隐式通信。

#### 6. 论文的主要结论与发现
*   **可靠性显著提升：** PlanCompiler 实现了 **92.67%** 的总成功率，显著优于 GPT-4.1（67%）和 Claude（62%）。
*   **成本效益极高：** 相比基准模型，PlanCompiler 的单次成功任务成本降低了 8 到 77 倍。
*   **失败模式局部化：** 系统的失败不再是随机的幻觉，而是集中在两个特定点：

#### 7. 优点（亮点）
*   **架构解耦：** 通过“规划-验证-编译”流程，将 LLM 的非确定性限制在规划阶段，执行阶段完全确定。
*   **结构化保证：** 静态验证层能在代码运行前拦截大部分逻辑错误（如类型不匹配、循环依赖）。
*   **高效廉价：** 证明了通过严密的架构设计，使用较弱/较便宜的模型（GPT-4o-mini）可以超越更强模型在特定任务上的表现。

#### 8. 不足与局限
*   **拓扑限制：** 当前实现主要支持单流（Single-stream）流水线，尚不支持复杂的分支、并行子图或循环控制流。
*   **库依赖性：** 系统的能力受限于预定义的节点注册表，如果任务超出注册表范围，规划器将无法处理。
*   **语义鸿沟：** 验证层只能检查结构正确性，无法完全保证语义正确性（例如 SQL 字符串内部的逻辑错误）。
*   **缺乏自动修复：** 当验证失败时，系统直接终止，尚未引入自动化的“验证-反馈-修正”循环。
