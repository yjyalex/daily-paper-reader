## Abstract 

Multi-agent systems (MAS) demonstrate clear ad-vantages in tackling complex problems by coor-dinating diverse agents and external tools. How-ever, most existing orchestration methods rely on static workflows or serial agent scheduling, and are further constrained by heterogeneous interface protocols between tools and agents. This leads to high system complexity and poor extensibility. To mitigate these issues, we propose Agent-as-Tool, a unified parallel orchestration paradigm that abstracts both agents and tools into a stan-dardized, learnable action space with protocol nor-malization and explicit state feedback. Building on this paradigm, we train a lightweight orches-trator, ParaManager, which decouples planning decisions from subtask solving, enabling state-aware parallel subtask decomposition, delegation, and asynchronous execution. For training, we adopt a two-stage ParaManager training pipeline. It improves robustness by incorporating super-vised fine-tuning (SFT) trajectories equipped with recovery mechanisms, and further applies rein-forcement learning (RL) to achieve an optimal balance among task success, protocol compliance, diversity, and reasoning efficiency. Experiments show that ParaManager achieves strong perfor-mance across multiple benchmarks and exhibits robust generalization under unseen model pools.

Consequently, MAS with tool-augmented rea-soning have emerged as key directions. These frameworks leverage role specialization and external tools to provide a more reliable approach for complex tasks (Hong et al., 2024; Ye et al., 2025). Many MAS rely on static orchestration, where fixed topolo-gies or workflow templates are predefined and remain un-changed throughout execution (Leong et al., 2025; Yu et al., 2025a). This design assumes that decomposition of sub-tasks and scheduling can be determined upfront, rather than being driven by intermediate evidence and observed fail-ures (Zhang et al., 2025b; Li et al., 2024b). In practice, orchestration is inherently stateful. The system must re-act to task states as well as execution signals (Wang et al., 2025a). Without explicit state feedback mechanisms, faults at execution can propagate across intermediate steps, lead-ing to brittle long-horizon trajectories (Su et al., 2025). To mitigate template rigidity, recent work explores learned routing and dynamic scheduling (Dang et al., 2025; Su et al., 2025). However, orchestration remains challenging because the action space is often constrained. On the one hand, workflows remain serial, limiting throughput and preventing multi-path execution (Yu et al., 2025a; Wang et al., 2025a). On the other hand, systems depend on heterogeneous tools and predefined agent subtasks, which require a set of spe-cialized agents for every potential function. This forces the orchestrator to handle incompatible tool schemas and failure semantics, hindering flexible repurposing of agents as the problem context evolves (Zhang et al., 2025c). We propose a unified parallel orchestration paradigm for agents and tools, Agent-as-Tool , which treats each agent as a tool with a standardized input-output protocol and an explicit execution state. This design places multi-agent col-laboration and external tool use into a single learnable tool space, enabling orchestration through a unified action rep-resentation. Under this paradigm, we train a lightweight parallel orchestrator, the ParaManager , as the decision core of the system. At each round, ParaManager conditions on the current reasoning states to generate state-grounded sub-1

This separation clarifies responsibilities: the ParaManager focuses on planning and decomposition, while invoked tools handle subproblem reasoning and evidence generation in isolated executions. This isolation reduces long-context interference and limits error propagation (Su et al., 2025; Shen et al., 2026). The multi-round parallel orchestration provides redundancy across independent reasoning paths and reduces single-path failure risk compared with purely serial reasoning (Wang et al., 2025b). To realize this paradigm, we propose a two-stage framework for training a lightweight LLM as an effective orchestrator. With a re-ward design that balances task success, protocol compliance, scheduling diversity, and efficiency, the framework yields a compact orchestrator capable of reliably coordinating agents and tools. Unified Agent-as-Tool paradigm with a lightweight Para-Manager. We unify agents and external tools into a single tool space via a standardized action schema and explicit feedback. By using a compact ParaManager to decouple high-level planning from low-level solving, we significantly reduce orchestration complexity. 

State-driven decomposition, subtask delegation, and par-allel orchestration. Our framework dynamically decom-poses complex queries and delegates subtasks to appropriate agents and tools based on task characteristics. By enabling multi-round parallel tool calls, we improve reasoning accu-racy through independent path validation and boost execu-tion efficiency compared to purely serial workflows. 

ParaManager training with failure-recovery supervision and system-level RL objectives. We develop a compre-hensive training pipeline with SFT and RL that combines recoverable failure trajectories and masked supervision, and optimizes for task success, protocol compliance, diversity, and efficiency. This approach ensures the manager remains robust against execution instability and prevents collapse. 

Superior performance and cross-model generalization. 

Extensive experiments on challenging benchmarks demon-strate that ParaManager consistently outperforms static workflows and strong orchestration baselines. Crucially, our framework exhibits robust generalization under model-pool shift, maintaining high efficacy when orchestrating unseen, larger-scale agent models without additional tuning. 

## 2. Related Work 

2.1. Automated Workflow Search and Optimization 

Optimizing agent collaboration structures and workflow topology is a critical challenge in MAS (Yang et al., 2025; Chen, 2025). Since manual design is labor-intensive and lim-ited in search scope (Hu et al., 2024), automated optimiza-tion has become a research focus. Several works optimize workflows as code or graphs. AutoFlow (Li et al., 2024b) leverages LLMs to iteratively construct task-specific work-flows. AFlow (Zhang et al., 2025b) formulates workflow search as a Monte Carlo Tree Search problem, exploring code-space operators to surpass manual designs. Evolution-ary algorithms are also prevalent: EvoFlow (Zhang et al., 2025a) uses multi-objective evolution to generate diverse, heterogeneous workflows, balancing performance and cost. Finally, X-MAS (Ye et al., 2025) targets heterogeneous node configuration, assigning specific models to optimal roles to enhance collective intelligence without altering topology. 

2.2. Agent Scheduling and Orchestration 

Early systems often depended on static topologies or manual heuristics, limiting adaptability in dynamic environments (Jiang et al., 2025; Leong et al., 2025; Du et al., 2025). Research has shifted toward learning-based dynamic orches-tration. To overcome static limitations, Puppeteer (Dang et al., 2025) models collaboration as a Markov Decision Process and trains an RL-based orchestrator to dynamically activate agents based on real-time states. ToolOrchestra (Su et al., 2025) introduces an efficient RL-based method by training a lightweight orchestrator to coordinate diverse tools using outcome, efficiency, and preference rewards, reducing overhead while enhancing performance. Existing studies improve MAS either by optimizing work-flow topology or by learning orchestration policies, yet most rely on heterogeneous interfaces and do not unify agent and tool calls within a single action space, making state-driven decomposition and parallel scheduling hard to learn end-to-end. As shown in Figure 1, we address this gap with Agent-as-Tool, featuring tool-space unification, state-aware task decomposition, and parallel orchestration.

## 4. Experiment 

4.1. Tools Basic tools. We execute model-generated Python code via the sandbox interpreter (Bytedance-Seed-Foundation-Code-Team et al., 2025). The executor returns stdout/stderr verbatim, so the model can iterate on failing programs rather than restarting from scratch. For retrieval, we follow Search-R1 (Jin et al., 2025b;a) with e5-base-v2 (Wang et al., 2022) as the retriever and Wikipedia as the corpus. 

Agent tools. Agent tools take a subtask description and a model identifier and return a solution under a fixed I/O protocol: Standard Reasoner produces direct answer with-out external tool calls. Critical Reviewer inspects the rea-soning history, detects potential issues, and suggests fixes. 

Code Reasoner writes and runs Python code for a subtask. 

Knowledge Searcher searches for evidence for a subtask. 

Ensemble Solver performs multi-path sampling and returns an aggregated result. Final Answer End the reasoning and return the final answer Detailed definitions and tool cost units are provided in Appendix E. 

4.2. Data Collection RL Data. We derive RL instances from DAPO-Math (Yu et al., 2025b). For each question, we sample eight outputs using Qwen3-30B-A3B-Instruct-2507 (Team, 2025) and discard questions whose eight samples are all correct or all incorrect: the former yields near-zero preference sig-nal, while the latter may reflect unsolvable or low-quality questions. The resulting RL set contains 3,742 instances. 

SFT Data. We collect SFT trajectories to initialize Para-Manager. We run Qwen3-30B-A3B-Instruct-2507 on the DAPO-Math training split using our framework and keep trajectories that end with the correct answer. To avoid leak-age, we remove any question that overlaps with the RL set. We sample eight trajectories per instance and keep at most one, prioritizing runs that recover from tool failures. To avoid tool imbalance, we enforce a simple distribution con-straint so that no single tool dominates the retained set. This yields 4,470 SFT instances.

4.4. Baselines and Evaluation Pure model baselines. This group measures the native performance of a single model. We also include two test-time scaling strategies, majority voting at 8 samples and self-refinement (Madaan et al., 2023). In addition, we evaluate tool-augmented single-model performance of each model. 

Static workflow baselines. This group uses predefined static workflows for task scheduling. Representative methods include Router (Ong et al., 2024), Meta Agent Search (Hu et al., 2024), and EvoFlow (Zhang et al., 2025a). For Router, we build a problem bank from the DAPO-Math training split, retrieve similar problems using k-nearest neighbors, and select the output from the model that scores best on the retrieved set. For Meta Agent Search, we use Qwen3-30B-A3B-Instruct-2507 as the base model to con-struct the workflow. For EvoFlow, we construct the work-flow using the full model pool. 

Serial orchestration baselines. This group evaluates dy-namic yet serial orchestration strategies. We compare against Puppeteer (Dang et al., 2025) and ToolOrchestra (Su et al., 2025). We use the officially released weights with our tool suite to ensure a fair comparison. 

Benchmarks. We evaluate both in-domain math bench-marks and out-of-domain general benchmarks. For in-domain evaluation, we adopt four math datasets: AIME24, AIME25, AMC (Li et al., 2024a), and HMMT25 (Balunovi ´c et al., 2025), which contain high-difficulty problems. ## 5. Analysis 

5.1. Why SFT Initialization Is Necessary 

In the Agent-as-Tool paradigm, the orchestrator must gen-erate structured calls that satisfy format and semantic con-straints in order to be executable and to yield valid envi-ronment feedback. Starting RL directly from a base model is often unstable because early policies rarely produce ex-ecutable actions consistently, causing most trajectories to terminate with parsing or execution failures and resulting in prolonged sparse rewards. Under such limited supervision, optimization can drift toward locally stable but degenerate behaviors, most notably avoiding tool usage and reverting to single-turn direct answering, which leads to the collapse shown in Figure 3. SFT initialization allows RL to focus on system-level objectives rather than learning action schemas and failure semantics from scratch. Moreover, by including trajectories that recover from errors, the model learns to leverage structured failure signals for self-repair, reducing invalid calls and improving the stability of subsequent RL. 

5.2. Efficiency Reward Analysis 

We analyze the role of the efficiency reward during training. Figure 4 shows the tool cost reward and the length reward over training. In the early stage, both signals drop rapidly, indicating that the policy explores more aggressively by invoking tools more often and producing longer reasoning traces. As training progresses, the cost and length signals gradually rise and then stabilize. This suggests that the model learns to reduce unnecessary tool invocations and redundant reasoning, controlling computational cost while maintaining output quality.

## C. Efficiency reward design 

We use the efficiency reward defined in Section 3.3 to regularize computation, discouraging redundant reasoning and excessive tool usage. We present a study that illustrates how the cost target trades off accuracy and computational budget. 

Choosing Ltar = 12 ,288 . In the main experiments, we set the length target to Ltar = 12 ,288 tokens. This value aligns with the global maximum generation length of 24 ,576 tokens. Setting Ltar to half of the cap calibrates the regularization so that typical-length trajectories receive little penalty, while substantially longer ones are gradually discouraged. 

Choosing Ctar = 8 . We use Ctar = 8 as the default tool-cost target to balance accuracy and efficiency. Table 6 motivates this choice. When the budget is reduced to Ctar = 4 , accuracy drops by 3.14 points, which indicates that the policy becomes overly conservative in tool usage and loses cross-verification benefits. When the budget is increased beyond 8, accuracy improves only modestly. The gain is 0.71 points at Ctar = 12 and 0.84 points at Ctar = 16 . At the same time, token consumption rises noticeably, which suggests diminishing returns. Therefore, Ctar = 8 provides a robust operating point that preserves most performance benefits of tool usage without unnecessary overhead. To illustrate the trade-off controlled by the tool-cost threshold Ctar , Table 6 reports the resulting accuracy and token usage as we vary Ctar ∈ { 4, 8, 12 , 16 } while keeping other settings fixed, including Ltar . Increasing Ctar permits more tool calls and cross-verification, which generally improves accuracy while increasing token consumption. The marginal gains become small once Ctar exceeds 12 .

E.1. Tool Use: Accuracy–Token Trade-off Across Backbones 

To characterize both the benefits and the overhead of tool use, we compare in-domain math accuracy and average reasoning tokens under B ASE versus B ASE +T OOL for three backbones (Table 8). The results show that tools are not a uniform “free lunch”: reducing tokens does not necessarily translate into higher accuracy, and the net effect depends strongly on the backbone. For Qwen3-30B-A3B-Instruct-2507, B ASE +T OOL reduces the average reasoning tokens by 31%, yet the math accuracy drops by 4.13 points. This suggests that, for this model, tool calls can substitute for and compress parts of the model’s internal reasoning, which may remove intermediate derivations that are crucial for harder competition-style problems. In contrast, GPT-OSS-20B exhibits only a minor token change (4% reduction) but gains 3.96 accuracy points, indicating that tools mainly serve as complementary modules for verification and computation rather than shortening the reasoning process. Qwen3-Coder-30B-A3B benefits most: B ASE +T OOL simultaneously reduces tokens by 43% and improves accuracy by 10.47 points, consistent with its stronger ability to translate code-centric tool feedback into effective numerical reasoning. Overall, these comparisons highlight that the value of tools is mediated by the quality of the calling policy. Triggering tools too early or too frequently can interrupt necessary chain-of-thought reasoning, especially on high-difficulty math benchmarks.
