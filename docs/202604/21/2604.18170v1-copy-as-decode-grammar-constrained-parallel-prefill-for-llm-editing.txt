## Abstract 

Large language models edit text and code by autoregressively regenerating the full output, even when most tokens appear verbatim in the input. We study a decoding-layer mechanism, Copy-as-Decode , that recasts edit generation as struc-tured decoding over a two-primitive grammar— <copy lines="i-j"/> references an input line range, <gen> ... </gen> emits new content. A token-level FSM guarantees syntactic validity, and a serving-layer primitive updates the KV cache for each copy span via a single parallel-prefill forward rather than N autoregressive steps, sharing the parallel-forward kernel of speculative decoding but with input tokens as the draft and program-enforced acceptance in place of probabilistic veri-fication. The contribution is the mechanism together with an upper-bound analysis 

of its payoff that requires no end-to-end model training. We report two quantitative bounds and one correctness property. (i) Kernel speedup: on Qwen2.5-{1.5B, 7B}, copying N tokens via parallel prefill is 6.8×–

303 × faster than emitting them autoregressively ( N ∈ [8 , 512] , A100 80GB bf16). 

(ii) Copy ceiling: on ProbeEdit and HumanEvalPack-Fix (Py/JS), 74 –98% of gold tokens are reachable under the line-level primitive; composed with the empirical kernel s(N ) over each corpus’s exact span histogram this yields the closed-form wall-clock bound bnd exact = 29 .0 × /3.4 × /4.2× (ProbeEdit / Py / JS; 13 .0×

pooled). A token-level extension reaches 91 –99% coverage with conservative floors of 4.5–6.5×. (iii) Pipeline losslessness: oracle programs round-trip through the deterministic resolver on all 482 cases, an identifiability precondition that localizes any downstream failure to span selection rather than the mechanism. A perturbation study shows pooled EM drops from 100% to 15 .48% under off-by-one noise, quantifying the span-selection precision required. A supervised fine-tuning pilot on Qwen2.5-Coder-1.5B ( 131 –385 oracle programs, 3 seeds, 99 

held-out attempts per configuration) lifts HEvalFix-Py EM from 0/33 (untrained, any strategy) to 12 –17% (Wilson 95% CI [7 .0, 26 .0] ), a learnability signal, not a production-accuracy span selector. We do not claim batched-serving integration or agentic multi-file coverage; these are scoped as direct follow-up. 

## 1 Introduction 

LLM-based editing of text and code is pervasive—code-assistant apply loops, notebook rewrites, collaborative document revision—yet the dominant serving strategy is wasteful. Given an input document D and an instruction, the model emits the edited output D′ autoregressively, one token at a time, from scratch. Because typical edits preserve most of D verbatim (e.g. 74 –98% of gold tokens reproducible from D under our line-level copy primitive, §4.2), most of those forward passes regenerate content the model has already attended to in its prompt. This has two compounding costs: wall-clock latency scales with output length, not with edit size, and the model is free to drift on tokens that should have been fixed by the edit contract.

Scope of this paper. We study a decoding-layer mechanism, not a production editing system. We (a) formulate the mechanism, (b) establish two quantitative upper bounds and one correctness property—composed into a single span-distribution-aware exact bound bnd exact (Table 2) that holds independently of base-model quality, (c) characterize, via a controlled perturbation study, the span-selection precision a downstream trained system would need to realize those bounds, and (d) run a small-scale SFT pilot (§5) that reaches 12 –17% mean-over-3-seeds held-out EM ( 95% 

CI [7 .0, 20 .1] and [11 .0, 26 .0] respectively) at 1.5B on 131 –385 examples—clearing the untrained 

0% baseline decisively but below deployment. We do not integrate the splice with batched-serving stacks (vLLM/SGLang) or evaluate on agentic multi-file workloads (SWE-bench-style); our end-to-end spot-check on a 7B untrained base (Appendix P) places the current model relative to the oracle ceiling but is not itself the efficiency claim. Prior work addresses edit serving only partly. Search-and-replace formats [ 6 ] and unified diffs reduce output tokens but still re-emit structural context through autoregressive decoding, and suffer anchor-ambiguity failures we quantify in §4.4 (search/replace round-trip-EM drops to 0.81 –0.94 on our corpora where C OPY -AS -D ECODE remains at 1.00 ). Speculative decoding [ 1, 13 , 14 , 28 ] accelerates free-form generation with probabilistic acceptance but offers no token-identity guarantees on copy-heavy regions; production apply tools [ 4 ]are trained on proprietary data and not directly reproducible. We take a simpler view: if 90%+ of output tokens are a priori present as contiguous input spans, the act of copying them should not cost the same as the act of generating new ones. Concretely: 

Grammar over two primitives. We replace free-form output with a programmatic grammar 

> program := ’<program>’ (copy | gen)* ’</program>’ copy := ’<copy lines="i-j"/>’ gen := ’<gen>’ free_text ’</gen>’

enforced by a token-level finite-state machine that restricts the model’s logits to tokens whose decoded prefix is consistent with the current grammar state. The decoder has three branches at each op boundary (start a copy, start a gen, end the program); inside a copy op the model picks only the numeric line indices; inside a gen op the model runs free until </gen> . The grammar guarantees 100% parse rate by construction. 

Copy spans cost one parallel prefill. After the model commits to <copy lines="i-j"/> , the tokens for lines i..j are already known—they are literal tokens from the input. We splice them into the output sequence and update the KV cache with a single parallel-prefill forward, skipping N

autoregressive decode steps for a span of N tokens. This shares the parallel-forward acceleration primitive of speculative decoding [ 1, 13 , 25 ] but replaces probabilistic verification with grammar-2enforced acceptance: input tokens serve as the “draft” with acceptance probability 1 by construction. The quality trade-off lives entirely in span selection. 

Two quantitative bounds + one correctness property, without end-to-end training. The mech-anism decomposes into three independently measurable properties. Two are quantitative bounds on the payoff any copy-aware decoder can achieve (kernel speedup and copy ceiling); the third is a correctness precondition, not an independent bound (pipeline losslessness—without it an EM failure cannot be attributed to span selection). Composed, the first two yield the span-distribution-aware exact bound bnd exact (Table 2) that defines the payoff envelope any future trained system would have to fit inside. None of the three requires training a span selector. 1. Kernel speedup (§4.1). For spans of length 8–512 and a 1024-token prefix on Qwen2.5-1.5B / 7B (single-sequence A100 80GB bf16), a parallel-prefill forward is 6.8×–303 × faster than an autoregressive loop over the same tokens. Speedup grows approximately linearly with span length in the overhead-dominated regime and saturates at the compute-bound regime. 2. Copy ceiling (§4.2). For ProbeEdit (154 short-text edits) and HumanEvalPack-Fix (164 Python + 164 JS bug fixes), 74 –98% of gold-output tokens are realizable as contiguous line-level copy spans under oracle alignment—the ceiling for the mechanism as defined in §2. Composed with the measured kernel s(N ) curve over each corpus’s actual span histogram, this yields the exact closed-form wall-clock bound bnd exact = 29 .0× (ProbeEdit), 3.4× (HEvalFix-Py), 4.2× (HEvalFix-JS), 

13 .0× pooled. A token-level contiguous-span cover (an aspirational upper bound, not the current primitive) reaches 91 –99% with conservative floors at 4.5–6.5×, motivating the natural extension to sub-line copy discussed in §7. 3. Pipeline losslessness (§4.3). With oracle programs induced from difflib.SequenceMatcher 

alignment of gold to input, the deterministic resolver reproduces gold on all 482 cases across three corpora under byte-equal comparison (up to a single trailing newline); whitespace-trimmed EM is also 482 /482 . This is an identifiability precondition, not a payoff bound: any downstream exact-match failure is therefore a model span-selection problem, not a mechanism bug. A perturbation study (Table 8) shows the mechanism is unforgiving: pooled EM drops from 100% to 15 .48% 

under off-by-one noise on oracle line indices. Kernel speedup and copy ceiling are systems and data properties that do not depend on model quality; pipeline losslessness establishes that the resolver faithfully inverts an oracle program. Closing the gap between the oracle ceiling and a realized deployed system is a model-training question—picking the right spans—that we scope as follow-up work (§7). The perturbation study in §5 quantifies how tight that selector has to be; it is a hardness diagnosis, not itself a feasibility result. A prompt-level variant of the grammar (the only contribution of the paper superseded by this one, CRP) is retained as an ablation for closed-model deployment without KV-cache access (Appendix Q). 

## 2 Method: Copy-as-Decode 

Copy-as-Decode sits at the intersection of structured constrained decoding and speculative decoding (Figure 1). The mechanism has three components—a two-primitive grammar, a token-level finite-state machine that enforces it, and a parallel-prefill splice that realizes a copy span in one forward pass—each introduced below. 

2.1 Grammar and Problem Formulation 

Given an input document D = l1l2 · · · ln (indexed by lines) and an edit instruction I, the goal is to produce the edited document D′. Rather than autoregressively generating D′ token by token, we have the model emit a program over a two-primitive grammar: 

Program ::= <program> Op ∗ </program> 

Op ::= Copy | Gen Copy ::= <copy lines=" i-j"/> 

Gen ::= <gen> FREE TEXT </gen> 

A deterministic resolver R consumes a Program P and the immutable input D, and yields D′ =

The deterministic resolver R (§2.2) expands P against D to yield D′ (right); the color bar on each output line indicates its origin. 2.2 Resolver Semantics 

The resolver is deterministic and pure: 41. For each <copy lines=" i-j"/> op, emit the verbatim text of li · · · lj from D, joined by newlines. 2. For each <gen> s</gen> op, emit s with at most one leading and one trailing newline stripped (a readability allowance, preserving internal blank lines). 3. Concatenate ops with a single newline separator. The resolver’s default behavior on out-of-range or malformed indices is to raise, but a single flag toggles between strict (raise on any malformed op) and clipped (clip indices to [1 , n ], emit a warning) semantics—we benchmark both modes explicitly in Table 4. It has no nondeterminism and no dependence on the model. §4.3 measures pipeline losslessness : on all 482 (doc, gold) pairs across three corpora, oracle programs induced from line-level alignment reconstruct gold exactly under strict semantics—the strict-vs-clipped distinction matters only at deployment under a model that can emit invalid indices, not for the oracle-derivation pipeline itself. 

Reserved literals and principled escape. The grammar’s five structural substrings ( <copy ,

</copy> , <gen> , </gen> , </program> ) must not appear verbatim inside a <gen> -body. We handle this in two complementary places: • Preprocessing escape. Each reserved substring in gold is replaced by a Unicode entity (e.g. 

</gen> →〈/gen 〉 via U+27E8/U+27E9/U+2024, unused by all tokenizers we checked); resolver reverses before emit. For any gold g, decode(encode( g)) = g and encode is injective on inputs free of the entities—holds on 482 /482 cases and 10 ,000 adversarial fuzz strings (Appendix I). • FSM end-of-gen detector. Exact token-id-sequence match against the 3-token Qwen BPE encoding of </gen> , not a substring check on decoded text. Byte-level overlaps split across token boundaries cannot mis-trigger the detector (Appendix I). 

2.3 Token-Level Grammar-Constrained Decoder 

A free-form decoder cannot be trusted to emit a valid Program: after a copy-span splice it has just “seen itself output” a long run of document-like tokens and naturally continues regenerating content. We enforce the grammar at the token level via a small FSM (5 states: AWAIT_OP , AFTER_LT ,

COPY_LINES , GEN_BODY , CLOSE ; see Appendix A) that intersects the model’s logits with the set of tokens consistent with the current state. For Qwen2.5-family tokenizers (and any BPE tokenizer encoding the structural literals as fixed token sequences), the relevant token ids are fixed constants. The FSM alternates three step types: • Masked model steps at decision points: argmax restricted to {<, </ } at op boundaries; {copy , gen }

for op type; digits plus -/" inside a copy’s line range. • Forced-template prefills for fixed literal sequences ( _lines=" , "/> , </gen> , </program> ), each spliced via a single parallel-prefill forward. • Free generation inside <gen> bodies until the 3-token Qwen encoding of </gen> is produced (Appendix I). The structural portion is therefore never subject to model error; the model chooses only op types, line indices, and <gen> -body content. Parse rate is 100% by construction. 

2.4 Copy-Span Splicing via Parallel Prefill 

When the FSM exits the copy-line-index state with a chosen range [i, j ], we tokenize li · · · lj directly from D and append those tokens to the KV cache in a single forward pass: model (input_ids = [[ li · · · lj ]] , past_key_values = KV) 

This is an ordinary transformer prefill forward over N = |[[ li · · · lj ]] | tokens, not N decode forwards. The forward still computes all self-attention and feed-forward activations over the N tokens, but it does so in parallel within one launch, amortizing the per-step kernel overhead and, on modern hardware, running in the GEMM-bound regime rather than the memory-bandwidth-bound regime typical of single-token decode. ## 4 Results 

We present three measurable properties: (i) kernel speedup (what a single copy splice buys in wall-clock, §4.1), (ii) copy ceiling (how often copy-heavy structure is actually present in editing data, §4.2), and (iii) pipeline losslessness (the resolver reproduces gold from oracle programs, §4.3). The first two are quantitative bounds; composing them yields the exact, span-distribution-aware wall-clock upper bound bnd exact on any copy-aware decoder for these corpora (Table 2). The third is a correctness property, not an independent payoff bound—without it a downstream EM failure cannot be attributed to span selection. All three are data-level / systems-level properties of the mechanism; none requires an end-to-end trained model.

4.3 Pipeline Losslessness 

Oracle programs are derived by deterministic difflib.SequenceMatcher line-level alignment (no model) and resolved through the same code path invoked at deployment. Reconstruction is byte-exact on all 482 /482 cases across ProbeEdit ( 154 ), HEvalFix-Py ( 164 ), and HEvalFix-JS ( 164 ), under both byte-exact (tolerant of a single trailing newline) and trimmed ( .strip() ) equality. A grammar-literal audit finds 0/482 gold outputs containing the reserved substrings </gen> or <copy . This establishes: (i) the resolver is lossless on all cases up to at most a single trailing newline; (ii) any downstream exact-match failure decomposes as span-selection error , not resolver error; (iii) the naive grammar suffices on these corpora, though general-purpose deployment requires the principled escape of §2.2.

Pointer-decoder baseline. A natural architectural alternative is a span-level pointer decoder [ 26 ]: attach three classification heads to a base LM (op-type, span-start, span-end), training end-to-end to emit the same { COPY , GEN } op sequence without serializing through XML text. The information content per COPY op is identical—two small integers—so selector-EM should match C OPY -AS -DECODE . Where they differ is fixed cost per op: a span-level pointer pays 3 autoregressive forward passes (op-type +start +end), while C OPY -AS -D ECODE pays 2 (FSM open/close) + 1 (forced template prefill) + N/s (N ) (parallel-prefill splice) = 3 + N/s (N ). The pointer is therefore strictly cheaper in fixed overhead (by ≈ 0.15 forward-pass-equivalents per copy op on our corpora; see Appendix L), but requires dedicated training and dedicated heads, while C OPY -AS -D ECODE is a format layered on any instruction-tuned LM. A token-level pointer (predict input-position at every copied token) is strictly dominated: N forward passes vs N/s (N ). The mechanism claim is thus not “copy is free”—a span-level pointer achieves that with architectural surgery—but “copy can be made format-level and still get most of the way there,” which is the deployable property.

## 6 Related Work 

Speculative decoding and draft-based acceleration. Blockwise parallel decoding [ 25 ] first ob-served that a single parallel forward can verify multiple draft tokens at once; speculative decod-ing [ 3, 13 , 28 ] generalized this with a draft-model/target-model distinction and a probabilistic acceptance rule that preserves the target distribution. Medusa [ 1] replaces the separate draft model with extra prediction heads on the target; EAGLE [ 14 , 15 ] drafts in feature space rather than token space; SpecInfer [ 19 ] extends to tree-based draft expansion; Yang et al. [29] , Zhang et al. [30] explore multi-candidate and self-drafting variants; REST [ 10 ] retrieves drafts from a corpus. Across this family, drafts are produced by some learned predictor and accepted probabilistically to preserve the target distribution. Copy-as-Decode shares the parallel-forward primitive but differs in two architectural respects: the “draft” is the input document (not a smaller-model prediction) selected by grammar commitment, and acceptance is unconditional (not probabilistic)—so Copy-as-Decode does not preserve speculative decoding’s free-form target distribution, redefining the decoded task as an edit program. In exchange it avoids draft-model training and obtains a trivial parallel-forward ratio equal to the AR-vs-PP kernel ratio. The two methods compose: within a <gen> body, any speculative-decoding scheme can further accelerate free-text generation. 

Prompt-lookup decoding and input-aware drafting. The closest prior mechanism is prompt-lookup decoding [22 ]: draft candidate continuations by suffix-matching against the prompt and 15 accept them via the ordinary speculative-decoding verification rule. Edit-as-program paradigms. Copy-as-Decode is, to our knowl-edge, the first construction that (i) enforces via grammar that the spliced KV segment is a contiguous input-token range, and (ii) uses the splice as a decode-time replacement for N autoregressive steps with unconditional acceptance. 

Constrained decoding. Structured constrained decoding enforces an output grammar at the token level: PICARD [ 23 ] monitors partial SQL for well-formedness; Outlines [ 27 ] formulates decoding as finite-state intersection by compiling regular expressions and context-free grammars into token-level FSMs with a vocabulary index; XGrammar [ 5] and SGLang [ 31 ] provide efficient parallel-safe grammar compilation; Geng et al. [7] extends this to unconstrained LMs by masking with EBNF/CFG grammars without fine-tuning. Our FSM (§2.3) is a small instance of this family: an FSM over the 16 <copy> /<gen> grammar, made efficient by exploiting that Qwen-family BPE tokenizers encode each structural literal as a fixed multi-token sequence. What is distinctive is not the FSM itself but that the FSM serves a dual role—guaranteeing parse rate and marking the points at which the parallel-prefill splice can be invoked. 

Grammar-engine masking overhead. A parallel line of systems work focuses on driving per-step mask construction near zero: XGrammar [ 5 ] compiles a context-free grammar into an adaptive token-mask cache; SGLang [ 31 ] amortizes mask computation across requests via runtime sharing. Our FSM sits at the lower bound of this cost spectrum: 5 states, at most 12 allowed token-ids at any masked step (Figure 7 and Appendix A), measured at cmask = 0 .02 ms per masked step on Qwen2.5-1.5B (Appendix H)—two orders of magnitude below a single decode step. Copy-as-Decode’s grammar-constrained path is therefore compatible with any of the above engines and adds no measurable masking overhead; the substantive cost is the parallel-prefill splice, which none of these engines provide. 

Edit formats and apply models. Production tools use a variety of output formats for code and document edits. Aider [6] uses search/replace blocks, where the model emits both the verbatim text to find and the verbatim text to replace; unified-diff emits the standard @@ hunk format used by git .Cursor’s instant-apply [ 4] and Anthropic’s text-editor tool are closed-model apply pipelines whose training recipes are not public. Related work on editing as a task—SWE-bench [ 11 ], CanItEdit [ 2], HumanEvalPack [ 20 ]—evaluates end-to-end correctness rather than the decoding mechanism itself. All of these require the model to re-emit structural context (search anchors, hunk headers, or the entire edited span) through ordinary autoregressive decoding. Copy-as-Decode differs in replacing the model-produced reference with a grammar-enforced one, and in splicing the referenced tokens into the KV cache rather than paying decode cost for them. §4.4 benchmarks these formats head-to-head on a unified parse+apply harness: search/replace matches our format-theoretic token budget but loses 

6–19% of cases to anchor ambiguity, while unified diff matches our round-trip fidelity but costs 

17 –64% more tokens per edit. 

Copy mechanisms at the architectural level. Pointer networks [ 26 ] and copy mechanisms [ 8, 24 ]augment encoder-decoder generation with a learned pointer distribution over input positions. Edit-as-tagging architectures—LaserTagger [ 18 ], FELIX [ 17 ], GECToR [ 21 ]—cast editing as classifying each input token with an operation. The Levenshtein Transformer [ 9] learns non-autoregressive insertion/deletion operations. These approaches modify the model itself. Copy-as-Decode instead operates at the decoding and serving layer: any pretrained autoregressive LM that can be coaxed to emit the <copy> /<gen> grammar under the FSM can be served with it. 

Prompt-level structured output for edits. Closed-model structured-output prompting [ 16 ] has been studied as an edit format—the model emits a line-referencing program that a deterministic resolver expands. Our earlier work on CRP falls in this category; we treat it here as the prompt-level variant of Copy-as-Decode (Appendix Q) that retains the output-token compression benefit but not the wall-clock kernel speedup. The mechanism-level contribution of this paper is the combination of grammar-constrained decoding, parallel-prefill splicing, and a lossless resolver. 

## 7 Discussion and Conclusion 

We recast edit generation as structured decoding over a two-primitive grammar, enforced by a token-level FSM and accelerated by a parallel-prefill splice that replaces N autoregressive decode steps with one forward. The contribution is the mechanism together with three measurable properties—two quantitative bounds and one correctness property—independent of model quality: (i) one parallel-prefill forward is 6.8×–303 × faster than N autoregressive decodes for N ∈ [8 , 512] on Qwen2.5-{1.5B, 7B}; (ii) 74 –98% of gold tokens are realizable under the line-level primitive on ProbeEdit and HumanEvalPack-Fix, which composed with the kernel curve over each corpus’s exact span histogram yields the closed-form wall-clock bound bnd exact = 29 .0 × /3.4 × /4.2× (ProbeEdit / Py / JS, 13 .0× pooled); (iii) oracle programs round-trip byte-exactly on all 482 cases—an identifiability precondition that localizes any downstream EM failure to span selection. A perturbation study (pooled EM 100% → 15 .48% under off-by-one noise) gives an operational target for any trained selector. 17 Scope. In scope: contiguous line-range copy (and, in the token-level extension of §5, contiguous token-range copy) from a single input document to a single output document, with K ≤ 20 ops per edit on the evaluated corpora. Out of scope for the present paper: multi-file and agentic workloads (SWE-bench-style); AST-granular edits whose operands are subtrees rather than spans; splice integration into the vLLM/SGLang scheduler (format-only latency gains on vLLM 0.19 continuous batching are measured in §4.5; paged-attention splice integration is scaffolded in Appendix K and deferred). 

Limitations and future work. The FSM’s fixed-id table is tokenizer-conditional and resolved once at load (Appendix K: ≥ 7/14 of our structural literals are single-token across Qwen2.5, DeepSeek-Coder, and CodeLlama-family tokenizers; fragmented tokenizers add ≈ 0.02 ms per literal). Reserved-literal substrings in gold are handled by a principled Unicode-entity escape (§2.2, I), so “the gold-audit holds on our corpora” is no longer the load-bearing claim. Line granularity loses coverage to sub-expression rewrites; the 74 –98% vs. 93 –99% line-vs-token gap (Table 2, Appendix N) is closed by the token-indexed <copy tokens> primitive, which we formalize and measure as an oracle-ceiling in this draft but whose realized selector training we defer. The pilot selector at 14 –18% EM is a learnability signal, not a deployment result. Four concrete follow-ups: (i) scale the selector to 7B QLoRA with larger corpora and span-end-targeted supervision (span-end accuracy is the dominant error, §5); (ii) productize the token-level <copy tokens> primitive under a trained selector; (iii) integrate the splice into vLLM/SGLang continuous batching with paged attention and report E2E latency/throughput at realistic QPS, above the format-only 2.1–3.3× floor we measure in §4.5; (iv) apply in code-editing agent loops and on SWE-bench-style benchmarks. ## B Oracle Construction 

Algorithm. Given a pair (doc , gold ), split both on "\n" and run difflib.SequenceMatcher 

with autojunk=False to obtain the matching-block sequence. Matching blocks of size ≥ 1 become 

CopyLines ops with 1-indexed inclusive ranges; non-matching runs in the gold-side sequence become Gen ops carrying the verbatim gold-slice text padded with a single leading and trailing newline. 

The strip-at-most-one convention. The resolver strips at most one leading and one trailing newline from each Gen before emission. A naive rstrip would eat all trailing newlines, collapsing intentional blank lines at a <gen> →<copy> boundary; the at-most-one rule preserves authorial blank lines while still letting the grammar embed a readability newline on either side of a <gen> body.

22 G Mechanism Ablation: Grammar vs. Splice 

The mechanism has two independently removable components: the token-level FSM (which guaran-tees parse rate) and the parallel-prefill splice (which provides kernel speedup). Their contributions act on different axes. • Remove the FSM, keep prompt-level grammar and the splice. The decoder can drift out of the grammar mid-generation; our 4-case pilot shows prompt-level CRP yielding 0/4 parse on Qwen2.5-7B-Instruct under greedy decoding (plausible copy tags but wandering line indices or missing closing tags). Wall-clock is below full regen because fewer tokens are emitted before cutoff, but EM collapses. • Remove the splice, keep the FSM. The decoder still produces a syntactically valid program and the resolver still reconstructs gold from an oracle program; what is lost is the s(N ) factor on copy spans. Wall-clock approaches full regeneration with output-compression advantage only—exactly the closed-model setting where KV-cache access is unavailable. Pipeline-losslessness is intact. The FSM is required for correctness; the splice is required for speed. Table 11 isolates the splice independently of model quality; §4.3 establishes the resolver’s 482 /482 oracle round-trip, isolating the FSM/grammar contract.

## I Principled Escape and Token-Sequence End-of-Gen 

Escape scheme. §2.2 defines five reserved substrings ( <copy , </copy> , <gen> , </gen> ,

</program> ) and maps each to a Unicode entity (U+27E8 . . . U+27E9; U+2024 for the slash-like separator). Implementation: copy_as_decode/escape.py . Both encode_gen_body and

Token-sequence end-of-gen detector. The streaming decoder ( constrained_decoder.py ) uses exact token-id-sequence match against the Qwen BPE encoding of </gen> (3 tokens: 522 , 4370 , 29 )rather than a substring match on decoded text. This matters when adversarial content includes partial-substring splits across token boundaries: a decoded-text substring check can mis-fire at the boundary, while an id-sequence check cannot. The id-sequence matcher is implemented as a fixed-length rolling window over the last three emitted token ids; constant-time per decode step. 

Tokenizer portability. The id sequence for </gen> is tokenizer-specific (Appendix K). For tok-enizers that fragment structural literals, the escape scheme is unchanged but the FSM must resolve a per-tokenizer id table once at model load. 

## J Minimum-Span Auto-Policy m∗

The paper picks m = 8 qualitatively as the “splice is worth it” threshold (§4.1: at N ≤ 2, parallel prefill is within 1.3× of autoregressive). With the extended cost model we can make this per-corpus and principled. 

Definition. Given measured s(N ) and fixed costs c = cmask + csync plus the single-decode latency 

τ (reference point: Table 16 single-decode row), a copy splice of length N is net-positive when 

N · τ > N · τ /s (N ) + c. 

Rearranging, the minimum span is 

m∗ = min {N : s(N ) > 1 + c/ (N · τ )}.

For small c, m∗ is essentially the break-even of s(N ) and 1, i.e. the point where parallel prefill stops being overhead-dominated. With c = 0 .245 ms and τ = 19 .0 ms (Qwen2.5-1.5B measurements), 

c/τ ≈ 0.013 , negligible relative to the s(N ) curve.
