## 摘要
利用大语言模型（LLM）实现优化建模自动化是通往可扩展决策智能的一条极具前景的路径，但现有方法要么依赖于基于闭源 LLM 构建的智能体流水线（具有高推理延迟），要么使用昂贵的步骤级监督（process supervision）微调较小的 LLM，而这往往会过拟合到单一的求解器 API。受具有可验证奖励的强化学习启发，我们提出了执行验证优化建模（EVOM），这是一种将数学规划求解器视为确定性交互式验证器的执行验证学习框架。给定自然语言问题和目标求解器，EVOM 生成特定于求解器的代码，在沙箱环境中执行，并将执行结果转换为标量奖励，通过 GRPO 和 DAPO 在“生成-执行-反馈-更新”的闭环过程中进行优化。这种仅基于结果的公式化方法消除了对步骤级监督的需求，并通过切换验证环境（而非重新构建特定于求解器的数据集）实现了跨求解器的泛化。在 NL4OPT、MAMO、IndustryOR 和 OptiBench 上针对 Gurobi、OR-Tools 和 COPT 进行的实验表明，EVOM 的表现与经过步骤级监督微调（SFT）的模型相当或更优，支持零样本求解器迁移，并通过在目标求解器后端下持续训练实现了有效且低成本的求解器适配。

## Abstract
Automating optimization modeling with LLMs is a promising path toward scalable decision intelligence, but existing approaches either rely on agentic pipelines built on closed-source LLMs with high inference latency, or fine-tune smaller LLMs using costly process supervision that often overfits to a single solver API. Inspired by reinforcement learning with verifiable rewards, we propose Execution-Verified Optimization Modeling (EVOM), an execution-verified learning framework that treats a mathematical programming solver as a deterministic, interactive verifier. Given a natural-language problem and a target solver, EVOM generates solver-specific code, executes it in a sandboxed harness, and converts execution outcomes into scalar rewards, optimized with GRPO and DAPO in a closed-loop generate-execute-feedback-update process. This outcome-only formulation removes the need for process-level supervision, and enables cross-solver generalization by switching the verification environment rather than reconstructing solver-specific datasets. Experiments on NL4OPT, MAMO, IndustryOR, and OptiBench across Gurobi, OR-Tools, and COPT show that EVOM matches or outperforms process-supervised SFT, supports zero-shot solver transfer, and achieves effective low-cost solver adaptation by continuing training under the target solver backend.

论文关注的是**自动化优化建模**任务，即如何将自然语言描述的现实问题准确转化为数学规划模型（如线性规划 LP、混合整数规划 MILP）并生成可执行代码。

    2.  **泛化性差**：模型往往过拟合于特定求解器（如 Gurobi）的 API 语法，一旦更换求解器（如换成 OR-Tools），性能会剧烈下降。
*   **核心思想**：受“可验证奖励强化学习（RLVR）”启发，论文提出将数学求解器视为一个**确定性的验证器**。通过执行生成的代码并对比最终结果（目标值），利用强化学习实现“仅基于结果”的闭环优化，从而摆脱对中间推理步骤标注的依赖。

### 2. 方法论：EVOM 框架
论文提出了 **EVOM (Execution-Verified Optimization Modeling)** 框架，其核心是一个“生成-执行-反馈-更新”的闭环流程：
*   **求解器条件化目标**：模型输入包含“问题描述”和“目标求解器标识”，输出结构化的 `<think>`（推理过程）和 `<code>`（特定求解器代码）块。
*   **沙箱执行环境**：在受限的沙箱中运行生成的 Python 代码，调用 Gurobi、OR-Tools 或 COPT 等求解器，捕获执行状态（如 OPTIMAL）、目标函数值和错误日志。
*   **执行验证奖励设计**：
    *   **格式奖励 ($r_{fmt}$)**：通过正则匹配强制模型遵循特定的输出协议。
    *   **结果正确性奖励 ($r_{ans}$)**：如果代码可运行且求解出的目标值与标准答案在误差范围内（或状态一致），则给予正奖励。**不使用任何参考代码或中间步骤作为监督信号。**
*   **策略优化算法**：采用无评论器（Critic-free）的强化学习算法，包括 **GRPO**（组相对策略优化）和 **DAPO**（解耦剪切与动态采样），通过组内奖励归一化来估计优势函数。

### 6. 主要结论与发现
*   **结果监督胜过过程监督**：仅使用答案反馈的 EVOM 在多个基准上达到或超过了使用昂贵过程标注的 ORLM (SFT) 模型。

*   **推理块的必要性**：移除 `<think>` 块会导致性能大幅下降，证明强化学习通过结果反馈间接优化了模型的内部逻辑推理。
*   **低成本适配**：对于预训练中未见过的求解器（如 COPT），只需极少量（约 100 条）的冷启动数据配合 RL，即可快速达到高性能。

### 7. 优点（亮点）
*   **数据效率**：将昂贵的专家级过程标注转化为廉价的“问题-答案”对+求解器自动反馈。
*   **环境即监督**：巧妙地利用了求解器的确定性特性，将代码执行结果作为客观的真值信号。
*   **通用性**：提出了一种“求解器条件化”的建模方式，使单个模型能够根据指令切换不同的求解器 API。

### 8. 不足与局限
*   **冷启动依赖**：对于模型完全没见过的求解器 API，纯 RL 探索效率低，仍需要少量 SFT 数据引导。
*   **复杂约束挑战**：错误分析显示，虽然 RL 显著减少了语法和变量类型错误，但在处理极其复杂的逻辑约束（Constraint Error）方面仍有提升空间。
*   **搜索空间限制**：目前的奖励机制主要基于最终数值，对于中间建模逻辑的细粒度引导（如建模风格、效率优化）涉及较少。
